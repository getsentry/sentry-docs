## Developer Documentation (develop-docs/)

When writing requirements in `develop-docs/`:

1. **Use RFC 2119 Keywords**: Use "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" as defined in [RFC 2119](https://www.ietf.org/rfc/rfc2119.txt). These keywords **MUST** be written in uppercase and bold, for example: **MUST**, **SHOULD**, **MAY**.

2. **Add RFC 2119 Alert**: When creating a new file with requirements, or adding requirements to an existing file, ensure the file has an Alert at the top (after frontmatter) to clarify RFC 2119 usage. If missing, add:
   ```mdx
   <Alert>
     This document uses key words such as "MUST", "SHOULD", and "MAY" as defined
     in [RFC 2119](https://www.ietf.org/rfc/rfc2119.txt) to indicate requirement
     levels.
   </Alert>
   ```

## Long-term Knowledge

### Architecture

<!-- lore:019dfa02-6cde-7ca8-af49-4b7e3ea33812 -->
* **docs-mcp worker: skills.sentry.dev based retrieval (not docs.sentry.io)**: The current docs-mcp (\`apps/docs-mcp\` in cli-init-api) does NOT fetch from docs.sentry.io directly. It fetches from \`skills.sentry.dev\` — a CDN serving pre-authored SKILL.md files per SDK (e.g. \`skills.sentry.dev/sentry-nextjs-sdk/SKILL.md\`). The router skill at \`skills.sentry.dev/sdks\` maps keyword slugs to skill IDs. Feature references are at \`skills.sentry.dev/{skillId}/references/{feature}.md\`. The docs.sentry.io \`.md\` exports (LLM-friendly markdown) are a separate system built for the ask-agent migration. CF Cache API is used with 1hr TTL for skill fetches.

<!-- lore:019dfa02-6ce3-7c81-9459-802029ffdf6d -->
* **docs.sentry.io LLM-friendly .md exports for ask-agent use**: Every page at \`docs.sentry.io/\<path>\` has a \`.md\` export at \`docs.sentry.io/\<path>.md\` generated by \`scripts/generate-md-exports.mjs\` as a post-build step. These exports include YAML frontmatter (title, description, URL) and clean markdown body. A \`doctree.json\` file (at \`docs.sentry.io/doctree.json\`) provides a navigable index of all pages with titles/descriptions/paths for search. These are the recommended source for building an ask-agent over Sentry docs — the ask-agent should search doctree.json then fetch individual .md pages.

<!-- lore:019dfa02-6cd9-7e6c-bda4-115aac725075 -->
* **Kapa AI widget integration points in sentry-docs**: Kapa AI is a third-party widget loaded via \`\<Script strategy="lazyOnload">\` in \`app/layout.tsx\`. It creates \`window.Kapa.open({query, submit})\` API and a CSS class trigger (\`kapa-ai-class\`). Four integration points: (1) desktop 'Ask AI' button in \`search/index.tsx\` (CSS class), (2) mobile 'Ask AI' button in \`header.tsx\` (CSS class), (3) inline 'Ask Sentry about \[query]' result in \`search/index.tsx\` (programmatic), (4) \`?askAI=true\` URL deep-link in \`askAiSearchParams.tsx\` (programmatic). Config hardcoded in layout.tsx data-\* attributes. Widget CSP whitelist in \`vercel.json\`. TypeScript type in \`src/globals.d.ts\`.

### Decision

<!-- lore:019dfa02-6cec-71c2-a455-f3ef94c28324 -->
* **Replacing Kapa AI with first-party docs-mcp ask endpoint**: Backend: \`POST /api/ask\` on docs-mcp Cloudflare Worker streams structured SSE (not raw text) with typed events: \`text\` (delta), \`tool\_start\`/\`tool\_end\` (tool execution signals), \`error\` (structured, not mixed into content), \`done\`. Agent calls Anthropic Messages API directly via \`fetch\` \[\[019dfa02-6cf2-7d76-a14a-dad79ea5e4a7]], using \`search\_docs\` + \`fetch\_doc\_page\` tools against docs.sentry.io \[\[019dfa4b-a65f-79c5-8553-7d551f946993]]. Frontend (\`AskAiProvider\` + \`AskAiModal\`) parses SSE events, shows per-tool status labels during silent tool-execution gaps, supports AbortController cancel (stop button). Code blocks rendered with \`refractor\` + \`hast-util-to-jsx-runtime\` for Prism syntax highlighting. \`prism-sentry/index.css\` imported in root layout (not \`\[\[...path]]\` layout) so modal has access.

### Gotcha

<!-- lore:019dfa4b-a65f-79c5-8553-7d551f946993 -->
* **docs-mcp doctree.json is a root object with frontmatter.title, not an array**: The \`doctree.json\` at \`docs.sentry.io/doctree.json\` is a single root node object (\`{path, slug, frontmatter, children}\`), NOT an array. The \`title\` is under \`frontmatter.title\`, not a top-level \`title\` field. The \`path\` field on each node already contains the full relative path from root (e.g. \`platforms/javascript/guides/nextjs\`) — do not concatenate parent paths. Flatten by reading the root node's \`children\` array recursively.

<!-- lore:019dfa02-6cf2-7d76-a14a-dad79ea5e4a7 -->
* **docs-mcp package.json lacks AI SDK — must add before implementing ask-agent**: AI SDK v5 + @ai-sdk/anthropic in bun monorepos has fundamental multi-step tool-use breakage: \`streamText\` with \`maxSteps > 1\` silently drops subsequent steps, yielding 0 text-delta events after tool calls. The \`patchedTool()\` workaround (setting both \`parameters\` and \`inputSchema\`) fixes schema serialization but not the multi-step loop. Solution: bypass AI SDK entirely and call the Anthropic Messages API directly via \`fetch\`, implementing the agentic loop manually. Also: SSE response parsing must handle split chunks and interleaved \`event:\` lines — don't parse line-by-line naively; buffer and extract full \`data:\` lines.

### Pattern

<!-- lore:019dfaa4-3666-7a09-a7a1-e93eef13f532 -->
* **Ask AI modal: SSE parsing and abort pattern**: The \`/api/ask\` SSE stream uses \`event: \<type>\ndata: \<json>\n\n\` blocks. Frontend parses with a buffer split on \`\n\n\`, extracting \`event:\` and \`data:\` lines per block. AbortController is created per request and wired to both the \`fetch()\` signal and checked between agent steps on the backend. Stop button calls \`abortRef.current?.abort()\`. On abort, \`DOMException\` with \`name === 'AbortError'\` is swallowed silently. \`tool\_start\`/\`tool\_end\` events map to human-readable labels via \`TOOL\_LABELS\` dict to show status during silent tool-execution gaps.

<!-- lore:019dfa02-6cf6-724b-a3f5-9293f9faab9c -->
* **Hono + Cloudflare Workers env binding pattern in docs-mcp**: The docs-mcp Hono app doesn't currently pass \`env\` to handlers — \`createMcpServer()\` takes no args and \`Env\` only has \`CF\_VERSION\_METADATA\`. To add secrets (like \`ANTHROPIC\_API\_KEY\`), the pattern is: update \`Env\` in \`types.ts\`, pass env to the new handler via \`c.env\` (Hono's built-in CF bindings access), and use \`Sentry.withSentry((env) => ({...}), app)\` already present in \`index.ts\`. New endpoints like \`/api/ask\` get env via \`c.env as Env\`.

<!-- lore:019dfaa4-366c-7397-b537-21a0d55a4e48 -->
* **Runtime syntax highlighting in AI modal: refractor + hast-util-to-jsx-runtime**: The Ask AI modal uses \`refractor\` (Prism-based virtual DOM) + \`hast-util-to-jsx-runtime\` for runtime syntax highlighting — NOT the MDX/rehype pipeline used by docs pages. \`refractor.highlight(code, lang)\` returns a hast \`Root\`; cast it to \`Parameters\<typeof toJsxRuntime>\[0]\` to satisfy types, then call \`toJsxRuntime(tree, {Fragment, jsx, jsxs})\`. The \`prism-sentry/index.css\` must be imported in the root layout (\`app/layout.tsx\`) — not \`app/\[\[...path]]/layout.tsx\` — so \`.token.\*\` classes are styled globally including in the modal.
