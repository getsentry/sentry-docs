Once Sentry knows _where_ errors happen in your code, the next questions are _who_ introduced them and _what_ changed. The GitHub integration answers both: Sentry uses commit metadata to surface a **suspect commit** on each issue and suggest the commit's author as an **assignee**. It also turns every frame in your stack trace into a link straight to GitHub.

This step sets that up for your **frontend** project. You can follow the same flow for your backend project afterward.

## 1. Integrate Your GitHub Account and Repository

1. Follow the [GitHub integration install instructions](/integrations/source-code-mgmt/github/#install). When prompted to choose repositories, give Sentry access to the `tracing-tutorial-frontend` repo you forked.

1. After installing, you'll be redirected to a page showing your GitHub repositories in Sentry. Verify that `tracing-tutorial-frontend` appears under **Repositories**.

1. Open the **Code Mappings** tab and click **Add Code Mapping**. Map your Sentry frontend project to the `main` (or `master`) branch of `tracing-tutorial-frontend`, with the source root set to `src/`.

## 2. Set Up Commit Tracking

A _release_ is a version of your code deployed to an environment. See the [Releases](/product/releases/) documentation to learn more. Sentry uses releases to associate commits with the code that's running, which powers the [Suspect Commits](/product/issues/suspect-commits/) feature.

The Sentry Vite plugin can create a release and associate commits automatically on each build. Extend the plugin config you added in the previous step:

```javascript {filename:vite.config.js}
sentryVitePlugin({
  org: "___ORG_SLUG___",
  project: "___PROJECT_SLUG___",
  authToken: process.env.SENTRY_AUTH_TOKEN,
  release: {
    // Create a release and associate commits from your local git history.
    setCommits: {
      auto: true,
    },
  },
}),
```

The `release.setCommits.auto` option uses your local git history to associate commits with each new release.

## 3. Make a Change and Trigger a New Error

To see suspect commits work, you need a new error tied to a new commit.

1. In `src/App.jsx`, make a small edit — for example, change a button label so git registers a change.

1. Commit and push:

   ```bash
   git commit -am "Trigger a new release"
   git push
   ```

1. Create a production build. The plugin creates a new release and associates your commit with it:

   ```bash {tabTitle:npm}
   npm run build
   ```

   ```bash {tabTitle:yarn}
   yarn build
   ```

   ```bash {tabTitle:pnpm}
   pnpm build
   ```

1. Serve the build and trigger the cross-project error again:

   ```bash {tabTitle:npm}
   npm run preview
   ```

   ```bash {tabTitle:yarn}
   yarn preview
   ```

   ```bash {tabTitle:pnpm}
   pnpm preview
   ```

## 4. View the Suspect Commit and Assignee

1. Open the new issue's [Issue Details](/product/issues/issue-details/) page.

1. Find the **Suspect Commits** section — it should point to the commit you just pushed. Click the commit link to jump to GitHub.

1. In the right side panel under **Assigned To**, open the dropdown to see suggested assignees. The author of the suspect commit appears as a suggestion.

## 5. Use Stack Trace Linking

Stack trace links turn each frame in your stack trace into a clickable link to the exact line on GitHub.

1. On the same Issue Details page, scroll to the stack trace.

1. If Sentry hasn't auto-resolved the path, click **Tell us where your source code is**, confirm the file path and GitHub URL, and save.

1. You should now see an **Open this line in GitHub** link on each frame. Click it to jump to the source.

## Do the Same for Your Backend

You've set up suspect commits and stack trace linking for your frontend project. To get the same for your **backend**, repeat the [GitHub integration](/integrations/source-code-mgmt/github/#install) and code mapping steps for the `tracing-tutorial-backend` repo and its Sentry project. (The backend runs uncompiled, so it doesn't need source maps — but suspect commits work just the same.)

## What's Next

You now have Sentry providing:

- Cross-project distributed tracing
- Readable stack traces with source code context
- Suspect commits and suggested assignees
- One-click navigation from a stack trace to the offending line on GitHub

- [Dashboards](/product/dashboards/sentry-dashboards/) — track error and performance trends across your services
