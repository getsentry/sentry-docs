<Alert>
  [Logs](/es/product/explore/logs/) está disponible actualmente en los planes Team, Business y Developer. Estos planes incluyen una determinada cantidad de datos de Logs para procesar y almacenar. Una vez que superes tu cuota incluida, se te cobrará según tu presupuesto de pago por uso (PAYG).
</Alert>

<div id="key-terms">
  ## Términos clave
</div>

* **Cuota de registros (Logs)**: La cuota inicial que tu suscripción asigna para procesar y almacenar datos de registros de tus aplicaciones.

* **Presupuesto de pago por uso (PAYG)**: El presupuesto que defines para pagar el procesamiento de registros adicional más allá de tu cuota incluida.

* **Entrada de registro (Log)**: Un único mensaje de registro enviado desde tu aplicación a Sentry, que incluye metadatos como marca de tiempo, nivel de registro y atributos personalizados.

* **Procesamiento de registros (Logs)**: La ingesta, el análisis, la indexación y el almacenamiento de datos de registros en la infraestructura de Sentry.

<div id="before-you-begin-check-your-logs-usage">
  ## Antes de comenzar: revisa el uso de tus Logs
</div>

Puedes consultar el uso de tus Logs en la página Stats &amp; Usage para entender el desglose de tus entradas de registro por ventana temporal y por proyecto. Esto puede ayudarte a identificar dónde necesitas ajustar aún más tu uso.

Esta página está disponible para todos los miembros de tu organización, por lo que los Owners de tu organización de Sentry podrán compartirla con los desarrolladores directamente responsables de un proyecto determinado. También podrás usar esta página para evaluar si tus cambios están funcionando como se pretende.

<div id="how-can-i-see-a-breakdown-of-incoming-logs">
  ### ¿Cómo puedo ver un desglose de los registros entrantes?
</div>

La página Stats &amp; Usage también muestra detalles sobre la cantidad total de datos de registro que Sentry ha recibido en toda tu organización durante un máximo de 90 días. La página desglosa los eventos por proyecto en tres categorías: aceptados, descartados o filtrados. Solo los registros aceptados afectan tu cuota:

![Ejemplo de la página Stats & Usage](./img/usage-stats-errors.png)

<div id="which-projects-are-consuming-my-logs-spend">
  ### ¿Qué proyectos están consumiendo mi presupuesto de logs?
</div>

La tabla “Projects” en la página de “Stats” desglosa tus datos por proyecto para que puedas identificar cuáles consumen más cuota. Se puede ordenar por total de logs, logs aceptados, logs filtrados, etc.

<div id="adjusting-payg-budget">
  ## Ajustar el presupuesto PAYG
</div>

Los presupuestos solo pueden ser actualizados por un miembro con nivel Billing u Owner de tu organización de Sentry.

Una vez que tu cuota de Logs se acerque o haya superado la cantidad incluida, las personas con permisos de organización “Owner” empezarán a recibir correos de [notificación](/es/product/alerts/notifications/#quota-notifications). Luego podrán elegir aumentar o disminuir el presupuesto PAYG para habilitar uso adicional.

<div id="increasing-logs-budget">
  ### Aumentar el presupuesto de Logs
</div>

Si no puedes enviar nuevos logs porque has superado tu cuota de Logs, puedes aumentar tu presupuesto en cualquier momento durante tu período de facturación incrementando tu presupuesto PAYG. Esto es ideal cuando estás lanzando una nueva versión de tu aplicación o necesitas aumentar el nivel de registro para depurar. Para añadir presupuesto PAYG, establece un presupuesto mensual máximo compartido en tu [página de Suscripciones](https://sentry.io/orgredirect/organizations/:orgslug/settings/billing).

[Más información sobre cómo configurar tu presupuesto PAYG](/es/pricing/#pricing-how-it-works).

  ## Gestión del gasto
</div>

<div id="sdk-configuration">
  ### Configuración del SDK
</div>

En varios SDK puedes filtrar los logs antes de enviarlos a Sentry, para evitar mandar los que no te interesan. Esta es la forma más eficaz de reducir el uso de tu cuota de Logs; a continuación, algunos ejemplos:

<div id="javascript-sdks">
  #### SDK de JavaScript
</div>

```js
Sentry.init({
  dsn: "___PUBLIC_DSN___",
  enableLogs: true,
  beforeSendLog: (log) => {
    if (log.level === "info") {
      // Filtrar todos los logs de tipo info
      return null;
    }
    return log;
  },
});
```

<div id="python-sdk">
  #### SDK de Python
</div>

```python
import sentry_sdk
from sentry_sdk.types import Log, Hint
from typing import Optional

def before_log(log: Log, _hint: Hint) -> Optional[Log]:
    # Filtrar todos los registros de nivel info
    if log["severity_text"] == "info":
        return None
    return log

sentry_sdk.init(
    dsn="___PUBLIC_DSN___",
    "enable_logs": True,
    "before_send_log": before_log,
)
```

<div id="logs-filtering">
  ### Filtrado de registros
</div>

También puedes usar [Inbound Filters](/es/concepts/data-management/filtering/#logs-filtering) para filtrar registros a nivel de servidor. Aunque esto no impide que se envíen datos a Sentry, sí evita que los registros filtrados cuenten contra tu cuota.

Los siguientes filtros de entrada se aplican a los registros:

* **Mensaje de registro**: Filtra registros según la coincidencia del mensaje de registro
* **Releases**: Filtra registros de versiones específicas

Para configurar el filtrado por mensaje de registro:

1. Ve a **[Project] &gt; Project Settings &gt; Inbound Filters**
2. Agrega patrones para excluir mensajes de registro específicos
3. Usa comodines para coincidencias flexibles; por ejemplo, `*Connection timeout*`, para filtrar todos los registros que contengan “Connection timeout”

<div id="best-practices-for-reducing-logs-quota-usage">
  ### Mejores prácticas para reducir el uso de la cuota de registros
</div>

1. **Configura niveles de registro adecuados**: Registra solo al nivel que necesitas para producción; por ejemplo, WARNING y superiores.
2. **Usa registro estructurado**: Incluye contexto relevante en los atributos del registro en lugar de mensajes extensos.
3. **Filtra en el origen**: Usa callbacks `beforeSendLog` para filtrar los registros antes de enviarlos.
4. **Monitorea los patrones de uso**: Revisa regularmente tus estadísticas de uso para identificar proyectos con alto consumo de registros.
5. **Configura alertas**: Activa notificaciones cuando te acerques a los límites de tu cuota.

<div id="monitoring-your-logs-usage">
  ### Supervisa el uso de tus Logs
</div>

Haz seguimiento de tu cuota de Logs:

* Consultando con regularidad la página de [Usage Stats](/es/product/stats/#usage-stats)
* Configurando [quota notifications](/es/product/alerts/notifications/#quota-notifications)
* Revisando el uso específico del proyecto en el desglose de Stats
* Supervisando el impacto de los cambios de filtrado con el tiempo

Al aplicar estas estrategias, podrás gestionar eficazmente tu cuota de Logs y mantener la visibilidad necesaria sobre el comportamiento de tu aplicación.
