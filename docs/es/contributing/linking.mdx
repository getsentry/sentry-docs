En nuestra documentación creamos distintos tipos de enlaces, pero el más común conecta contenido entre las guías de producto y las páginas de la plataforma, o entre las propias páginas de la plataforma. Cada uno puede adoptar un formato diferente, según si es necesario especificar una plataforma.

<div id="product-platform-links">
  ## Enlaces de productos y plataforma
</div>

<div id="from-product-to-platform">
  ### De producto a plataforma
</div>

Usamos este formato al enlazar desde las guías del producto al contenido de la plataforma porque permite que el usuario especifique su plataforma, si aún no está definida. Por ejemplo:

`Las versiones también ofrecen funcionalidades adicionales significativas cuando [configuras por completo tu SDK](/platform-redirect/?next=/configuration/releases/).`

En el ejemplo anterior, cuando el usuario hace clic en el enlace al contenido de configuración de versiones y aún no ha visitado una página de plataforma, se le pedirá que especifique su plataforma para que la página se rellene con el ejemplo de código correcto al visualizarla. Si el usuario ha navegado al producto desde el contenido de la plataforma y ahora está volviendo a su plataforma, se mantiene el contexto de la plataforma.

<div id="within-platforms">
  ### Dentro de las plataformas
</div>

Usamos este formato al enlazar entre o dentro del contenido de la plataforma:

`Puedes configurar tus propias <PlatformLink to="/enriching-events/breadcrumbs/">breadcrumbs</PlatformLink> para que sean más útiles al depurar.`

En el ejemplo anterior, cuando el usuario hace clic en el enlace al contenido de breadcrumbs, se le llevará al contenido de breadcrumbs **dentro de su plataforma específica**. Si este enlace se usa en las guías de producto y no se especifica una plataforma, el usuario será dirigido a la primera plataforma de la lista, que actualmente es la familia de SDKs de Apple.

<div id="hardcoding-the-platform">
  ### Codificar explícitamente la plataforma
</div>

Usamos este formato cuando solo hay una opción para la plataforma, el contenido no es común a varias plataformas o queremos obligar al lector a usar esta fuente de verdad:

`[NDK integration](/platforms/android/configuration/using-ndk/) está incluido con el SDK y requiere el nivel de API 16, aunque se admiten otros niveles.`

En el ejemplo anterior, solo hay un lugar donde los usuarios pueden aprender a usar el NDK de Android, que está dentro del directorio de Android, por lo que podemos codificar la ruta de forma explícita.

<div id="to-product-pages">
  ### A páginas de producto
</div>

Usamos este formato al enlazar a nuestro contenido de producto. Actualmente, las guías de producto no necesitan especificar una plataforma. Por lo tanto, el enlace puede definir la ruta al contenido correcto en la carpeta `product`. Por ejemplo:

`También puedes personalizar cómo se agrupan los eventos. Obtén más información sobre la agrupación personalizada en [Issue Grouping](/concepts/data-management/event-grouping/).`

<div id="to-external-pages">
  ### A páginas externas
</div>

Usamos este formato cuando enlazamos a páginas externas. Normalmente se refiere a la interfaz:

`La página **Performance** es la vista principal en [sentry.io](https://sentry.io), donde puedes buscar o explorar datos de transacciones.`

<div id="to-sandbox">
  ### A Sandbox
</div>

Sandbox permite explorar la función en una aplicación real de Sentry con datos simulados.

`Obtén más información sobre esta función en nuestro <SandboxLink scenario="releases">sandbox</SandboxLink>.`

Esto llevará al usuario a un proyecto según la plataforma seleccionada. Sin embargo, también puedes especificar manualmente el proyecto de destino usando el parámetro `projectSlug`.

El parámetro `scenario` indica la página de inicio dentro de la aplicación Sandbox que verá el usuario y, si no se especifica, se usará por defecto la página de `issues`. Los valores posibles son:

* `performance`
* `releases`
* `alerts`
* `discover`
* `dashboards`
* `projects`
* `oneDiscoverQuery`
* `oneIssue`
* `oneBreadcrumb`
* `oneStackTrace`
* `oneTransaction`
* `oneWebVitals`
* `oneTransactionSummary`
* `oneRelease`

Además, puedes usar el parámetro `errorType` para seleccionar un tipo específico de error.

`Obtén más información sobre las breadcrumbs en nuestro <SandboxLink scenario="oneBreadcrumb" errorType="EXC_BAD_ACCESS" projectSlug="ios">sandbox</SandboxLink>.`

<div id="anchor-links">
  ## Enlaces ancla
</div>

Los enlaces ancla (añadidos con el símbolo #) llevan a los usuarios a una sección específica, en lugar de a la página en general.

<div id="within-the-same-page">
  ### Dentro de la misma página
</div>

Usamos este formato para enlazar a contenido que está más abajo en la página (normalmente por debajo del primer pliegue):

`En esta página puedes controlar la frecuencia de tus [notificaciones por correo electrónico](#notifications), [cambiar tu correo electrónico principal](#emails) y actualizar la configuración de seguridad.`

<div id="on-another-page">
  ### En otra página
</div>

Usamos este formato para dirigir a los usuarios a un encabezado específico en otra página:

`Para ampliar tu cuota o revisar qué ocurre cuando la superas, consulta [Aumentar cuotas](/pricing/quotas/#increasing-quotas).`

Este tipo de enlace también puede usarse para apuntar a anclas en páginas externas:

`Las sesiones duran [la duración de sesión predeterminada de Django](https://docs.djangoproject.com/en/1.11/topics/http/sessions/#using-cookie-based-sessions), que es de dos semanas.`

<div id="images">
  ## Imágenes
</div>

Todas las imágenes se encuentran en la carpeta `/public/`. Puedes hacer referencia a las imágenes de esta carpeta usando su ruta completa (sin `/public/`).

Por ejemplo, para usar la imagen `/public/product/alerts/issue-alert.png` en cualquier archivo:

`![Texto alternativo de la imagen](product/alerts/issue-alert.png)`

La estructura de carpetas de `docs` se replica en `public`, de modo que puedes seguir usando rutas relativas para referenciar imágenes en tus archivos de contenido. Para más detalles, consulta [Imágenes](/es/contributing/pages/images).

<div id="email-address">
  ## Dirección de correo electrónico
</div>

Usamos este formato cuando queremos enlazar una dirección de correo (a menudo usado para funciones de Early Adopter o beta):

`Para obtener más información sobre el acceso a Metric Alerts, no dudes en escribir a [alerting-feedback@sentry.io](mailto:alerting-feedback@sentry.io).`
