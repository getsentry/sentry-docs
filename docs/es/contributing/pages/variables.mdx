Hay una transformación disponible para archivos Markdown y MDX que admite el procesamiento de variables al estilo Django/Jekyll.
Las variables disponibles tienen alcance global y se configuran en `remark-variables.js`.

Por ejemplo:

```markdown {tabTitle:Replacment}
SDK de JavaScript: {{@inject packages.version("sentry.javascript.browser") }}
```

```markdown {tabTitle:Source}
SDK de JavaScript: {{\@inject packages.version("sentry.javascript.browser") }}
```

En este caso, exponemos `packages` como un accesor en el registro de paquetes, por lo que hay disponible una función `packages.version`. Además, exponemos una variable de contexto predeterminada llamada `page`, que contiene el front matter del nodo Markdown correspondiente. Por ejemplo, `{{\@inject page.title }}`.

Cuando una llamada a una variable es inválida (o produce errores) o no coincide con algo en el ámbito conocido, se mostrará un mensaje de error en la salida en modo de desarrollo. En producción, hará que la compilación falle.

Para escapar una expresión, agrega una barra invertida (`\`) delante de `@inject`.


<div id="packages">
  ## `packages`
</div>

El helper `packages` es una instancia de `PackageRegistry` y expone varios métodos.

<div id="packagesversion">
  ### `packages.version`
</div>

Devuelve la última versión del paquete indicado.

```text {tabTitle:Source}
{{\@inject packages.version("sentry.javascript.browser") }}
```

También puedes especificar opcionalmente un valor de reserva para cuando no se encuentre el paquete (o ocurra un error aguas arriba):

```text {tabTitle:Source}
{{\@inject packages.version("sentry.javascript.browser", "1.0.0") }}
```


<div id="packageschecksum">
  ### `packages.checksum`
</div>

Devuelve la suma de verificación de un archivo específico dentro de un paquete.

```text {tabTitle:Source}
{{\@inject packages.checksum("sentry.javascript.browser", "bundle.min.js", "sha384") }}
```
