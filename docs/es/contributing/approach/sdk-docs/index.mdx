Nuestra estructura para el contenido del SDK se centra en ayudar a la persona a empezar rápidamente y, después, en explorar todas las funciones que pueden configurarse o habilitarse en subpáginas. Estas subpáginas ofrecen material de referencia, con ejemplos de código específicos del SDK.

<div id="content-sections">
  ## Secciones de contenido
</div>

Los SDK proporcionan contenido centrado en 10 secciones (dos de ellas se generan automáticamente) que comparten contenido común. Cada sección se explica en detalle en su propia página. Un vistazo rápido:

1. **Inicio rápido / Getting Started (legado)**

   Una única página fundamental.

   Para SDK nuevos y guías actualizadas, usa:
   **[Cómo escribir: Inicio rápido](/es/contributing/approach/sdk-docs/write-quick-start/)**

   Para actualizar guías existentes de Getting Started, usa:
   **[Cómo escribir: Getting Started](/es/contributing/approach/write-getting-started)**

   Este contenido se usa en dos áreas distintas para clientes de Sentry:

   1. [docs.sentry.io](https://docs.sentry.io)
   2. Asistente dentro de la aplicación para clientes que crean o agregan un proyecto

2) **Configuración**

   Cubre todas las formas en que los clientes pueden gestionar su implementación del SDK. Consta de subpáginas (todas pueden activarse o desactivarse, según corresponda al SDK).

   **[Cómo actualizar: Configuración](/es/contributing/approach/write-configuration)**

3) **Uso**

   Explica cómo capturar errores y mensajes, con una subpágina para establecer el nivel.

   **[Cómo actualizar: Uso](/es/contributing/approach/write-usage)**

4) **Solución de problemas**

   Cubre los casos límite que conocemos. Este contenido se desarrolla a nivel del SDK principal, por lo que se comparte entre el SDK principal y sus guías asociadas. No es común a todos los SDK y plataformas.

   El contenido se encuentra en este directorio: `/docs/platforms/<SDK>/common/troubleshooting/`

5) (Opcional) **Guía de migración**

   Para clientes que necesitan actualizar desde la versión Raven del SDK. No es necesaria para SDK más recientes.

6) Sección de **Tracing**

   Explica cómo habilitar el tracing (para clientes que antes solo usaban el monitoreo de errores) según lo habilite cada SDK, además de subpáginas sobre Sampling y Transactions.

   *Esta sección está en desarrollo.*

7) Sección de **Enriquecimiento de eventos**

   Cubre los datos enviados a Sentry y cómo personalizarlos.

   **[Cómo actualizar: Enriquecimiento de eventos](/es/contributing/approach/write-enriching-events)**

8) Sección de **Gestión de datos**

   Explica nuestro algoritmo de agrupación y cómo puede modificarse, así como cómo depurar datos sensibles usando el SDK.

   **[Cómo actualizar: Gestión de datos](/es/contributing/approach/write-data-management)**

9) (Generado automáticamente) **Otras guías**

   Enumera otras plataformas, integraciones y SDK relacionados con este SDK.

10) (Generado automáticamente) **Otras plataformas**

    Enumera los otros SDK/plataformas principales.

Este modelo es bastante flexible: algunos SDK incorporarán más información; otros requerirán menos.

Tenemos varios objetivos al usar este formato:

* Asegurar que los clientes tengan una única página que les permita comenzar rápidamente.
* Actualizar el contenido del SDK para que sea **técnicamente preciso** y **completo**.
* Asegurar que hemos diseñado la estructura de nuestra documentación para admitir la separación entre contenido de Plataforma/SDK y de Producto. Piensa en esto como: &quot;la documentación de Plataforma/SDK ayuda a un cliente a enviar datos a Sentry&quot; y &quot;la documentación de Producto ayuda a un cliente a entender qué hay en [sentry.io](http://sentry.io) y luego a determinar una respuesta adecuada&quot;.

<div id="where-to-start">
  ## Por dónde empezar
</div>

1. Lee las [directrices para colaboradores](/es/contributing/). Te ayudarán a configurar tu entorno de documentación (el cambio a nuestras nuevas herramientas es importante) y a conocer detalles técnicos clave sobre el contenido del SDK, Buscar, las redirecciones y las páginas del asistente. Si necesitas ayuda, abre un issue en GitHub.
2. Usa las páginas de “Cómo hacerlo” para entender qué contenido se trata y dónde.
3. Usa la estructura de archivos y carpetas en JavaScript como modelo.
4. Crea un PR y, por favor, solicita a getsentry/docs como revisores.

<div id="what-not-to-do">
  ## Qué no hacer
</div>

* Darle demasiadas vueltas. El modelo no está pensado para funcionar el 100% del tiempo. Se espera que parte del contenido no funcione a la perfección. Si no estás seguro de dónde debería mostrarse el contenido, abre un issue en GitHub para preguntar.
* No olvides actualizar las páginas del Asistente que usan el contenido de Introducción para orientar a los clientes en sentry.io.
* No olvides actualizar la lista de SDK que admiten el monitoreo de rendimiento, según corresponda.
