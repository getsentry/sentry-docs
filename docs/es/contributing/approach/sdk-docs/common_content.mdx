Nuestra documentación busca un equilibrio, pensado para quienes contribuyen, entre duplicación y complejidad. Esto significa que, por lo general, no intentamos compartir contenido entre SDK en nuestra documentación, incluso cuando su arquitectura de información, secciones e incluso redacción son similares.
En lugar de imponer la coherencia mediante contenido compartido, animamos a las personas colaboradoras a consultar la documentación de otros SDK para mantener la consistencia siempre que sea posible.

Para los SDK que tienen documentación específica de un framework (a lo que llamamos guías o subpáginas), el contenido común para los frameworks de ese SDK se encuentra en `/docs/platforms/<platform_name>/common/`.

<div id="hierarchy">
  ## Jerarquía
</div>

Lo que se muestra al usuario depende de la jerarquía del contenido, tal como se detalla en nuestra sección sobre [Plataformas y guías](/es/contributing/platforms/). En resumen, si hay una guía disponible, se muestra su contenido; de lo contrario, se muestra el contenido de la plataforma.

<div id="tuning-common-content">
  ## Ajuste del contenido común
</div>

Para garantizar que el contenido común pueda escalar a medida que damos soporte a más plataformas y frameworks, usamos archivos `include`. Estos archivos pueden abarcar desde, típicamente, un ejemplo de código específico para una plataforma o framework, hasta información que complementa el contenido principal para una plataforma o framework concreta. Estos archivos `include` específicos de la plataforma están en [`platform-includes/`](https://github.com/getsentry/sentry-docs/tree/master/platform-includes). También contamos con [archivos `include` independientes de la plataforma](/es/contributing/approach/product-docs/write-index/#includes), que se abordan en nuestra documentación de producto.
