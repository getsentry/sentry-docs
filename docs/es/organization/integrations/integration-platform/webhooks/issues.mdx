Las integraciones de Sentry suscritas a los webhooks de incidencias pueden recibir información cuando se crea una incidencia y cuando esta cambia de estado.

<div id="sentry-hook-resource-header">
  ## Header Sentry-Hook-Resource
</div>

`'Sentry-Hook-Resource': 'issue'`

<div id="attributes">
  ## Atributos
</div>

<div id="action">
  ### action
</div>

* tipo: string
* descripción: puede ser `created`, `resolved`, `assigned`, `archived` o `unresolved`

### data[&#39;issue&#39;]

* type: objeto
* description: la incidencia

### data[&#39;issue&#39;][&#39;url&#39;]

* type: string
* description: la URL de la API del issue

### data[&#39;issue&#39;][&#39;web&#95;url&#39;]

* type: string
* description: la URL web de la incidencia

### data[&#39;issue&#39;][&#39;project&#95;url&#39;]

* type: string
* description: la URL de la API del proyecto al que pertenece la incidencia

<div id="payload">
  ## Payload
</div>

Payload incluye información adicional sobre el problema. Algunos campos importantes a los que conviene revisar son:

<div id="status">
  ### estado
</div>

Algunos webhooks de incidencias se activan cuando cambia el estado de una incidencia. El estado de una incidencia puede ser `resolved`, `unresolved` o `ignored` (archived).

<div id="substatus">
  ### substatus
</div>

Un estado puede abarcar varias cosas, por lo que el substatus aporta más detalles sobre dicho estado. Por ejemplo, el estado de un issue puede pasar a `unresolved` porque ahora está `escalating` o porque un issue resuelto ha `regressed`. Los posibles substatuses son: `archived_until_escalating`, `archived_until_condition_met`, `archived_forever`, `escalating`, `ongoing`, `regressed` y `new`.

<div id="statusdetails">
  ### statusDetails
</div>

`statusDetails` incluye información más detallada sobre los cambios de estado `resolved` y `archived` en un issue. Estos detalles pueden incluir:

* `inRelease`: La versión en la que se incluye la resolución (por ejemplo, `latest`)
* `inNextRelease`: `True` si la resolución estará en la próxima versión
* `inCommit`: Incluye información sobre el commit que contiene la resolución (como `repository` y `commit`)
* `ignoreCount`: Número máximo de ocurrencias en un issue archivado antes de que se escale
* `ignoreWindow`: Usado con `ignoreCount`, indica la cantidad de minutos durante los cuales se ignorarán esas ocurrencias
* `ignoreUserCount`: Número máximo de usuarios afectados por un issue archivado antes de que se escale
* `ignoreUserWindow`: Usado con `ignoreUserCount`, indica la cantidad de minutos durante los cuales se ignorarán esos usuarios afectados
* `ignoreDuration`: La duración (en minutos) durante la cual un issue archivado debe ignorarse antes de que se escale. El aumento en el número de usuarios afectados o en el número de ocurrencias no provocará una escalada antes de que pase este tiempo, a menos que haya un pico. Para obtener más información sobre cómo funciona, consulta [Algoritmo de escalado de issues](/es/product/issues/states-triage/escalating-issues/).
