Un *[problema de reproducción](/es/product/issues/issue-details/replay-issues/)* es un problema detectado mediante datos capturados de Session Replay. Si tu aplicación está configurada con [Session Replay](/es/product/session-replay/), los problemas se detectarán del lado del servidor durante la ingesta de la reproducción y se agruparán en problemas. Agrupamos eventos similares en problemas según una huella digital. Para los problemas de reproducción, la huella digital se basa principalmente en el tipo de problema y en la URL o el nombre de la transacción donde ocurrió el problema.

Puedes configurar qué tipos de problemas se crean yendo a Project Settings &gt; Replay y activando o desactivando la detección para cada tipo. De forma predeterminada, todos los tipos de problemas están habilitados.

<div id="configuring-rage-click-issues">
  ## Configuración de problemas por clics de ira
</div>

Los “clics muertos” (también llamados “clics lentos”) solo se detectan en elementos `<button>`, `<input>` y `<a>` que no provocan actualizaciones del DOM ni un desplazamiento de la página en un plazo de 7 segundos. Cuando el usuario hace clic en uno de estos elementos 3 o más veces dentro de ese periodo de 7 segundos, se interpreta como frustración y el SDK registra un [“clic de ira”](/es/product/issues/issue-details/replay-issues/rage-clicks/).

A veces se detectan clics de ira o clics muertos en elementos que no se espera que modifiquen el DOM, como un botón de “Imprimir” o “Descargar”. En ese caso, puedes configurar `slowClickIgnoreSelectors`, lo que evitará que esos botones generen nuevos problemas.

<Alert>
  Necesitas la versión 7.60.1 o superior del SDK para poder ver
  **problemas por clics de ira**.
</Alert>

<div id="configuring-hydration-errors">
  ## Configuración de errores de hidratación
</div>

Cuando ocurre un [error de hidratación](/es/product/issues/issue-details/replay-issues/hydration-error/) en tu aplicación React, el SDK emite un objeto de error y una marca (breadcrumb) de reproducción, ambos con datos sobre el problema de hidratación. Ambos se envían al servidor para su procesamiento. Al revisar el flujo de incidencias verás incidencias de Hydration Error que pueden provenir de cualquiera de las dos fuentes de datos. Sin embargo, solo verás las herramientas de comparación (diff) que facilitan la depuración si hay una reproducción asociada con el error.

Por esta razón recomendamos tener habilitados tanto los [Inbound Filters](/es/concepts/data-management/filtering/) como el ajuste de Replay “Create Hydration Error Issues”. ¡Esta también es la configuración predeterminada!

Puedes configurar tus preferencias de una de cuatro maneras:
| Inbound Filters | Replay Hydration Error toggle | Resultado |
| --- | --- | --- |
| enabled | enabled | (recomendado) Incidencia creada a partir de datos de reproducción. |
| enabled | disabled | No se crearán incidencias. |
| disabled | enabled | Se crean dos incidencias, tanto de la reproducción como de los datos del error. Las incidencias son duplicadas. |
| disabled | disabled | Incidencia creada a partir de los datos del error capturado. |

<Alert>
  Necesitarás tener la versión 7.87.0 del SDK o superior para poder detectar
  **incidencias de error de hidratación** a partir de datos de reproducción.
</Alert>
