<div id="how-to-decide-which-installation-method-to-use">
  ## Cómo decidir qué método de instalación usar
</div>

La mayoría de las aplicaciones de Node hoy en día están escritas en CommonJS (CJS) o se compilan a CJS antes de ejecutarlas.
CommonJS usa `require()` para cargar módulos. Nuestro método de instalación recomendado al usar CommonJS es requerir el archivo `instrument.js` al inicio de tu aplicación. Sin embargo, si tu aplicación se ejecuta en modo ESM, esto no funcionará. En ese caso, puedes seguir la [documentación de ESM](./esm).

Ten en cuenta que, incluso si tu aplicación está escrita en ESM (usando `import`), aún puede _ejecutarse_ en CJS. En ese caso, debes seguir las [instrucciones de CommonJS](./commonjs).

<div id="my-application-is-in-typescript">
  ### Mi aplicación está en TypeScript
</div>

Si usas TypeScript, es probable que tu aplicación se compile a CommonJS antes de ejecutarse. En ese caso, debes seguir las [instrucciones de CommonJS](./commonjs).

<div id="my-application-uses-require">
  ### Mi aplicación usa `require`
</div>

Si estás usando `require()` en tu aplicación, debes seguir las [instrucciones de CommonJS](./commonjs).

<div id="my-application-uses-import">
  ### Mi aplicación usa `import`
</div>

Si estás usando `import` en tu aplicación, tu método de instalación depende de cómo se _ejecuta_ tu aplicación. Si compilas tu aplicación (por ejemplo, en una carpeta `/dist` o similar) antes de ejecutarla, necesitas revisar cómo se ve el código compilado. ¿El código compilado usa `require`? Entonces debes seguir las [instrucciones de CommonJS](./commonjs). Si el código compilado usa `import`, debes seguir las [instrucciones de ESM](./esm).

Si no compilas tu código, tendrás que seguir las [instrucciones de ESM](./esm).

</PlatformCategorySection>

<div id="how-to-decide-which-installation-method-to-use">
  ## Cómo decidir qué método de instalación usar
</div>

Según las necesidades concretas de tu aplicación, puede que te preguntes qué método de instalación deberías usar.

<div id="why-choose-the-loader-script-over-npm">
  ### Por qué elegir el Loader Script en lugar de npm
</div>

Usar el Loader Script es la manera más fácil de empezar con Sentry. Agrega una etiqueta de script a tu aplicación y nosotros nos encargamos de configurar correctamente todo lo demás por ti.

- **Fácil de configurar**: Solo [agrega una etiqueta de script](./loader/#using-the-loader) a tu aplicación.

- **Fácil de actualizar**: Nos encargaremos de entregar siempre la última versión del SDK a tus usuarios. No necesitas hacer nada. Esto garantiza que siempre obtendrás las últimas funciones y correcciones de errores del SDK de Sentry.

- **Fácil de añadir configuración adicional**: Puedes [configurar el SDK](./loader/#sdk-configuration) a tu gusto mediante `window.sentryOnLoad`.

- **Carga diferida (lazy-loading)**: Si solo quieres usar Sentry para errores, cargaremos el SDK únicamente cuando ocurra un error, reduciendo hasta entonces la cantidad de JavaScript cargado en tu página.

  <Alert>
    Ten en cuenta que, al habilitar Session Replay y/o Tracing, el SDK se
    cargará inmediatamente porque necesitamos capturar lo que sucede en la página
    lo antes posible.
  </Alert>

<div id="why-choose-npm-over-the-loader-script">
  ### Por qué elegir npm en lugar del Loader Script
</div>

Aunque usar el Loader Script tiene ventajas claras, también trae algunas desventajas.
En estos escenarios, usar el paquete de npm es la mejor opción:

- **Inicialización específica del framework**: Por ahora, el Loader Script solo proporciona instrumentación genérica de JavaScript en el navegador. Esto significa que no obtendrás funciones específicas de frameworks como React, Vue o similares para Sentry. Para obtener estas funciones, deberás instalar el SDK del framework (por ejemplo, `@sentry/react` o `@sentry/vue`) mediante npm.

- **API pública completa**: El Loader Script solo expone un subconjunto de las API públicas (por ejemplo, `Sentry.captureException()`, ...). Si necesitas mucha funcionalidad personalizada, probablemente te convenga más el paquete de npm.

- **Control total sobre la versión del SDK**: Si necesitas control total sobre la versión del SDK, deberás instalar el SDK mediante npm.
