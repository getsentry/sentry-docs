<div id="core-options">
  ## Opciones principales
</div>

<SdkOption name="dsn" type="string" envVar="SENTRY_DSN">
  El DSN le indica al SDK adónde enviar los eventos. Si no se configura, el SDK no enviará ningún evento.
  Obtén más información sobre el [uso del DSN](/es/product/sentry-basics/dsn-explainer/#dsn-utilization).

  <PlatformContent includePath="configuration/dsn" />
</SdkOption>

<SdkOption name="orgId" type="`${number}` | number" categorySupported={["server", "serverless"]}>
  El ID de la organización de tu proyecto en Sentry.

  El SDK intentará extraer el ID de la organización del DSN. Si no lo encuentra o necesitas sobrescribirlo,
  puedes indicar el ID con esta opción. El ID de la organización se usa para la propagación de trazas y funciones como `strictTraceContinuation`.

  El ID de la organización se usa para funciones como <PlatformLink to="/configuration/options#strictTraceContinuation">continuación estricta de trazas</PlatformLink>.
</SdkOption>

<SdkOption name="debug" type="boolean" defaultValue="false">
  Activa o desactiva el modo de depuración. Si la depuración está habilitada, el SDK intentará imprimir información útil sobre lo que está haciendo.
</SdkOption>

<SdkOption name="release" type="string" envVar="SENTRY_RELEASE">
  Establece la versión. Los nombres de versión son cadenas, pero Sentry detecta algunos formatos y podría representarlos de forma diferente. Obtén más información sobre cómo enviar datos de versiones para que Sentry pueda informarte sobre regresiones entre versiones e identificar la posible fuente en [la documentación de versiones](/es/product/releases/) o la <SandboxLink scenario="releases">sandbox</SandboxLink>.

  <PlatformCategorySection supported={["server", "serverless"]}>
    En el servidor, el SDK intentará leer este valor de la variable
    de entorno `SENTRY_RELEASE`.
  </PlatformCategorySection>

  <PlatformCategorySection supported={["browser"]}>
    En el navegador, el SDK intentará leer este valor desde
    `window.SENTRY_RELEASE.id` si está disponible.
  </PlatformCategorySection>

  <PlatformSection supported={["javascript.electron"]}>
    El SDK leerá propiedades del módulo `app` de Electron para crear la
    versión con el formato `appName@version`.
  </PlatformSection>
</SdkOption>

<SdkOption name="environment" type="string" defaultValue="production" envVar="SENTRY_ENVIRONMENT">
  Establece el entorno. De forma predeterminada es `development` o `production`, según si la aplicación está empaquetada.

  Los entornos te indican dónde ocurrió un error, ya sea en tu sistema de producción, tu servidor de staging u otro lugar.

  Sentry crea automáticamente un entorno cuando recibe un evento con el parámetro de entorno establecido.

  Los entornos distinguen entre mayúsculas y minúsculas. El nombre del entorno no puede contener saltos de línea, espacios ni barras diagonales, no puede ser la cadena &quot;None&quot; ni superar los 64 caracteres. No puedes eliminar entornos, pero puedes ocultarlos.
</SdkOption>

<SdkOption name="tunnel" type="string">
  Establece la URL que se usará para transportar los eventos capturados. Esto puede servir para sortear bloqueadores de anuncios o tener un control más granular sobre los eventos enviados a Sentry. Agregar tu DSN sigue siendo necesario al usar esta opción para que se puedan establecer los atributos necesarios en los datos generados de Sentry. Esta opción **requiere implementar** un endpoint de servidor personalizado. Obtén más información y ejemplos en [Cómo lidiar con bloqueadores de anuncios](/es/platforms/javascript/troubleshooting/#dealing-with-ad-blockers).
</SdkOption>

<SdkOption name="sendDefaultPii" type="boolean" defaultValue="false">
  Establece esta opción en `true` para enviar datos PII predeterminados a Sentry. Entre otras cosas, habilitar esto permitirá la recopilación automática de direcciones IP en los eventos.
</SdkOption>

<SdkOption name="maxBreadcrumbs" type="number" defaultValue="100">
  Esta variable controla la cantidad total de breadcrumbs que se deben capturar. Ten en cuenta que Sentry tiene un [tamaño máximo de carga](https://develop.sentry.dev/sdk/data-model/envelopes/#size-limits) y cualquier evento que exceda ese tamaño se descartará.
</SdkOption>

<SdkOption name="attachStacktrace" type="boolean" defaultValue="false">
  Cuando está habilitado, las trazas de pila se adjuntan automáticamente a todos los mensajes registrados. Las trazas de pila siempre se adjuntan a las excepciones; sin embargo, cuando se habilita esta opción, las trazas de pila también se envían con los mensajes. Esta opción, por ejemplo, hace que las trazas de pila aparezcan junto a todos los mensajes capturados con `Sentry.captureMessage()`.

  La agrupación en Sentry es diferente para eventos con y sin trazas de pila. Como resultado, obtendrás nuevos grupos a medida que habilites o deshabilites este indicador para ciertos eventos.
</SdkOption>

<SdkOption name="serverName" type="string" categorySupported={['server', 'serverless']}>
  Esta opción puede usarse para proporcionar un nombre de servidor. Cuando se especifica, el nombre del servidor se envía y se conserva en el evento. En muchas integraciones, el nombre del servidor corresponde al nombre de host del dispositivo, incluso cuando la máquina no es realmente un servidor.

  La mayoría de los SDK intentarán detectar automáticamente este valor.
</SdkOption>

<PlatformSection supported={['javascript.electron']}>
  <SdkOption name="autoSessionTracking" type="boolean" defaultValue="true">
    Cuando no se configura en `false`, el SDK hace seguimiento de las sesiones vinculadas al ciclo de vida del proceso principal de Electron.
  </SdkOption>
</PlatformSection>

<SdkOption name="initialScope" type="CaptureContext">
  Datos que se establecerán en el alcance inicial. El alcance inicial puede definirse como un objeto o una función de callback, como se muestra a continuación.

  ```javascript
  // Using an object
  Sentry.init({
    dsn: "___PUBLIC_DSN___",
    initialScope: {
      tags: { "my-tag": "my value" },
      user: { id: 42, email: "john.doe@example.com" },
    },
  });

  // Using a callback function
  Sentry.init({
    dsn: "___PUBLIC_DSN___",
    initialScope: (scope) => {
      scope.setTags({ a: "b" });
      return scope;
    },
  });
  ```
</SdkOption>

<SdkOption name="maxValueLength" type="number">
  Número máximo de caracteres que puede tener cada propiedad de tipo string en los eventos enviados a Sentry antes de ser truncada.
</SdkOption>

<SdkOption name="normalizeDepth" type="number" defaultValue="3">
  Los SDK de Sentry normalizan los datos contextuales hasta una profundidad determinada. Cualquier dato más allá de esta profundidad se recortará y se marcará usando su tipo (`[Object]` o `[Array]`), sin recorrer más el árbol. De forma predeterminada, el recorrido se realiza hasta tres niveles.
</SdkOption>

<SdkOption name="normalizeMaxBreadth" type="number" defaultValue="1000">
  Este es el número máximo de propiedades o entradas que se incluirán en cualquier objeto o array cuando el SDK normalice datos contextuales. Cualquier dato más allá de este límite se descartará.
</SdkOption>

<SdkOption name="enabled" type="boolean" defaultValue="true">
  Especifica si este SDK debe enviar eventos a Sentry. Configurarlo en `enabled: false` no evita toda la sobrecarga de la instrumentación de Sentry. Para deshabilitar Sentry por completo, según el entorno, llama condicionalmente a `Sentry.init`.
</SdkOption>

<SdkOption name="sendClientReports" type="boolean" defaultValue="true">
  Establece esta opción en `false` para deshabilitar el envío de informes del cliente. Los informes del cliente son una característica del protocolo que permite a los clientes enviar reportes de estado sobre sí mismos a Sentry. Actualmente se usan principalmente para emitir resultados de eventos que nunca se enviaron.
</SdkOption>

<SdkOption name="includeLocalVariables" type="boolean" defaultValue="false" categorySupported={['server', 'serverless']}>
  Establece esta opción en `true` para añadir variables locales de la pila a los stack traces.

  <PlatformSection notSupported={["javascript.bun", "javascript.cloudflare", "javascript.deno"]}>
    Para opciones de configuración más avanzadas, consulta la documentación sobre las <PlatformLink to="/configuration/integrations/localvariables">opciones de la integración Local Variables</PlatformLink>.
  </PlatformSection>
</SdkOption>

<SdkOption name="integrations" type="Array<Integration> | (integrations: Array<Integration>) => Array<Integration>" defaultValue="[]">
  Pasa integraciones adicionales que deban inicializarse con el SDK. Las integraciones son componentes de código que pueden usarse para ampliar la funcionalidad del SDK. Pueden servir para añadir procesadores de eventos personalizados, proveedores de contexto o para conectarse al ciclo de vida del SDK.

  Consulta la <PlatformLink to="/configuration/integrations/">documentación de integraciones</PlatformLink> para más información.
</SdkOption>

<SdkOption name="defaultIntegrations" type="undefined | false">
  Esto puede usarse para deshabilitar las integraciones agregadas por defecto. Cuando se establece en `false`, no se agregan integraciones predeterminadas.

  Consulta la <PlatformLink to="/configuration/integrations/#modifying-default-integrations">documentación de integraciones</PlatformLink> para ver cómo puedes modificar las integraciones predeterminadas.
</SdkOption>

<SdkOption name="beforeBreadcrumb" type="(breadcrumb: Breadcrumb, hint?: BreadcrumbHint) => Breadcrumb | null">
  Esta función se llama con un objeto breadcrumb antes de agregarlo al scope. Si la función no devuelve nada, el breadcrumb se descarta. Para dejar pasar el breadcrumb, devuelve el primer argumento, que contiene el objeto breadcrumb.
  El callback recibe un segundo argumento (llamado “hint”) que contiene el objeto original a partir del cual se creó el breadcrumb para personalizar mejor su aspecto.
</SdkOption>

<SdkOption name="transport" type="(transportOptions: TransportOptions) => Transport">
  El SDK de JavaScript utiliza un transport para enviar eventos a Sentry. En navegadores modernos, la mayoría de los transports usan la API fetch del navegador para enviar eventos. Los transports descartarán un evento si no pueden enviarlo por falta de conexión.

  <PlatformCategorySection supported={["browser"]}>
    En el navegador, se utiliza por defecto un transport basado en `fetch`.
  </PlatformCategorySection>

  <PlatformCategorySection supported={["server", "serverless"]}>
    En el servidor, se utiliza por defecto un transport basado en `https`.
  </PlatformCategorySection>
</SdkOption>

<SdkOption name="transportOptions" type="TransportOptions">
  Opciones para configurar el transport. Es un objeto con las siguientes claves opcionales:

  <PlatformCategorySection supported={["server", "serverless"]}>
    <PlatformCategorySection supported={["browser", "desktop"]}>
      **Opciones para el transport `httpTransport` de Node:**
    </PlatformCategorySection>

    * `headers`: Un objeto con los encabezados que se enviarán con cada solicitud.
    * `proxy`: Un proxy para solicitudes salientes. Puede ser http o https.
    * `caCerts`: Una ruta o lista de rutas a un certificado de CA, o un buffer de certificados de CA.
    * `httpModule`: Un módulo HTTP personalizado para usar en las solicitudes. Por defecto, usa los módulos nativos `http` y `https`.
    * `keepAlive`: Determina si se mantiene el socket activo entre solicitudes. Por defecto: `false`.
  </PlatformCategorySection>

  <PlatformCategorySection supported={["browser", "desktop"]}>
    <PlatformCategorySection supported={["server", "serverless"]}>
      **Opciones para el transport `fetch` del navegador:**
    </PlatformCategorySection>

    * `headers`: Un objeto con los encabezados que se enviarán con cada solicitud.
    * `fetchOptions`: Un objeto con las opciones que se pasarán a la llamada a `fetch`. Usado por el transport basado en fetch del SDK.
  </PlatformCategorySection>
</SdkOption>

<SdkOption name="shutdownTimeout" type="number" defaultValue="2000" categorySupported={['server', 'serverless']}>
  Controla cuántos segundos esperar antes de cerrar. Los SDK de Sentry envían eventos desde una cola en segundo plano. A esta cola se le da un tiempo para vaciar los eventos pendientes. El valor predeterminado es específico del SDK pero suele ser de unos dos segundos. Establecerlo demasiado bajo puede causar problemas al enviar eventos desde aplicaciones de línea de comandos. Establecerlo demasiado alto hará que la aplicación se bloquee durante mucho tiempo para usuarios con problemas de conectividad de red.
</SdkOption>

<SdkOption name="disableInstrumentationWarnings" type="boolean" defaultValue="false" categorySupported={['server']}>
  Según tu configuración, el SDK de Sentry intentará detectar si se ha configurado de forma incorrecta. Esto puede generar advertencias como estas en los registros:

  > [Sentry] &lt; libraryName &gt; no está instrumentada. Probablemente se deba a que hiciste require/import de &lt; libraryName &gt; antes de llamar a `Sentry.init()`.

  O

  > [Sentry] &lt; libraryName &gt; no está instrumentada. Asegúrate de inicializar Sentry en un archivo separado que `--importes` al ejecutar Node; consulta: &lt; docs link &gt;.

  Esto significa que el SDK detectó que la biblioteca no se ha envuelto para la instrumentación automática de rendimiento. Esto puede hacer que algunos spans no se informen correctamente. Si esto no te afecta (por ejemplo, porque la advertencia es un falso positivo o no te importan esos spans específicos), puedes desactivar esta advertencia estableciendo esta opción en `true`.
</SdkOption>

<PlatformSection supported={['javascript.electron']}>
  <SdkOption name="ipcMode" type="'IPCMode.Classic' | 'IPCMode.Protocol' | 'IPCMode.Both'" defaultValue="'IPCMode.Both'">
    Modo de comunicación entre procesos para recibir actualizaciones de eventos y de ámbito desde procesos de renderizado.

    Las opciones disponibles son:

    * `IPCMode.Classic` - Configura los módulos IPC de Electron
    * `IPCMode.Protocol` - Configura un protocolo personalizado y `fetch`
    * `IPCMode.Both` (predeterminado) - Configura ambos modos para lograr la máxima compatibilidad
  </SdkOption>

  <SdkOption name="ipcNamespace" type="string">
    Espacio de nombres personalizado para los canales de comunicación entre procesos (IPC) usados por el SDK de Sentry para Electron. Es útil cuando tu aplicación de Electron utiliza múltiples canales IPC y quieres evitar posibles conflictos entre ellos.

    ```javascript
    import * as Sentry from "@sentry/electron/main";

    Sentry.init({
      dsn: "___PUBLIC_DSN___",
      ipcNamespace: "myCustomNamespace",
    });
    ```

    Cuando se establece, los canales IPC utilizados por Sentry se anteponen con el espacio de nombres especificado. Ten en cuenta que debes configurar el mismo espacio de nombres en el proceso principal, los procesos de renderizado y los scripts de preload para una comunicación correcta. Consulta la [sección Custom IPC Namespace](/es/platforms/javascript/guides/electron/#custom-ipc-namespace) para ver las instrucciones completas de configuración.
  </SdkOption>

  <SdkOption name="getSessions" type="() => Electron.Session[]" defaultValue="() => [session.defaultSession]">
    Función que devuelve una matriz de objetos `session` de Electron. Estas sesiones se utilizan para configurar la comunicación entre los procesos principal y de renderizado de Electron.
  </SdkOption>

  <SdkOption name="getRendererName" type="(contents: WebContents) => string | undefined">
    Función de devolución de llamada que permite asignar nombres personalizados a los procesos de renderizado.
    Si no se establece la devolución de llamada o devuelve `undefined`, se utiliza el esquema de nombres predeterminado.
  </SdkOption>
</PlatformSection>

<SdkOption name="sampleRate" type="number" defaultValue="1.0">
  Configura la tasa de muestreo para eventos de error, en el rango de `0.0` a `1.0`. El valor predeterminado es `1.0`, lo que significa que se enviará el 100% de los eventos de error. Si se establece en `0.1`, solo se enviará el 10% de los eventos de error. Los eventos se seleccionan al azar.
</SdkOption>

<SdkOption name="beforeSend" type="(event: Event, hint: EventHint) => Event | null">
  Esta función se invoca con un mensaje específico del SDK o un objeto de evento de error, y puede devolver un evento modificado o `null` para omitir el envío del evento. Por ejemplo, puede usarse para eliminar manualmente PII antes de enviarlo.

  Para cuando se ejecuta `beforeSend`, todos los datos del alcance (scope) ya se han aplicado al evento. Cualquier modificación posterior del alcance no tendrá efecto.
</SdkOption>

<PlatformSection supported={["javascript.vue", "javascript.nuxt"]}>
  <SdkOption name="attachProps" type="boolean" defaultValue="true" categorySupported={['browser']}>
    Cuando está habilitado, todas las props del componente de Vue donde se produjo el error se adjuntan al evento.
  </SdkOption>

  <SdkOption name="attachErrorHandler" type="boolean" defaultValue="true" categorySupported={['browser']}>
    De forma predeterminada, el SDK adjunta un [manejador de errores de Vue](https://vuejs.org/api/application.html#app-config-errorhandler)
    para capturar excepciones de Vue y reportarlas a Sentry.
    Cuando `attachErrorHandler` se establece en `false`, se desactiva el envío automático de errores.

    Por lo general, esta opción debería permanecer habilitada, a menos que quieras configurar tú mismo el reporte de errores de Sentry.
    Por ejemplo, el SDK de Sentry para Nuxt no adjunta un manejador de errores, ya que captura errores a través de los hooks de error proporcionados por Nuxt.
  </SdkOption>

<div id="tracing-options">
  ## Opciones de tracing
</div>

<SdkOption name="tracesSampleRate" type="number" envVar="SENTRY_TRACES_SAMPLE_RATE">
  Un número entre `0` y `1` que controla la probabilidad (%) de que una transacción determinada se envíe a Sentry. (`0` representa 0% y `1` representa 100%.) Se aplica por igual a todas las transacciones creadas en la app. Debe definirse este o `tracesSampler` para habilitar el tracing.
</SdkOption>

<SdkOption name="tracesSampler" type="(samplingContext: SamplingContext) => number | boolean">
  Una función responsable de determinar la probabilidad (%) de que una transacción determinada se envíe a Sentry. Recibe automáticamente información sobre la transacción y el contexto en el que se crea, y debe devolver un número entre `0` (0% de probabilidad de envío) y `1` (100% de probabilidad de envío). También puede usarse para filtrar transacciones, devolviendo 0 para las no deseadas. Debe definirse este o `tracesSampleRate` para habilitar el tracing.

  El objeto `samplingContext` pasado a la función tiene las siguientes propiedades:

  * `parentSampled`: La decisión de muestreo de la transacción padre. Es `true` si la transacción padre fue muestreada y `false` si no lo fue.
  * `name`: El nombre del span tal como se inició.
  * `attributes`: Los atributos iniciales del span.
    <PlatformCategorySection supported={["server", "serverless"]}>
      * `normalizedRequest`: Información de la solicitud HTTP actualmente activa
        del servidor, si corresponde.
    </PlatformCategorySection>
</SdkOption>

<SdkOption name="tracePropagationTargets" type="Array<string | RegExp>">
  Propiedad opcional que controla qué servicios downstream reciben datos de tracing, en forma de los headers `sentry-trace` y `baggage` adjuntos a cualquier solicitud HTTP saliente.

  La opción puede contener una lista de strings o expresiones regulares con las que se comparan las URL de las solicitudes salientes.
  Si una de las entradas de la lista coincide con la URL de una solicitud saliente, se adjuntarán datos de tracing a esa solicitud.
  Las entradas de tipo string no tienen que coincidir por completo, es decir, la URL de una solicitud coincide cuando *contiene* una string proporcionada mediante la opción.

  <PlatformCategorySection supported={["browser"]}>
    En el navegador, todas las solicitudes salientes al mismo origen se propagan
    de forma predeterminada.
  </PlatformCategorySection>

  <PlatformCategorySection supported={["server", "serverless"]}>
    En el servidor, todas las solicitudes salientes se propagan de forma predeterminada.
  </PlatformCategorySection>

  Si quieres desactivar la propagación de trazas, puedes establecer esta opción en `[]`.
</SdkOption>

<SdkOption name="strictTraceContinuation" type="boolean" defaultValue="false" categorySupported={["server", "serverless"]}>
  Si se establece en `true`, el SDK solo continuará una traza si el ID de la organización de la traza entrante encontrado en el
  header `baggage` coincide con el ID de la organización del cliente de Sentry actual.

  El ID de la organización del cliente se extrae del DSN o puede definirse con la <PlatformLink to={'/configuration/options#orgId'}>opción `orgId`</PlatformLink>.

  Si los IDs de organización no coinciden, el SDK iniciará una traza nueva en lugar de continuar la entrante.
  Esto es útil para evitar que se continúen en tu aplicación trazas de servicios de terceros desconocidos.
</SdkOption>

<SdkOption name="beforeSendTransaction" type="(event: TransactionEvent, hint: EventHint) => TransactionEvent | null">
  Esta función se llama con un objeto de evento de transacción y puede devolver un objeto de evento de transacción modificado, o `null` para omitir el envío del evento. Puede usarse, por ejemplo, para eliminar PII manualmente antes de enviar.
</SdkOption>

<SdkOption name="beforeSendSpan" type="(span: SpanJSON) => SpanJSON">
  Esta función se llama con un objeto de span serializado y puede devolver un objeto de span modificado. Esto puede ser útil para eliminar PII manualmente de los spans.
  Esta función se llama tanto para los spans raíz como para todos los spans secundarios.
  Si quieres descartar el span raíz, incluidos todos sus spans secundarios, usa [`beforeSendTransaction`](#beforeSendTransaction).

  Ten en cuenta que el `span` que recibes como argumento es un objeto serializado, no una instancia de la clase `Span`.
</SdkOption>

<SdkOption name="ignoreTransactions" type="Array<string | RegExp>" defaultValue="[]">
  <Include name="platforms/configuration/options/ignore-transactions.mdx" />
</SdkOption>

<SdkOption name="ignoreSpans" type="Array<string | RegExp | {name?: string | RegExp, op?: string | RegExp}>" defaultValue="[]" availableSince="10.2.0">
  Una lista de cadenas o expresiones regulares que coinciden con nombres de spans que no deben enviarse a Sentry. Al usar cadenas, se filtrarán las coincidencias parciales; si necesitas filtrar por coincidencia exacta, usa expresiones regulares. También puedes proporcionar un objeto con las propiedades `name` y `op` para hacer coincidir tanto el nombre del span como el de la operación. Ten en cuenta que se debe proporcionar al menos `name` u `op`.

  Si un span raíz coincide con cualquiera de los patrones especificados, se descartará todo el trace local. Si coincide un span hijo, sus hijos se reasignarán al span padre del span descartado.

  De forma predeterminada, no se ignora ningún span.

  Aquí tienes algunas coincidencias predefinidas para spans comunes para empezar:

  ```javascript
  Sentry.init({
    ignoreSpans: [
      // Browser connection events
      {op: /^browser\.(cache|connect|DNS)$/},

      // Fonts
      {op: 'resource.other', name: /.+\.(woff2|woff|ttf|eot)$/},

      // CSS files
      {op: 'resource.link', name: /.+\.css.*$/},

      // JS files
      {op: /resource\.(link|script)/, name: /.+\.js.*$/},

      // Images
      {op: /resource\.(other|img)/, name: /.+\.(png|svg|jpe?g|gif|bmp|tiff?|webp|avif|heic?|ico).*$/},

      // Measure spans
      {op: 'measure'},
    ]
  });
  ```
</SdkOption>

<SdkOption name="propagateTraceparent" type="boolean" defaultValue="false" availableSince="10.10.0" categorySupported={['browser']}>
  Si se establece en `true`, el SDK añade el encabezado [W3C `traceparent`](https://www.w3.org/TR/trace-context/) a las solicitudes HTTP salientes realizadas mediante `fetch` o `XMLHttpRequest`.
  Este encabezado se adjunta además de los encabezados `sentry-trace` y `baggage`.
  Configura esta opción en `true` si tus servicios de backend están instrumentados con, por ejemplo, OpenTelemetry u otras bibliotecas compatibles con W3C Trace Context y quieres continuar los traces desde el cliente.

  <PlatformSection notSupported={["javascript.cordova"]}>
    **Importante:** Asegúrate de que la configuración CORS de tus servicios de backend permita el encabezado `traceparent`.
    De lo contrario, las solicitudes podrían bloquearse.
    Usa la opción [`tracePropagationTargets`](#tracepropagationtargets) para controlar a qué solicitudes se adjunta el encabezado `traceparent`.
    Consulta <PlatformLink to="/tracing/distributed-tracing/dealing-with-cors-issues/">Cómo gestionar problemas de CORS</PlatformLink> para obtener más información.
  </PlatformSection>
</SdkOption>

  ## Opciones de logs

  <SdkOption name="enableLogs" type="boolean" defaultValue="false">
    Establece esta opción en `true` para habilitar la captura de logs en Sentry. Solo cuando esto esté habilitado, las API de `logger` enviarán logs a Sentry.
  </SdkOption>

  <SdkOption name="beforeSendLog" type="(log: Log) => Log | null">
    Esta función se invoca con un objeto de log y puede devolver un objeto de log modificado o `null` para omitir el envío de ese log a Sentry. Puede utilizarse, por ejemplo, para eliminar PII manualmente antes de enviar o para añadir atributos personalizados a todos los logs.
  </SdkOption>

  ## Opciones de Session Replay

  <SdkOption name="replaysSessionSampleRate" type="number" categorySupported={['browser']}>
    La tasa de muestreo para replays que comienzan a grabar de inmediato y abarcan toda la sesión del usuario. `1.0` recopila todos los replays y `0` no recopila ninguno.
  </SdkOption>

  <SdkOption name="replaysOnErrorSampleRate" type="number" categorySupported={['browser']}>
    La tasa de muestreo para replays que se graban cuando ocurre un error. Este tipo de replay registrará hasta un minuto de eventos previos al error y continuará grabando hasta que finalice la sesión. `1.0` recopila todas las sesiones con error y `0` no recopila ninguna.
  </SdkOption>
</PlatformCategorySection>

  ## Opciones de perfiles
</div>

<SdkOption name="profilesSampleRate" type="number">
  Un número entre `0` y `1` que controla el porcentaje de probabilidad de que una transacción muestreada sea perfilada. (`0` representa el 0% y `1` el 100%.) Se aplica por igual a todas las transacciones creadas en la app. Es relativo a la tasa de muestreo de trazas; por ejemplo, `0.5` significa que se perfilará el 50% de las transacciones muestreadas.
</SdkOption>
