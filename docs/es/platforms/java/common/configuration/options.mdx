<div id="core-options">
  ## Opciones principales
</div>

<TableOfContents ignoreIds={["core-options"]} />

Las opciones que pueden leerse desde una variable de entorno (`SENTRY_DSN`, `SENTRY_ENVIRONMENT`, `SENTRY_RELEASE`) se leen automáticamente.

<SdkOption name="dsn" type="string" envVar="SENTRY_DSN">
  El DSN le indica al SDK adónde enviar los eventos. Si no se proporciona este valor, el SDK intentará leerlo de la variable de entorno `SENTRY_DSN`. Si esa variable tampoco existe, el SDK simplemente no enviará ningún evento.

  En entornos de ejecución sin variables de proceso (como el navegador), ese mecanismo de respaldo no aplica.

  Más información sobre la [utilización de DSN](/es/product/sentry-basics/dsn-explainer/#dsn-utilization).
</SdkOption>

<SdkOption name="debug" type="boolean" defaultValue="false">
  Activa o desactiva el modo de depuración. Si `debug` está habilitado, el SDK intentará imprimir información útil de depuración si algo falla al enviar el evento. El valor predeterminado siempre es `false`. Generalmente no se recomienda activarlo en producción, aunque habilitar el modo `debug` no implica riesgos de seguridad.
</SdkOption>

<SdkOption name="diagnosticLevel" type="enum" defaultValue="DEBUG">
  Habilitar el modo `debug` hace que el SDK genere la mayor cantidad posible de datos de diagnóstico. Sin embargo, si prefieres reducir la verbosidad de los registros de diagnóstico del SDK de Sentry, configura esta opción para establecer el nivel apropiado:

  * `debug`: **predeterminado** El modo más detallado
  * `info`: Mensajes informativos
  * `warning`: Advertencia de que algo podría no estar bien
  * `error`: Solo se imprimen errores internos del SDK
  * `fatal`: Solo se imprimen errores críticos
</SdkOption>

<SdkOption name="dist" type="string">
  Establece la distribución de la aplicación. Las distribuciones se usan para diferenciar variantes de compilación o despliegue de la misma versión de una aplicación. Por ejemplo, la `dist` puede ser el número de compilación de una build de Xcode o el código de versión de una build de Android. La `dist` tiene una longitud máxima de 64 caracteres.
</SdkOption>

<SdkOption name="release" type="string" envVar="SENTRY_RELEASE">
  Establece la versión (`release`). Algunos SDK intentarán configurarla automáticamente de forma predeterminada, pero es mejor establecerla manualmente para garantizar que la versión esté sincronizada con tus integraciones de despliegue o cargas de `source maps`. Los nombres de `release` son cadenas, pero Sentry detecta algunos formatos y podría mostrarlos de forma diferente. Obtén más información sobre cómo enviar datos de `release` para que Sentry pueda informarte sobre regresiones entre versiones e identificar la fuente potencial en [la documentación de releases](/es/product/releases/) o en el <SandboxLink scenario="releases">sandbox</SandboxLink>.

  De forma predeterminada, el SDK intentará leer este valor de la variable de entorno `SENTRY_RELEASE` (en el SDK del navegador, se leerá de `window.SENTRY_RELEASE.id` si está disponible).
</SdkOption>

<SdkOption name="environment" type="string" defaultValue="production" envVar="SENTRY_ENVIRONMENT">
  Establece el entorno. Esta cadena es de formato libre y se establece en `production` de forma predeterminada. Una `release` puede asociarse con más de un entorno para separarlos en la interfaz (piensa en `staging` vs `prod` o similares).

  De forma predeterminada, el SDK intentará leer este valor de la variable de entorno `SENTRY_ENVIRONMENT` (excepto en el SDK del navegador, donde esto no aplica). Si este valor no está establecido, usará el valor predeterminado `production`.
</SdkOption>

<SdkOption name="sampleRate" type="float" defaultValue="1.0">
  Configura la tasa de muestreo para eventos de error, en el rango de `0.0` a `1.0`. El valor predeterminado es `1.0`, lo que significa que se enviará el 100% de los eventos de error. Si se establece en `0.1`, solo se enviará el 10% de los eventos de error. Los eventos se seleccionan aleatoriamente.
</SdkOption>

<SdkOption name="maxBreadcrumbs" type="integer" defaultValue="100">
  Esta variable controla la cantidad total de breadcrumbs que deben capturarse. El valor predeterminado es `100`, pero puedes establecerlo en cualquier número. Sin embargo, debes tener en cuenta que Sentry tiene un [tamaño máximo de payload](https://develop.sentry.dev/sdk/data-model/envelopes/#size-limits) y cualquier evento que exceda ese tamaño de payload se descartará.
</SdkOption>

<SdkOption name="maxCacheItems" type="integer" defaultValue="30">
  El número máximo de [envelopes](https://develop.sentry.dev/sdk/data-model/envelopes/) que se guardan en caché. Los SDK usan envelopes para enviar datos a sentry.io, como eventos, adjuntos, comentarios de usuario y sesiones. Un envelope puede contener varios elementos, por ejemplo, un evento con una sesión y dos adjuntos. Según el uso del SDK, el tamaño de un envelope puede variar. Si la cantidad de envelopes en la caché local supera `max-cache-items`, el SDK elimina el envelope más antiguo y migra las sesiones al siguiente para mantener la integridad de las estadísticas de salud de tu release. El valor predeterminado es `30`.
</SdkOption>

<SdkOption name="attachStacktrace" type="boolean" defaultValue="true">
  Cuando está habilitada, las trazas de pila se adjuntan automáticamente a todos los mensajes registrados. Las trazas de pila siempre se adjuntan a las excepciones; sin embargo, cuando esta opción está activada, las trazas de pila también se envían con los mensajes. Esta opción, por ejemplo, hace que las trazas de pila aparezcan junto a todos los mensajes de log.

  Esta opción está activada de forma predeterminada.

  El agrupamiento en Sentry es diferente para eventos con y sin traza de pila. Como resultado, obtendrás nuevos grupos al habilitar o deshabilitar esta opción para ciertos eventos.
</SdkOption>

<SdkOption name="attachThreads" type="boolean" defaultValue="false">
  Cuando está habilitada, se adjunta información sobre todos los hilos a los eventos, no solo sobre el hilo que falla. Esto ofrece una visión más completa del estado de la aplicación en el momento del error, incluidas las trazas de pila de todos los hilos en ejecución.

  Esta opción está desactivada de forma predeterminada.

  Ten en cuenta que habilitar esta opción puede aumentar el tamaño de la carga útil de los eventos enviados a Sentry.
</SdkOption>

<SdkOption name="sendDefaultPii" type="boolean" defaultValue="false">
  Si esta opción está habilitada, ciertas informaciones de identificación personal (PII) son añadidas por las integraciones activas. De forma predeterminada, no se envían dichos datos.

  Esta opción está desactivada de forma predeterminada.

  Si la habilitas, asegúrate de eliminar manualmente lo que no quieras enviar usando nuestras funciones para gestionar [*Sensitive Data*](../../data-management/sensitive-data/).
</SdkOption>

<SdkOption name="serverName" type="string">
  Esta opción se puede usar para especificar un nombre de servidor. Cuando se proporciona, el nombre del servidor se envía y se conserva en el evento. En muchas integraciones, el nombre del servidor corresponde al nombre de host del dispositivo, incluso cuando la máquina no es realmente un servidor.

  La mayoría de los SDK intentarán detectar este valor automáticamente.
</SdkOption>

<SdkOption name="inAppIncludes" type="array">
  Lista de prefijos de cadena de nombres de módulos que pertenecen a la app. Esta opción tiene prioridad sobre `in-app-excludes`.

  Sentry diferencia los frames de la pila que están directamente relacionados con tu aplicación (“in application”) de los que provienen de otros paquetes como la biblioteca estándar, frameworks u otras dependencias. El paquete de la aplicación se marca automáticamente como `inApp`. La diferencia es visible en [sentry.io](https://sentry.io), donde, de forma predeterminada, solo se muestran los frames “in application”.
</SdkOption>

<SdkOption name="inAppExcludes" type="array">
  Lista de prefijos de cadena de nombres de módulos que no pertenecen a la app, sino a paquetes de terceros. Los módulos considerados fuera de la app se ocultarán de las trazas de pila de forma predeterminada.

  Esta opción se puede anular usando <PlatformIdentifier name="inAppIncludes" />.
</SdkOption>

<SdkOption name="ignoredExceptionsForType" type="array">
  Lista de tipos de excepciones que se filtrarán antes de enviarlas a Sentry.
</SdkOption>

<SdkOption name="maxRequestBodySize" type="enum" defaultValue="NONE">
  Este parámetro controla si las integraciones deben capturar los cuerpos de las solicitudes HTTP. Se puede configurar con uno de los siguientes valores:

  * `none`: valor predeterminado. Nunca se envían cuerpos de solicitud.
  * `small`: Solo se capturarán cuerpos de solicitud pequeños. El límite para “small” depende del SDK (normalmente 4 KB).
  * `medium`: Se capturarán solicitudes medianas y pequeñas (normalmente 10 KB).
  * `always`: El SDK siempre capturará el cuerpo de la solicitud siempre que Sentry pueda interpretarlo.

  Para capturar los cuerpos de solicitud, la opción <PlatformIdentifier name="sendDefaultPii" /> también debe estar habilitada.
  Consulta la <PlatformLink to="/configuration/options/#sendDefaultPii">documentación de <PlatformIdentifier name="sendDefaultPii" /></PlatformLink> para evaluar si deseas habilitarla, asegurándote de eliminar manualmente los datos que no quieras enviar usando nuestras funciones para <PlatformLink to="/data-management/sensitive-data">gestionar Sensitive Data</PlatformLink>.
</SdkOption>

<SdkOption name="enabled" type="boolean" defaultValue="true">
  Especifica si este SDK debe enviar eventos a Sentry. De forma predeterminada es `true`. Configurarlo como `enabled: false` no elimina toda la sobrecarga de la instrumentación de Sentry. Para desactivar Sentry por completo, según el entorno, llama a `Sentry.init` de forma condicional.
</SdkOption>

<SdkOption name="sendClientReports" type="boolean" defaultValue="true" availableSince="6.0.0">
  Establece este booleano en `false` para desactivar el envío de informes del cliente. Los informes del cliente son una función del protocolo que permite a los clientes enviar a Sentry informes de estado sobre sí mismos. Actualmente se utilizan principalmente para emitir resultados de eventos que nunca se enviaron.

  *(Nuevo en la versión 6.0.0)*
</SdkOption>

<SdkOption name="forceInit" type="boolean" defaultValue="false" availableSince="8.0.0">
  Establece este booleano en `true` para forzar que una llamada a `Sentry.init` vuelva a inicializar el SDK, incluso si el SDK ya fue inicializado por una integración de alta prioridad.

  *(Nuevo en la versión 8.0.0)*
</SdkOption>

<SdkOption name="enableBackpressureHandling" type="boolean" defaultValue="true" availableSince="7.1.0">
  Establece este booleano en `false` para desactivar el muestreo automático a la baja de transacciones mientras el sistema está bajo carga. El `tracesSampleRate` se reduce a la mitad por cada verificación de estado fallida hasta 10 veces (aproximadamente el 0,001 % del `tracesSampleRate` original). Cualquier verificación de estado positiva restablecerá `tracesSampleRate` a su valor original. Las verificaciones de estado se ejecutan en segundo plano cada 10 segundos, comprobando pérdidas en la cola y limitación de tasa. Nota: A partir de la versión 7.8.0, el manejo de backpressure pasó de opt-in a opt-out.

  *(Nuevo en la versión 7.1.0)*
</SdkOption>

<div id="integration-configuration">
  ## Configuración de integraciones
</div>

En muchos SDK de plataforma, las integraciones pueden configurarse junto con el SDK. En algunas plataformas esto sucede como parte de la llamada a `init()`, y en otras se aplican patrones diferentes.

<SdkOption name="integrations" type="array">
  En algunos SDK, las integraciones se configuran mediante este parámetro durante la inicialización de la biblioteca. Para obtener más información, consulta la documentación de la integración específica.
</SdkOption>

<div id="hooks">
  ## Hooks
</div>

Estas opciones se pueden usar para integrar el SDK de varias maneras para personalizar el envío de eventos o para monitorear el tipo y la cantidad de datos descartados.

<SdkOption name="beforeSend" type="function">
  Esta función se invoca con la carga útil del evento y puede devolver un objeto de evento modificado, o `null` para omitir el envío del evento. Por ejemplo, se puede usar para eliminar manualmente PII antes de enviar.

  Para cuando se ejecuta <PlatformIdentifier name="beforeSend" />, todos los datos del scope ya se han aplicado al evento. Cualquier modificación posterior del scope no tendrá efecto.
</SdkOption>

<SdkOption name="beforeSendTransaction" type="function">
  Esta función se invoca con un objeto de evento de transacción y puede devolver un objeto de evento de transacción modificado, o `null` para omitir el envío del evento. Una forma de usarlo es eliminar manualmente PII antes de enviar.
</SdkOption>

<SdkOption name="beforeBreadcrumb" type="function">
  Esta función se invoca con un objeto breadcrumb antes de que el breadcrumb se agregue al scope. Si la función no devuelve nada, el breadcrumb se descarta. Para permitir el breadcrumb, devuelve el primer argumento, que contiene el objeto breadcrumb.
  La función de retorno (callback) normalmente recibe un segundo argumento (llamado &quot;hint&quot;) que contiene el objeto original a partir del cual se creó el breadcrumb, para personalizar mejor cómo debería verse el breadcrumb.
</SdkOption>

<SdkOption name="onDiscard" type="function">
  Esta función se invoca cuando se descartan datos, por motivos como muestreo, errores de red o desbordamientos de la cola. Los argumentos de la función ofrecen información sobre el motivo del descarte, el tipo de datos descartados y la cantidad de elementos descartados.
</SdkOption>

<div id="transport-options">
  ## Opciones de transporte
</div>

Los transportes se usan para enviar eventos a Sentry. Pueden personalizarse hasta cierto punto para adaptarse mejor a despliegues muy específicos.

<SdkOption name="transport" type="function">
  Sustituye el transporte usado para enviar eventos. Por ejemplo, puede utilizarse para capturar eventos en pruebas unitarias o para enviarlos mediante una configuración más compleja que requiera autenticación de proxy.
</SdkOption>

<SdkOption name="httpProxy" type="string">
  Si se establece, puede configurarse un proxy que se usará para las solicitudes salientes. También se utilizará para solicitudes HTTPS, a menos que se configure por separado un `https-proxy`. No todos los SDK admiten un proxy HTTPS independiente. Los SDK intentarán usar, si es posible, el proxy configurado a nivel del sistema por defecto. Por ejemplo, en sistemas Unix se detectará la variable de entorno `http_proxy`.
</SdkOption>

<SdkOption name="shutdownTimeoutMillis" type="integer" defaultValue="2000">
  Controla cuántos segundos esperar antes de cerrar. El SDK envía eventos desde una cola en segundo plano. A esta cola se le concede un tiempo determinado para vaciar los eventos pendientes. El valor predeterminado es de dos segundos. Establecer un valor demasiado bajo puede causar problemas al enviar eventos desde aplicaciones de línea de comandos. Establecerlo demasiado alto hará que la aplicación se bloquee durante mucho tiempo para usuarios que experimenten problemas de conectividad de red.
</SdkOption>

<div id="tracing-options">
  ## Opciones de tracing
</div>

<SdkOption name="tracesSampleRate" type="float">
  Un número entre `0` y `1` que controla el porcentaje de probabilidad de que una transacción se envíe a Sentry. (`0` representa el 0% y `1` el 100%.) Se aplica por igual a todas las transacciones creadas en la app. Para habilitar el tracing, debes definir esta opción o <PlatformIdentifier name="tracesSampler" />.
</SdkOption>

<SdkOption name="tracesSampler" type="function">
  Función responsable de determinar el porcentaje de probabilidad de que una transacción se envíe a Sentry. Recibe automáticamente información sobre la transacción y el contexto en el que se crea, y debe devolver un número entre `0` (0% de probabilidad de envío) y `1` (100% de probabilidad de envío). También puede usarse para filtrar transacciones devolviendo 0 para aquellas no deseadas. Para habilitar el tracing, debes definir esta opción o <PlatformIdentifier name="tracesSampleRate" />.
</SdkOption>

<SdkOption name="tracePropagationTargets" type="array">
  Propiedad opcional que controla qué servicios downstream reciben datos de tracing, mediante los encabezados `sentry-trace` y `baggage` adjuntos a cualquier solicitud HTTP saliente.

  La opción puede contener una lista de cadenas o expresiones regulares contra las cuales se comparan las URL de las solicitudes salientes.
  Si alguna entrada de la lista coincide con la URL de una solicitud saliente, se adjuntarán datos de tracing a esa solicitud.
  Las entradas de tipo cadena no tienen que coincidir por completo; es decir, la URL de una solicitud coincide cuando *contiene* una cadena indicada en la opción.

  Si no se proporciona <PlatformIdentifier name="tracePropagationTargets" />, se adjuntan datos de tracing a todas las solicitudes salientes del cliente instrumentado.
</SdkOption>

<SdkOption name="propagateTraceparent" type="boolean" defaultValue="false" availableSince="8.22.0">
  Controla si el SDK debe propagar el encabezado HTTP W3C `traceparent` junto con los encabezados `sentry-trace` y `baggage` para tracing distribuido. De forma predeterminada, esta opción es `false` y está disponible a partir de la versión 8.22.0 del SDK.
</SdkOption>

<SdkOption name="traceOptionsRequests" type="boolean" defaultValue="true">
  Establece este booleano en `false` para deshabilitar el tracing de solicitudes `OPTIONS`. Es probable que el valor predeterminado de esta opción cambie en la próxima versión principal; en ese caso, tendrás que establecerla en `true` si quieres mantener el tracing de solicitudes `OPTIONS`.
</SdkOption>
