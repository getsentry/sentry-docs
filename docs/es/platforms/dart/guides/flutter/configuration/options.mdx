<div id="core-options">
  ## Opciones principales
</div>

<TableOfContents ignoreIds={["core-options"]} />

Las opciones que pueden leerse desde una variable de entorno (`SENTRY_DSN`, `SENTRY_ENVIRONMENT`, `SENTRY_RELEASE`) se leen automáticamente.

<SdkOption name="dsn" type="string" envVar="SENTRY_DSN">
  El DSN le indica al SDK adónde enviar los eventos. Si no se proporciona este valor, el SDK intentará leerlo de la variable de entorno `SENTRY_DSN`. Si esa variable tampoco existe, el SDK no enviará ningún evento.

  En entornos de ejecución sin variables de proceso (como el navegador), esa alternativa no aplica.

  Más información sobre la [utilización del DSN](/es/product/sentry-basics/dsn-explainer/#dsn-utilization).
</SdkOption>

<SdkOption name="debug" type="bool" defaultNote="true in debug mode, false otherwise">
  Activa o desactiva el modo de depuración. Si `debug` está habilitado, el SDK intentará imprimir información de depuración útil si algo sale mal al enviar el evento. El valor predeterminado es `true` si tu app se ejecuta en [modo debug](https://docs.flutter.dev/testing/build-modes#debug); de lo contrario, es `false`. Por lo general, no se recomienda activarlo en producción, aunque habilitar el modo `debug` no implica riesgos de seguridad.
</SdkOption>

<SdkOption name="diagnosticLevel" type="enum" defaultValue="warning">
  Habilitar el modo `debug` hace que el SDK genere la mayor cantidad posible de datos de diagnóstico. Sin embargo, si prefieres reducir la verbosidad de los registros de diagnóstico del SDK de Sentry, configura esta opción para establecer el nivel adecuado:

  * `debug`: El modo más detallado
  * `info`: Mensajes informativos
  * `warning`:  **predeterminado** Advertencias de que algo podría no estar bien
  * `error`: Solo se imprimen errores internos del SDK
  * `fatal`: Solo se imprimen errores críticos
</SdkOption>

<SdkOption name="dist" type="string">
  Establece la distribución de la aplicación. Las distribuciones se usan para diferenciar variantes de compilación o despliegue de la misma release de una aplicación. Por ejemplo, la distribución puede ser el número de compilación de una build de Xcode o el código de versión de una build de Android. La distribución tiene una longitud máxima de 64 caracteres.
</SdkOption>

<SdkOption name="release" type="string" envVar="SENTRY_RELEASE">
  Establece la release. Algunos SDK intentarán configurar automáticamente una release lista para usar, pero es mejor establecerla manualmente para garantizar que la release esté sincronizada con tus integraciones de despliegue o cargas de source maps. Los nombres de release son cadenas, pero Sentry detecta algunos formatos y podría mostrarlos de manera diferente. Obtén más información sobre cómo enviar datos de release para que Sentry pueda informarte sobre regresiones entre releases e identificar la fuente potencial en [la documentación de releases](/es/product/releases/) o en el <SandboxLink scenario="releases">sandbox</SandboxLink>.

  De forma predeterminada, el SDK intentará leer este valor de la variable de entorno `SENTRY_RELEASE` (en el SDK del navegador, se leerá de `window.SENTRY_RELEASE.id` si está disponible).
</SdkOption>

<SdkOption name="environment" type="string" envVar="SENTRY_ENVIRONMENT">
  Establece el entorno. Esta cadena es de formato libre y no está definida de forma predeterminada. Una release puede asociarse con más de un entorno para separarlos en la interfaz (piensa en `staging` vs `prod` o similar).

  De forma predeterminada, el SDK intentará leer este valor de la variable de entorno `SENTRY_ENVIRONMENT` (excepto en el SDK del navegador, donde esto no aplica).
</SdkOption>

<SdkOption name="sampleRate" type="float" defaultValue="1.0">
  Configura la tasa de muestreo para eventos de error, en el rango de `0.0` a `1.0`. El valor predeterminado es `1.0`, lo que significa que se enviará el 100% de los eventos de error. Si se establece en `0.1`, solo se enviará el 10% de los eventos de error. Los eventos se seleccionan aleatoriamente.
</SdkOption>

<SdkOption name="maxBreadcrumbs" type="int" defaultValue="100">
  Esta variable controla la cantidad total de breadcrumbs que deben capturarse. El valor predeterminado es `100`, pero puedes establecerlo en cualquier número. Ten en cuenta, sin embargo, que Sentry tiene un [tamaño máximo de payload](https://develop.sentry.dev/sdk/data-model/envelopes/#size-limits) y cualquier evento que exceda ese tamaño de payload se descartará.
</SdkOption>

<SdkOption name="maxCacheItems" type="int" defaultValue="30">
  El número máximo de [envelopes](https://develop.sentry.dev/sdk/data-model/envelopes/) que se mantendrán en caché. Los SDK usan envelopes para enviar datos, como eventos, adjuntos, comentarios de usuario y sesiones a sentry.io. Un envelope puede contener varios elementos, como un evento con una sesión y dos adjuntos. Según cómo se use el SDK, el tamaño de un envelope puede variar. Si la cantidad de envelopes en la caché local supera `max-cache-items`, el SDK elimina el envelope más antiguo y migra las sesiones al siguiente para mantener la integridad de las estadísticas de salud de tu versión. El valor predeterminado es `30`.
</SdkOption>

<SdkOption name="attachStacktrace" type="bool" defaultValue="true">
  Cuando está habilitada, las trazas de pila se adjuntan automáticamente a todos los mensajes registrados. Las trazas de pila siempre se adjuntan a las excepciones; sin embargo, cuando esta opción está activada, también se envían con los mensajes. Esto, por ejemplo, hace que las trazas de pila aparezcan junto a todos los mensajes de log.

  Esta opción está desactivada de forma predeterminada.

  La agrupación en Sentry es distinta para eventos con y sin trazas de pila. Como resultado, obtendrás grupos nuevos al activar o desactivar este indicador para ciertos eventos.
</SdkOption>

<SdkOption name="sendDefaultPii" type="bool" defaultValue="false">
  Si este indicador está habilitado, ciertas informaciones de identificación personal (PII) se agregan mediante integraciones activas. De forma predeterminada, no se envía ningún dato de este tipo.

  <Alert>
    Si usas Sentry en tu app móvil, lee nuestras [preguntas frecuentes sobre privacidad de datos móviles](/es/security-legal-pii/security/mobile-privacy/) para ayudarte con los detalles de privacidad de apps en Apple App Store y Google Play.
  </Alert>

  Esta opción está desactivada de forma predeterminada.

  Si habilitas esta opción, asegúrate de eliminar manualmente lo que no quieras enviar usando nuestras funciones para gestionar [*datos sensibles*](../../data-management/sensitive-data/).
</SdkOption>

<SdkOption name="serverName" type="string">
  Esta opción se puede usar para proporcionar un nombre de servidor. Cuando se proporciona, el nombre del servidor se envía y se conserva en el evento. En muchas integraciones, el nombre del servidor corresponde al hostname del dispositivo, incluso cuando la máquina no es realmente un servidor.

  La mayoría de los SDK intentarán detectar este valor automáticamente.
</SdkOption>

<SdkOption name="autoSessionTracking" type="bool" defaultValue="true">
  Cuando se establece en `true`, el SDK enviará eventos de sesión a Sentry. Esto es compatible en todos los SDK de navegador, emitiendo una sesión por carga y navegación de página. En los SDK móviles, cuando la app pasa a segundo plano por más de 30 segundos, las sesiones finalizan.
</SdkOption>

<SdkOption name="considerInAppFramesByDefault" type="bool">
  Configura si, de forma predeterminada, los frames de la traza de pila se consideran como frames de la app.
  Puedes usar esto para convertir `inAppIncludes` o `inAppExcludes` en una lista de permitidos o de denegados.
  Este valor solo se usa si Sentry no puede encontrar el origen del frame.

  * Si `considerInAppFramesByDefault` se establece en `true`, solo necesitas mantener `inAppExcludes`.
  * Por el contrario, si `considerInAppFramesByDefault` se establece en `false`, solo necesitas mantener `inAppIncludes`.
</SdkOption>

<SdkOption name="inAppInclude" type="array">
  Una lista de prefijos de cadena de nombres de módulos que pertenecen a la app. Esta opción tiene prioridad sobre `in-app-exclude`.

  Sentry diferencia los frames de pila que están directamente relacionados con tu aplicación (&quot;en la aplicación&quot;) de los que provienen de otros paquetes, como la biblioteca estándar, frameworks u otras dependencias. El paquete de la aplicación se marca automáticamente como `inApp`. La diferencia es visible en [sentry.io](https://sentry.io), donde, de forma predeterminada, solo se muestran los frames &quot;en la aplicación&quot;.
</SdkOption>

<SdkOption name="inAppExclude" type="array">
  Una lista de prefijos de cadena de nombres de módulos que no pertenecen a la app, sino a paquetes de terceros. Los módulos considerados fuera de la app se ocultarán de las trazas de pila de forma predeterminada.

  Esta opción se puede anular usando <PlatformIdentifier name="inAppInclude" />.
</SdkOption>

<SdkOption name="attachScreenshot" type="bool" defaultValue="false">
  Toma una captura de pantalla de la aplicación cuando ocurre un error y la incluye como adjunto.
  Activa esta opción configurando <PlatformIdentifier name="attachScreenshot" /> en `true` y envolviendo tu widget raíz con `SentryWidget(child: MyApp())`.
  Obtén más información sobre cómo enriquecer eventos con capturas de pantalla en nuestra <PlatformLink to="/enriching-events/screenshots/">documentación sobre capturas de pantalla</PlatformLink>.
</SdkOption>

<SdkOption name="screenshotQuality" type="enum" defaultValue="high">
  La calidad de la captura de pantalla adjunta. Se puede configurar en `full`, `high`, `medium` o `low`.
</SdkOption>

<SdkOption name="attachViewHierarchy" type="bool" defaultValue="false">
  Genera una representación JSON de toda la jerarquía de vistas de la aplicación cuando ocurre un error y la incluye como archivo adjunto.
  Obtén más información sobre cómo enriquecer eventos con la jerarquía de vistas en nuestra <PlatformLink to="/enriching-events/viewhierarchy/">documentación de View Hierarchy</PlatformLink>.
</SdkOption>

<SdkOption name="idleTimeout" type="int" defaultValue="3000">
  El tiempo de inactividad, medido en ms, que se espera antes de finalizar automáticamente una transacción. La transacción usará la marca de tiempo de finalización del último span terminado como hora de finalización de la transacción.

  El valor predeterminado es `3000`.
</SdkOption>

<SdkOption name="sendClientReports" type="bool" defaultValue="true" availableSince="6.6.0">
  Establece este valor booleano en `false` para desactivar el envío de informes del cliente. Los informes del cliente son una función del protocolo que permite a los clientes enviar informes de estado sobre sí mismos a Sentry. Actualmente se usan principalmente para emitir resultados de eventos que nunca se enviaron.

  *(Nuevo en la versión 6.6.0)*
</SdkOption>

<SdkOption name="captureFailedRequests" type="bool" defaultValue="true" availableSince="7.0.0">
  Una vez habilitada, esta función captura automáticamente errores del cliente HTTP, como códigos de respuesta no satisfactorios, como eventos de error y los reporta a Sentry.

  *(Nuevo en la versión 7.0.0)*
</SdkOption>

<SdkOption name="maxRequestBodySize" type="enum" defaultValue="never">
  Este parámetro controla si las integraciones deben capturar los cuerpos de las solicitudes HTTP. Se puede configurar en uno de los siguientes valores:

  * `never`: Los cuerpos de las solicitudes nunca se envían.
  * `small`: Solo se capturarán cuerpos de solicitud pequeños. El umbral para &quot;small&quot; depende del SDK (normalmente 4 KB).
  * `medium`: Se capturarán cuerpos de solicitud medianos y pequeños (normalmente 10 KB).
  * `always`: El SDK siempre capturará el cuerpo de la solicitud siempre que Sentry pueda interpretarlo.
</SdkOption>

<div id="integration-configuration">
  ## Configuración de integraciones
</div>

En muchos SDK de la plataforma, las integraciones pueden configurarse junto con el SDK. En algunas plataformas esto ocurre como parte de la llamada a `init()`, y en otras se aplican patrones diferentes.

<SdkOption name="integrations" type="array">
  En algunos SDK, las integraciones se configuran mediante este parámetro durante la inicialización de la biblioteca. Para obtener más información, consulta la documentación de la integración específica.
</SdkOption>

<div id="hooks">
  ## Hooks
</div>

Estas opciones pueden usarse para enganchar el SDK de varias maneras y personalizar el envío de eventos.

<SdkOption name="beforeSend" type="function">
  Esta función se invoca con un mensaje específico del SDK o un objeto de evento de error, y puede devolver un objeto de evento modificado o `null` para omitir el envío del evento. Puede usarse, por ejemplo, para eliminar manualmente PII antes de enviar.

  Para cuando se ejecuta <PlatformIdentifier name="beforeSend" />, todos los datos del scope ya se han aplicado al evento. Cualquier modificación posterior del scope no tendrá efecto.
</SdkOption>

<SdkOption name="beforeBreadcrumb" type="function">
  Esta función se invoca con un objeto breadcrumb específico del SDK antes de que el breadcrumb se agregue al scope. Si la función no devuelve nada, el breadcrumb se descarta. Para dejar pasar el breadcrumb, devuelve el primer argumento, que contiene el objeto del breadcrumb.
  El callback normalmente recibe un segundo argumento (llamado &quot;hint&quot;) que contiene el objeto original a partir del cual se creó el breadcrumb para personalizar aún más cómo debería verse.
</SdkOption>

<div id="transport-options">
  ## Opciones de transporte
</div>

Los transportes se usan para enviar eventos a Sentry. Se pueden personalizar hasta cierto punto para adaptarse mejor a despliegues muy específicos.

<SdkOption name="transport" type="function">
  Sustituye el transporte utilizado para enviar eventos. Su funcionamiento depende del SDK. Por ejemplo, puede usarse para capturar eventos en pruebas unitarias o para enviarlos mediante una configuración más compleja que requiera autenticación por proxy.
</SdkOption>

<div id="tracing-options">
  ## Opciones de trazas
</div>

<SdkOption name="tracesSampleRate" type="float">
  Un número entre `0` y `1` que controla el porcentaje de probabilidad de que una transacción determinada se envíe a Sentry. (`0` representa 0% y `1` representa 100%.) Se aplica por igual a todas las transacciones creadas en la app. Debe definirse esta opción o <PlatformIdentifier name="tracesSampler" /> para habilitar las trazas.
</SdkOption>

<SdkOption name="tracesSampler" type="function">
  Una función responsable de determinar el porcentaje de probabilidad de que una transacción determinada se envíe a Sentry. Recibirá automáticamente información sobre la transacción y el contexto en el que se está creando, y debe devolver un número entre `0` (0% de probabilidad de envío) y `1` (100% de probabilidad de envío). También puede usarse para filtrar transacciones devolviendo 0 para las no deseadas. Debe definirse esta opción o <PlatformIdentifier name="tracesSampleRate" /> para habilitar las trazas.
</SdkOption>

<SdkOption name="tracePropagationTargets" type="array">
  Una propiedad opcional que controla qué servicios de destino reciben datos de trazas, en forma de un encabezado `sentry-trace` y un encabezado `baggage` adjuntos a cualquier solicitud HTTP saliente.

  La opción puede contener una lista de cadenas o expresiones regulares con las que se comparan las URL de las solicitudes salientes.
  Si una de las entradas de la lista coincide con la URL de una solicitud saliente, se adjuntarán datos de trazas a esa solicitud.
  Las entradas de tipo cadena no tienen que coincidir por completo, lo que significa que la URL de una solicitud coincide cuando *contiene* una cadena proporcionada mediante la opción.

  Si no se proporciona <PlatformIdentifier name="tracePropagationTargets" />, los datos de trazas se adjuntan a todas las solicitudes salientes del cliente instrumentado.
</SdkOption>

<SdkOption name="propagateTraceparent" type="bool" defaultValue="false" availableSince="9.7.0">
  Controla si el SDK debe propagar el encabezado HTTP W3C `traceparent` junto con los encabezados `sentry-trace` y `baggage` en las solicitudes HTTP salientes.
</SdkOption>

<div id="experimental-features">
  ## Funciones experimentales
</div>

<SdkOption name="experimental" type="object">
  Propiedad opcional que determina qué funciones están en modo experimental. Según el lenguaje, esta propiedad puede ser un `Object Type` con propiedades o un `TypedDict` de pares clave/valor. Las funciones experimentales siguen en desarrollo y pueden tener errores. Sí, lo sabemos: es irónico.
</SdkOption>

<div id="hybrid-sdk-options">
  ## Opciones del SDK híbrido
</div>

<SdkOption name="autoInitializeNativeSdk" type="bool" defaultValue="true">
  Configura este booleano en `false` para desactivar la inicialización automática del SDK de la capa nativa. Al hacerlo, tendrás que inicializar el SDK nativo manualmente. No uses esto para desactivar la capa nativa.

  Debes seguir la [guía de inicialización nativa](/es/platforms/dart/guides/flutter/native-init/) si eliges usar esta opción.
</SdkOption>

<SdkOption name="enableNativeCrashHandling" type="bool" defaultValue="true">
  Configura este booleano en `false` para desactivar el manejo de bloqueos graves desde la capa nativa. Al hacerlo, el SDK no capturará eventos de bloqueos graves en Android e iOS si el error fue causado por código nativo.
</SdkOption>

<SdkOption name="enableAutoSessionTracking" type="bool" defaultValue="true">
  Configura este booleano en `false` para desactivar la función de [salud de la versión](/es/product/releases/health/).
</SdkOption>

<SdkOption name="sessionTrackingIntervalMillis" type="int" defaultValue="30000">
  Configura esto para cambiar el intervalo predeterminado para finalizar una sesión (salud de la versión) si la app pasa a segundo plano. El valor predeterminado es 30.000.
</SdkOption>

<SdkOption name="enableNdkScopeSync" type="bool" defaultValue="true">
  Configura este booleano en `false` para desactivar la sincronización del scope de Java a NDK en Android.
</SdkOption>

<SdkOption name="attachThreads" type="bool" defaultValue="false">
  Configura este booleano en `true` para adjuntar automáticamente todos los hilos a todos los eventos registrados en Android.
</SdkOption>

<SdkOption name="enableAutoPerformanceTracing" type="bool" defaultValue="true">
  Configura este booleano en `false` para desactivar el seguimiento automático de [tracing](/es/product/insights/overview/).
</SdkOption>

<SdkOption name="enableWatchdogTerminationTracking" type="bool" defaultValue="true">
  <Alert>
    Disponible desde la versión 8.0.0 del Sentry Apple SDK. Antes de la versión 8.0.0 se llamaba
    `enableOutOfMemoryTracking`.
  </Alert>

  Configura este booleano en `false` para desactivar el seguimiento de [terminaciones por watchdog](/es/platforms/apple/guides/ios/configuration/watchdog-terminations/) en iOS.
</SdkOption>

<SdkOption name="enableScopeSync" type="bool" defaultValue="true">
  Configura este booleano en `false` para desactivar la sincronización de datos de `Scope` con los SDK de Android e iOS.
</SdkOption>

<SdkOption name="reportPackages" type="bool" defaultValue="true">
  Configura este booleano en `false` para desactivar el reporte de todas las dependencias de paquetes.
</SdkOption>

<SdkOption name="anrEnabled" type="bool" defaultValue="true">
  Configura este booleano en `true` para habilitar la detección de ANR (Application Not Responding) en Android.
</SdkOption>

<SdkOption name="anrTimeoutInterval" type="int" defaultValue="5000">
  Configura esto para cambiar el intervalo predeterminado de la detección de ANR. El valor predeterminado es `5` segundos.
</SdkOption>

<SdkOption name="reportSilentFlutterErrors" type="bool" defaultValue="false">
  Configura este booleano en `true` para habilitar el reporte automático de errores [FlutterErrorDetails.silent](https://api.flutter.dev/flutter/foundation/FlutterErrorDetails/silent.html).
</SdkOption>

<SdkOption name="enableAutoNativeBreadcrumbs" type="bool" defaultValue="true">
  Configura este booleano en `false` para desactivar las breadcrumbs automáticas en las plataformas nativas.
</SdkOption>

<SdkOption name="enableUserInteractionBreadcrumbs" type="bool" defaultValue="true">
  Configura este booleano en `false` para desactivar las breadcrumbs automáticas de interacciones de usuario.
</SdkOption>

<SdkOption name="enableUserInteractionTracing" type="bool" defaultValue="true">
  Configura este booleano en `false` para desactivar el tracing automático de interacciones de usuario.
</SdkOption>

<SdkOption name="proguardUuid" type="string">
  Configura el UUID de Proguard para la plataforma Android
</SdkOption>
