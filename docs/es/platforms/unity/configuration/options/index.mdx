<div id="programmatic-configuration">
  ## Configuración por código
</div>

El SDK ofrece una ventana de configuración que te permite establecer la mayoría de las opciones de Sentry directamente en el Editor de Unity. Algunas opciones, como el callback `BeforeSend`, solo pueden configurarse por código. Consulta nuestra documentación sobre <PlatformLink to="/configuration/options/programmatic-configuration/">cómo configurar el SDK por código</PlatformLink> para más información.

<div id="core-options">
  ## Opciones principales
</div>

<TableOfContents ignoreIds={["core-options", "programmatic-configuration"]} />

Las opciones que pueden leerse desde una variable de entorno (`SENTRY_DSN`, `SENTRY_ENVIRONMENT`, `SENTRY_RELEASE`) se cargan automáticamente.

<SdkOption name="Dsn" type="string" envVar="SENTRY_DSN">
  El DSN indica al SDK dónde enviar los eventos. Si no se proporciona este valor, el SDK intentará leerlo de la variable de entorno `SENTRY_DSN`. Si esa variable tampoco existe, el SDK simplemente no enviará eventos.

  En entornos de ejecución sin variables de proceso (como el navegador), ese mecanismo de respaldo no aplica.

  Obtén más información sobre el [uso del DSN](/es/product/sentry-basics/dsn-explainer/#dsn-utilization).
</SdkOption>

<SdkOption name="Environment" type="string" defaultValue="production" envVar="SENTRY_ENVIRONMENT">
  Configura el entorno. Esta cadena es de formato libre y tiene un valor predeterminado. Una versión puede asociarse a más de un entorno para diferenciarlos en la interfaz (por ejemplo, `staging` vs `prod`).

  De forma predeterminada, el SDK informa `editor` cuando se ejecuta dentro de Unity Editor. En caso contrario, el entorno predeterminado es `production`.
</SdkOption>

<SdkOption name="Release" type="string">
  Configura la versión (release). El SDK intentará configurar automáticamente una versión lista para usar. Los nombres de versión son cadenas, pero Sentry detecta algunos formatos y puede representarlos de forma diferente. Obtén más información sobre cómo enviar datos de versiones para que Sentry pueda informarte sobre regresiones entre versiones e identificar la posible fuente en [la documentación de versiones](/es/product/releases/) o en el <SandboxLink scenario="releases">sandbox</SandboxLink>.

  De forma predeterminada, el SDK leerá `Application.productName` y `Application.version` para crear la versión con el formato `$"{Application.productName}@{Application.version}".`
</SdkOption>

<SdkOption name="Debug" type="bool" defaultValue="false">
  Activa o desactiva el modo de depuración. Si la depuración está habilitada, el SDK intentará imprimir información útil de diagnóstico si algo sale mal al enviar el evento. El valor predeterminado siempre es `false`. Por lo general, no se recomienda activarlo en producción, aunque activar el modo `debug` no supone ningún riesgo de seguridad.
</SdkOption>

<SdkOption name="DiagnosticLevel" type="enum">
  Habilitar el modo `debug` hace que el SDK genere la mayor cantidad posible de datos de diagnóstico. Sin embargo, si prefieres reducir la verbosidad de los registros de diagnóstico del SDK de Sentry, configura esta opción para establecer el nivel adecuado:

  * `debug`: **predeterminado** El modo más detallado
  * `info`: Mensajes informativos
  * `warning`: Advertencia de que algo podría no estar bien
  * `error`: Solo se imprimen errores internos del SDK
  * `fatal`: Solo se imprimen errores críticos

  Para modelos de aplicación que no tienen una consola donde imprimir, puedes <PlatformLink to="/configuration/diagnostic-logger/">personalizar el registrador de diagnóstico del SDK</PlatformLink> para escribir en un archivo o en la ventana de depuración de Visual Studio.
</SdkOption>

<div id="data-enrichment-attachments">
  ## Enriquecimiento de datos y adjuntos
</div>

<SdkOption name="InAppInclude" type="array">
  Una lista de prefijos de cadena de nombres de módulos que pertenecen a la app. Esta opción tiene prioridad sobre `in-app-exclude`.

  Sentry distingue los frames de la pila directamente relacionados con tu juego (“en la aplicación”) de los que provienen de otros paquetes, como la biblioteca estándar, complementos de terceros u otras dependencias. El código del juego que forma parte de “Assembly-CSharp” se marca automáticamente como `inApp`. La diferencia es visible en [sentry.io](https://sentry.io), donde de forma predeterminada solo se muestran los frames “en la aplicación”.
</SdkOption>

<SdkOption name="InAppExclude" type="array">
  Una lista de prefijos de cadena de nombres de módulos que no pertenecen a la app, sino a paquetes de terceros. Los módulos considerados fuera de la app se ocultarán de las trazas de pila de forma predeterminada.

  Esta opción se puede anular usando <PlatformIdentifier name="InAppInclude" />.
</SdkOption>

<SdkOption name="MaxBreadcrumbs" type="int" defaultValue="100">
  Esta variable controla la cantidad total de breadcrumbs que deben capturarse. El valor predeterminado es `100`, pero puedes establecer cualquier número. Sin embargo, ten en cuenta que Sentry tiene un [tamaño máximo de payload](https://develop.sentry.dev/sdk/data-model/envelopes/#size-limits) y cualquier evento que exceda ese tamaño se descartará.
</SdkOption>

<SdkOption name="AttachStacktrace" type="bool" defaultValue="true">
  Cuando está habilitada, las trazas de pila se adjuntan automáticamente a todos los mensajes registrados. Las trazas de pila siempre se adjuntan a las excepciones; sin embargo, cuando esta opción está activada, las trazas de pila también se envían con los mensajes. Por ejemplo, esta opción hace que las trazas de pila aparezcan junto a todos los mensajes de log.

  El agrupamiento en Sentry difiere entre eventos con y sin trazas de pila. Como resultado, obtendrás grupos nuevos al habilitar o deshabilitar este indicador para ciertos eventos.
</SdkOption>

<SdkOption name="AttachScreenshot" type="bool" defaultValue="false">
  Toma una captura de pantalla de la aplicación cuando ocurre un error y la incluye como adjunto.
  Obtén más información sobre cómo enriquecer eventos con capturas de pantalla en nuestra <PlatformLink to="/enriching-events/screenshots/">documentación de capturas de pantalla</PlatformLink>.
</SdkOption>

<SdkOption name="ScreenshotQuality" type="enum" defaultValue="high">
  La calidad de la captura de pantalla adjunta. Se puede configurar como `full`, `high`, `medium` o `low`.
</SdkOption>

<SdkOption name="AttachViewHierarchy" type="bool" defaultValue="false">
  Captura la jerarquía de la escena activa y crea una `ViewHierarchy` cuando ocurre un error, incluyéndola como adjunto. La creación está influenciada por `MaxViewHierarchyRootObjects`, `MaxViewHierarchyDepth` y `MaxViewHierarchyObjectChildCount`.
</SdkOption>

<SdkOption name="SendClientReports" type="bool" defaultValue="true">
  Establece este booleano en `false` para desactivar el envío de client reports. Los client reports son una función del protocolo que permite a los clientes enviar a Sentry informes de estado sobre sí mismos. Actualmente se usan principalmente para emitir resultados de eventos que nunca se enviaron.
</SdkOption>

<SdkOption name="CaptureFailedRequests" type="bool" defaultValue="true">
  Una vez habilitada, esta función captura automáticamente errores del cliente HTTP, como códigos de respuesta erróneos, como eventos de error y los reporta a Sentry.
</SdkOption>

<SdkOption name="IsGlobalModeEnabled" type="bool" defaultValue="true">
  El valor predeterminado es `true`. Especifica si se debe usar el modo de gestión de alcance global. Debe ser `true` para aplicaciones cliente y `false` para aplicaciones de servidor.
</SdkOption>

<SdkOption name="Distribution" type="string">
  Establece la distribución de la aplicación. Las distribuciones se usan para diferenciar variantes de compilación o despliegue de la misma versión de una aplicación. Por ejemplo, el dist puede ser el número de compilación de una build de Xcode o el código de versión de una build de Android. El dist tiene una longitud máxima de 64 caracteres.
</SdkOption>

<div id="privacy-pii">
  ## Privacidad y PII
</div>

<SdkOption name="SendDefaultPii" type="bool" defaultValue="false">
  Si esta opción está habilitada, ciertas informaciones de identificación personal (PII) las agregan las integraciones activas. De forma predeterminada, no se envía ningún dato de este tipo.

  <Alert>
    Si usas Sentry en tu app móvil, consulta nuestras [preguntas frecuentes sobre privacidad de datos móviles](/es/security-legal-pii/security/mobile-privacy/) para ayudarte con los detalles de privacidad en Apple App Store y Google Play.
  </Alert>

  Esta opción está desactivada de forma predeterminada.

  Si la habilitas, asegúrate de eliminar manualmente lo que no quieras enviar usando nuestras funciones para gestionar [*datos sensibles*](../../data-management/sensitive-data/).
</SdkOption>

<SdkOption name="ServerName" type="string">
  Esta opción puede usarse para proporcionar un nombre de servidor. Cuando se indica, el nombre del servidor se envía y se conserva en el evento. En muchas integraciones, el nombre de servidor corresponde al nombre de host del dispositivo, incluso cuando la máquina no es realmente un servidor.
</SdkOption>

<div id="hooks">
  ## Hooks
</div>

Estas opciones se pueden usar para integrar el SDK de varias formas y personalizar la notificación de eventos.

Los callbacks que configures como hooks se ejecutarán en el hilo donde ocurrió el evento. Por lo tanto, solo puedes usar API seguras para hilos y usar API específicas de Unity únicamente después de verificar que estás en el hilo de la interfaz de usuario.

<SdkOption name="SetBeforeSend" type="function">
  Esta función se llama con un mensaje específico del SDK o un objeto de evento de error, y puede devolver un objeto de evento modificado, o `null` para omitir la notificación del evento. Esto puede usarse, por ejemplo, para eliminar manualmente PII antes de enviar.

  Para cuando se ejecute <PlatformIdentifier name="SetBeforeSend" />, ya se habrán aplicado todos los datos del scope al evento. Cualquier modificación posterior del scope no tendrá efecto.
</SdkOption>

<SdkOption name="SetBeforeBreadcrumb" type="function">
  Esta función se llama con un objeto breadcrumb específico del SDK antes de que el breadcrumb se agregue al scope. Cuando la función no devuelve nada, el breadcrumb se descarta. Para dejar pasar el breadcrumb, devuelve el primer argumento, que contiene el objeto breadcrumb.
  El callback normalmente recibe un segundo argumento (llamado “hint”) que contiene el objeto original a partir del cual se creó el breadcrumb para personalizar más cómo debería verse.
</SdkOption>

<SdkOption name="SetBeforeSendTransaction" type="function">
  Esta función se llama con un objeto de transacción específico del SDK y puede devolver un objeto de transacción modificado, o `null` para omitir la notificación de la transacción. Esto puede usarse, por ejemplo, para eliminar manualmente PII antes de enviar.

  Para cuando se ejecute <PlatformIdentifier name="SetBeforeSendTransaction" />, ya se habrán aplicado todos los datos del scope al evento y cualquier modificación posterior del scope no tendrá efecto.
</SdkOption>

<SdkOption name="SetBeforeCaptureScreenshot" type="function">
  Un callback que controla el comportamiento de captura de pantallas. Devuelve `false` para evitar que se capture una captura de pantalla para un evento específico. Esto permite un control más granular sobre cuándo se incluyen capturas de pantalla en los informes de errores.
</SdkOption>

<SdkOption name="SetBeforeCaptureViewHierarchy" type="function">
  Un callback que controla el comportamiento de captura de la jerarquía de vistas. Devuelve `false` para evitar que se capture la jerarquía de vistas para un evento específico. Esto permite un control más granular sobre cuándo se incluyen jerarquías de vistas en los informes de errores.
</SdkOption>

<div id="transport-network-options">
  ## Opciones de transporte y red
</div>

Los transports se usan para enviar eventos a Sentry. Se pueden personalizar hasta cierto punto para adaptarse mejor a implementaciones muy específicas.

<SdkOption name="Transport" type="function">
  Sustituye el transport usado para enviar eventos. Su funcionamiento depende del SDK. Por ejemplo, puede usarse para capturar eventos en pruebas unitarias o para enviarlos mediante una configuración más compleja que requiera autenticación por proxy.
</SdkOption>

<SdkOption name="MaxCacheItems" type="int" defaultValue="30">
  El número máximo de [envelopes](https://develop.sentry.dev/sdk/data-model/envelopes/) que se conservarán en caché. Los SDK usan envelopes para enviar datos, como eventos, adjuntos, comentarios de usuarios y sesiones a sentry.io. Un envelope puede contener varios elementos, como un evento con una sesión y dos adjuntos. Según cómo se use el SDK, el tamaño de un envelope puede variar. Si la cantidad de envelopes en la caché local supera `max-cache-items`, el SDK elimina el envelope más antiguo y migra las sesiones al siguiente para mantener la integridad de las estadísticas de salud de tu release. El valor predeterminado es `30`.
</SdkOption>

<SdkOption name="HttpProxy" type="string">
  Si se establece, se puede configurar un proxy que se usará para las solicitudes salientes. También se usará para solicitudes HTTPS, a menos que se configure un `https-proxy` por separado. Sin embargo, no todos los SDK admiten un proxy HTTPS independiente. Los SDK intentarán usar por defecto el proxy configurado a nivel del sistema, si es posible. Por ejemplo, en sistemas Unix se detectará la variable de entorno `http_proxy`.
</SdkOption>

<SdkOption name="InitCacheFlushTimeout" type="int" defaultValue="1">
  Cuando la caché está habilitada (es decir, se establece <PlatformIdentifier name="CacheDirectoryPath" />), esta opción controla el tiempo de espera que limita cuánto tiempo intentará el SDK vaciar la caché existente durante la inicialización. Ten en cuenta que vaciar la caché implica enviar la carga útil a Sentry en una operación bloqueante. Establecer esta opción en cero significa que Sentry **no** intentará vaciar la caché durante la inicialización, sino que lo hará cuando se encole la siguiente carga útil.

  El valor predeterminado es `1` (un) segundo.
</SdkOption>

<SdkOption name="ShutdownTimeout" type="int">
  Controla cuántos segundos esperar antes de cerrar. Los SDK de Sentry envían eventos desde una cola en segundo plano. A esta cola se le concede un tiempo determinado para vaciar los eventos pendientes. El valor predeterminado depende del SDK, pero suele rondar los dos segundos. Establecer este valor demasiado bajo puede causar problemas al enviar eventos desde aplicaciones de línea de comandos. Establecerlo demasiado alto hará que la aplicación se bloquee durante mucho tiempo para usuarios con problemas de conectividad de red.
</SdkOption>

<div id="tracing-options">
  ## Opciones de trazas
</div>

<SdkOption name="TracesSampleRate" type="float" defaultValue="0.0">
  Un número entre `0` y `1` que controla el porcentaje de probabilidad de que una transacción determinada se envíe a Sentry. (`0` representa el 0% y `1` representa el 100%.) Se aplica por igual a todas las transacciones creadas en la app. Para habilitar las trazas, debe definirse este o <PlatformIdentifier name="TracesSampler" />.
</SdkOption>

<SdkOption name="TracesSampler" type="function">
  Una función responsable de determinar el porcentaje de probabilidad de que una transacción determinada se envíe a Sentry. Recibirá automáticamente información sobre la transacción y el contexto en el que se crea, y debe devolver un número entre `0` (0% de probabilidad de envío) y `1` (100% de probabilidad de envío). También puede usarse para filtrar transacciones devolviendo 0 para aquellas no deseadas. Para habilitar las trazas, debe definirse este o <PlatformIdentifier name="TracesSampleRate" />.
</SdkOption>

<SdkOption name="AutoStartupTraces" type="bool" defaultValue="true">
  Controla si el SDK registrará automáticamente trazas del arranque de la app.
</SdkOption>

<SdkOption name="AutoSceneLoadTraces" type="bool" defaultValue="true">
  Controla si el SDK registrará automáticamente trazas al cargar escenas.
</SdkOption>
