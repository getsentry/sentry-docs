  ## Problemas de compilación

<div id="disabling-the-sentry-sdk-for-selected-platforms">
  ### Desactivar el SDK de Sentry en plataformas específicas
</div>

Si quieres desactivar el SDK para una plataforma concreta, puedes usar la [configuración mediante código](/es/platforms/unity/configuration/options/programmatic-configuration/) para ejecutar tu propio código durante la inicialización y desactivar el SDK como en el siguiente ejemplo:

```csharp
using Sentry.Unity;

public class SentryOptionConfiguration : SentryOptionsConfiguration
{
    public override void Configure(SentryUnityOptions options)
    {
#if UNITY_ANDROID
        options.Enabled = false;
#endif
    }
}

```

### No se encuentra el archivo de entrada de la compilación: &#39;**/Libraries/io.sentry.unity/Plugins/iOS/SentryNativeBridge.m&#39;

Puedes solucionar este problema creando una compilación limpia o eligiendo “Replace” cuando se te solicite. También puedes quitar el `SentryNativeBridge.m` sin uso de **UnityFramework Target &gt; Compile Sources** en la configuración del proyecto. En lugar de depender de Unity para copiar `SentryNativeBridge.m` al proyecto de Xcode generado, a partir de la versión 0.12.0 el SDK lo copia manualmente en `**/Libraries/io.sentry.unity/SentryNativeBridge.m`.

<div id="failed-to-locate-the-sentry-package">
  ### No se pudo encontrar el paquete de Sentry
</div>

Esto ocurre si copiaste algunos archivos del SDK directamente en la carpeta Assets. Puedes resolverlo instalando el SDK con UPM.

<div id="android-initialization-fails-apkobb">
  ### Android: La inicialización falla - APK+OBB
</div>

Al exportar tu proyecto de Unity como APK+OBB, el archivo de configuración `ScriptableSentryOptions` almacenado en `/Resources/Sentry` puede terminar en el archivo OBB en lugar del APK principal. Esto puede provocar problemas de inicialización si el archivo OBB no está disponible cuando el SDK intenta inicializarse automáticamente.

**Solución alternativa (requiere la versión `4.0.0 o posterior` del SDK de Unity):**

En lugar de depender de la configuración de ScriptableObject y la inicialización automática, puedes inicializar el SDK manualmente desde tu código:

```csharp
using Sentry;

public class GameInitializer : MonoBehaviour
{
    private void Start()
    {
        SentrySdk.Init(options =>
        {
            options.Dsn = "TU_DSN_AQUÍ";
            options.Debug = true;
            // Agrega otras opciones de configuración según sea necesario
        });
    }
}
```

Como en este enfoque las opciones van embebidas en el código, deja de ser necesario que `/Resources` esté disponible en el momento de la inicialización.
Antes de la versión `4.0.0`, el soporte nativo era limitado al inicializar el SDK de forma programática.

<div id="library-not-loaded-rpathsentryframeworksentry">
  ### Biblioteca no cargada: @rpath/Sentry.framework/Sentry
</div>

Si te aparece el siguiente error:

```
2021-10-29 15:21:40.011452-0400 MyApp[2180:186329] Error al cargar /var/containers/Bundle/Application/88CC4619-7C5D-4BB1-9F4B-5AAD7EC4BF9C/MyApp.app/Frameworks/UnityFramework.framework/UnityFramework:  dlopen(/var/containers/Bundle/Application/88CC4619-7C5D-4BB1-9F4B-5AAD7EC4BF9C/MyApp.app/Frameworks/UnityFramework.framework/UnityFramework, 265): Biblioteca no cargada: @rpath/Sentry.framework/Sentry
  Referenciado desde: /private/var/containers/Bundle/Application/88CC4619-7C5D-4BB1-9F4B-5AAD7EC4BF9C/MyApp.app/Frameworks/UnityFramework.framework/UnityFramework
  Motivo: imagen no encontrada
```

Esto puede ocurrir si copiaste algunos archivos del SDK directamente en la carpeta Assets y `Sentry.Unity.Editor.iOS.dll`, que se ejecuta al compilar un player de iOS, no pudo copiar `Sentry.framework` a la app final. Al habilitar el modo de depuración en la ventana del editor de Sentry, se mostrará más información en la consola de Unity.

Puedes resolver este problema instalando el SDK mediante UPM.

<div id="mono-jit-stack-trace-is-incomplete-on-crashes">
  ### El rastro de pila de Mono JIT está incompleto en caso de fallos
</div>

Esta es una [limitación conocida](https://github.com/getsentry/sentry-unity/issues/884#issuecomment-1233035296) del backend de scripting de Mono JIT, que se debe a que el código nativo no puede ver el código realmente generado en el momento del fallo. No hay archivos de depuración que podamos usar para aportar más información a los rastros de pila, ya que todo el código se genera al ejecutar el código administrado. Usar el backend de scripting IL2CPP puede proporcionar más información, si esa es una opción para ti.

<div id="sentry-didnt-capture-unitys-testhelper-function-forcecrash">
  ### Sentry no capturó la función de prueba/asistencia de Unity «ForceCrash»
</div>

Unity proporciona una función para probar fallos llamada `UnityEngine.Diagnostics.Utils.ForceCrash()`. El SDK de Sentry se suscribe a los controladores de señales y captura **todas las señales** que recibe.
Sin embargo, el resultado de esa función no activa nuestro callback.
Fuera de esta función auxiliar, Sentry no conoce ningún tipo de fallo del **mundo real** que no active nuestros controladores de señales.

<div id="screenshot-contain-artifacts">
  ### La captura de pantalla presenta artefactos
</div>

La captura de pantalla se basa en la función `ScreenCapture.CaptureScreenshotIntoRenderTexture` de Unity. Al capturar errores en el Editor, esto puede provocar artefactos o problemas de escala en la captura.
