Un lanzamiento es una versión de tu código que se despliega en un <PlatformLink to="/configuration/environments/">entorno</PlatformLink>. Cuando proporcionas a Sentry información sobre tus lanzamientos, puedes:

* Detectar problemas y regresiones introducidos en un nuevo lanzamiento
* Predecir qué commit provocó un problema y quién probablemente es responsable
* Resolver problemas incluyendo el número de la incidencia en tu mensaje de commit
* Recibir notificaciones por correo electrónico cuando tu código se despliegue

<div id="setup">
  ## Configuración
</div>

Para usar las funciones de Release Health, debes activar la feature flag `release-health` del crate `sentry` (habilitada de forma predeterminada).
Además, debes inicializar el SDK especificando opciones adicionales:

```rust
use sentry;

let _guard = sentry::init(("___PUBLIC_DSN___", sentry::ClientOptions {
    // La macro `sentry::release_name` infiere automáticamente la versión desde los metadatos de cargo, y es la forma recomendada de establecer la versión
    release: sentry::release_name!(),
    // O manualmente:
    // release: Some("my-project-name@2.3.12".into()),

    // Habilita el seguimiento automático de sesiones, según el `session_mode` configurado.
    // Si el valor de esta opción es `false`, las sesiones deben gestionarse manualmente usando `sentry::start_session` y `sentry::stop_session`.
    auto_session_tracking: true,

    // Establece el modo de seguimiento automático de sesiones.
    // Los valores posibles son `sentry::SessionMode::Application` y `sentry::SessionMode::Request`.
    // `Application` debe usarse para programas interactivos, que típicamente tienen una única sesión de larga duración que abarca toda la vida de la aplicación.
    // `Request` debe usarse para servidores, donde cada solicitud entrante se considera su propia sesión.
    session_mode: sentry::SessionMode::Application,

    ..Default::default()
}));
```

<div id="setting-a-release">
  ## Configurar una versión
</div>

Asignar el nombre de la versión etiqueta cada evento con ese nombre. Te recomendamos informar a Sentry sobre una nueva versión antes de enviar eventos con ese nombre, ya que esto habilitará algunas funciones adicionales. Obtén más información en nuestra documentación sobre [Versiones](/es/product/releases/).

Si no informas a Sentry sobre una nueva versión, Sentry creará automáticamente una versión en el sistema la primera vez que vea un evento con ese ID de versión.

Después de configurar tu SDK, puedes instalar una integración de repositorio o proporcionar manualmente a Sentry tus propios metadatos de confirmaciones. Lee nuestra documentación sobre [configurar versiones](/es/product/releases/setup/) para obtener más información sobre integraciones, asociación de confirmaciones y cómo informar a Sentry al implementar versiones.

<div id="release-health">
  ## Estado de la versión
</div>

Supervisa la [salud de las versiones](/es/product/releases/health/) observando la adopción por parte de los usuarios, el uso de la aplicación, el porcentaje de [fallos](/es/product/releases/health/#crashes) y los [datos de sesión](/es/product/releases/health/#sessions). La salud de la versión ofrece información sobre el impacto de los fallos y errores en la experiencia del usuario, y revela tendencias con cada nueva incidencia mediante los gráficos y filtros de [Detalles de la versión](/es/product/releases/release-details/).

Para supervisar la salud de la versión, el SDK envía datos de sesión.

<div id="sessions">
  ### Sesiones
</div>

Una sesión representa la interacción entre el usuario y la aplicación. Las sesiones incluyen una marca de tiempo, un estado (si la sesión fue correcta o si se bloqueó) y siempre están vinculadas a una versión.
