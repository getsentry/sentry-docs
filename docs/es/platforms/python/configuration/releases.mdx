Un `release` es una versión de tu código que se despliega en un <PlatformLink to="/configuration/environments/">entorno</PlatformLink>. Al proporcionar a Sentry información sobre tus lanzamientos, puedes:

* Identificar problemas y regresiones introducidos en un nuevo lanzamiento
* Predecir qué commit causó un problema y quién probablemente es responsable
* Resolver problemas incluyendo el número del issue en el mensaje del commit
* Recibir notificaciones por correo electrónico cuando se despliegue tu código

Incluye el `release` al inicializar el SDK. El nombre del lanzamiento no puede:

* contener saltos de línea, caracteres de tabulación, barras (`/`) ni barras invertidas (`\`)
* ser (en su totalidad) un punto (`.`), dos puntos (`..`) o un espacio ( )
* superar los 200 caracteres

El valor puede ser arbitrario, pero recomendamos el [versionado semántico](https://semver.org/), el [versionado por calendario](https://calver.org/) o el SHA del commit de Git.

Recomendamos que el nombre del lanzamiento lleve como prefijo un identificador de paquete específico del proyecto (como `"mypackage@1.0.0"`).

El comportamiento de algunas funciones depende de si un proyecto usa versionado semántico o basado en el tiempo, incluido el prefijo `"mypackage@"`:

* Detección de regresiones
* Filtrado de datos por `release:latest`

Detectamos automáticamente si un proyecto usa versionado semántico o basado en el tiempo.

<div id="setting-a-release">
  ## Configurar una versión
</div>

<PlatformContent includePath="set-release" notateUnsupported />

Cómo pongas la `release` a disposición de tu código depende de ti. Por ejemplo, podrías usar una variable de entorno que se establezca durante el proceso de compilación o durante el arranque inicial.

Si no configuras `release` en `init()`, el SDK de Sentry intentará deducirla. Primero comprobará la variable de entorno `SENTRY_RELEASE`. Si esa variable no está definida, el SDK verificará si hay un repositorio Git presente y, de ser así, tomará el SHA del último commit. Si eso no funciona, el SDK buscará variables de entorno usadas por proveedores de alojamiento como `HEROKU_SLUG_COMMIT`, `SOURCE_VERSION`, `CODEBUILD_RESOLVED_SOURCE_VERSION`, `CIRCLE_SHA1` y `GAE_DEPLOYMENT_ID`.

<div id="unlocking-more-release-features">
  ## Desbloquea más funciones de las versiones
</div>

Te recomendamos informar a Sentry sobre una nueva versión antes de enviar eventos con ese nombre de versión, ya que así se habilitan funciones adicionales, como identificar regresiones y asociar commits a versiones. Obtén más información en nuestra documentación sobre [Versiones](/es/product/releases/).

Después de configurar tu SDK, puedes instalar una integración de repositorio o proporcionar manualmente a Sentry los metadatos de tus commits. Consulta nuestra documentación sobre [cómo configurar versiones](/es/product/releases/setup/) para obtener más información sobre integraciones, asociación de commits y cómo informar a Sentry al desplegar versiones.

<div id="release-health">
  ## Estado de las versiones
</div>

Supervisa la [salud de las versiones](/es/product/releases/health/) observando la adopción por parte de los usuarios, el uso de la aplicación, el porcentaje de [cierres inesperados](/es/product/releases/health/#crashes) y los [datos de sesiones](/es/product/releases/health/#sessions). El estado de las versiones te ayudará a entender el impacto de los cierres inesperados y los errores en la experiencia del usuario, y a detectar tendencias con cada nueva incidencia mediante los gráficos y filtros de [Detalles de la versión](/es/product/releases/release-details/).

Para supervisar el estado de las versiones, el SDK envía <PlatformLink to="/configuration/sessions/">sesiones</PlatformLink>.
