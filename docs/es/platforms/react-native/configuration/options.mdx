<div id="core-options">
  ## Opciones principales
</div>

<TableOfContents ignoreIds={["core-options"]} />

Las opciones que pueden leerse desde una variable de entorno (`SENTRY_DSN`, `SENTRY_ENVIRONMENT`, `SENTRY_RELEASE`) se leen automáticamente.

<SdkOption name="dsn">
  El DSN le indica al SDK adónde enviar los eventos. Si no se proporciona este valor, el SDK intentará leerlo de la variable de entorno `SENTRY_DSN`. Si esa variable tampoco existe, el SDK simplemente no enviará ningún evento.

  En entornos de ejecución sin entorno de procesos (como el navegador) esa alternativa no aplica.

  Obtén más información sobre el [uso del DSN](/es/product/sentry-basics/dsn-explainer/#dsn-utilization).
</SdkOption>

<SdkOption name="debug">
  Activa o desactiva el modo de depuración. Si la depuración está habilitada, el SDK intentará imprimir información útil de depuración si algo sale mal al enviar el evento. El valor predeterminado es siempre `false`. En general, no se recomienda activarlo en producción, aunque activar el modo `debug` no ocasionará problemas de seguridad.
</SdkOption>

<SdkOption name="dist">
  Establece la distribución de la aplicación. Las distribuciones se usan para diferenciar variantes de compilación o despliegue de la misma versión de una aplicación. Por ejemplo, la distribución puede ser el número de compilación de un build de Xcode o el código de versión de un build de Android. La distribución tiene una longitud máxima de 64 caracteres.
</SdkOption>

<SdkOption name="release">
  Establece la versión (release). Algunos SDK intentarán configurar una versión automáticamente, pero es mejor establecerla manualmente para garantizar que esté sincronizada con tus integraciones de despliegue o cargas de mapas de origen. Los nombres de las versiones son cadenas, pero Sentry detecta algunos formatos y podría mostrarlos de forma diferente. Obtén más información sobre cómo enviar datos de versiones para que Sentry pueda informarte sobre regresiones entre versiones e identificar la fuente potencial en [la documentación de versiones](/es/product/releases/) o en el <SandboxLink scenario="releases">sandbox</SandboxLink>.

  De forma predeterminada, el SDK intentará leer este valor de la variable de entorno `SENTRY_RELEASE` (en el SDK del navegador, se leerá de `window.SENTRY_RELEASE.id` si está disponible).
</SdkOption>

<SdkOption name="environment">
  Establece el entorno. Esta cadena es de formato libre y no se configura de forma predeterminada. Una versión puede asociarse con más de un entorno para separarlos en la interfaz (piensa en `staging` vs `prod` o similar).

  De forma predeterminada, el SDK intentará leer este valor de la variable de entorno `SENTRY_ENVIRONMENT` (excepto en el SDK del navegador, donde no aplica).
</SdkOption>

<SdkOption name="sampleRate">
  Configura la tasa de muestreo para eventos de error, en el rango de `0.0` a `1.0`. El valor predeterminado es `1.0`, lo que significa que se enviará el 100% de los eventos de error. Si se establece en `0.1`, solo se enviará el 10% de los eventos de error. Los eventos se seleccionan aleatoriamente.
</SdkOption>

<SdkOption name="maxBreadcrumbs">
  Esta variable controla la cantidad total de breadcrumbs que deben capturarse. El valor predeterminado es `100`, pero puedes establecer cualquier número. Sin embargo, debes tener en cuenta que Sentry tiene un [tamaño máximo de payload](https://develop.sentry.dev/sdk/data-model/envelopes/#size-limits) y cualquier evento que exceda ese tamaño de payload se descartará.
</SdkOption>

<SdkOption name="maxCacheItems">
  El número máximo de [envelopes](https://develop.sentry.dev/sdk/data-model/envelopes/) que se mantendrán en caché. Los SDK usan envelopes para enviar datos, como eventos, archivos adjuntos, comentarios de usuarios y sesiones a sentry.io. Un envelope puede contener varios elementos, como un evento con una sesión y dos archivos adjuntos. Dependiendo del uso del SDK, el tamaño de un envelope puede variar. Si la cantidad de envelopes en la caché local supera `maxCacheItems`, el SDK elimina el envelope más antiguo y migra las sesiones al siguiente envelope para mantener la integridad de las estadísticas de salud de tu versión. El valor predeterminado es `30`.
</SdkOption>

<SdkOption name="attachStacktrace">
  Cuando está habilitado, las trazas de pila se adjuntan automáticamente a todos los mensajes registrados. Las trazas de pila siempre se adjuntan a las excepciones; sin embargo, cuando se establece esta opción, también se envían con los mensajes. Esta opción, por ejemplo, hace que las trazas de pila aparezcan junto a todos los mensajes de registro.

  Esta opción está activada de forma predeterminada.

  El agrupamiento en Sentry es diferente para eventos con y sin trazas de pila. Como resultado, obtendrás grupos nuevos al habilitar o deshabilitar esta opción para ciertos eventos.
</SdkOption>

<SdkOption name="sendDefaultPii">
  Si esta bandera está activada, las integraciones activas añadirán cierta información de identificación personal (PII). De forma predeterminada, no se envía ningún dato de este tipo.

  <Alert>
    Si usas Sentry en tu app móvil, consulta nuestras [preguntas frecuentes sobre privacidad de datos móviles](/es/security-legal-pii/security/mobile-privacy/) para ayudarte con los detalles de privacidad de apps en Apple App Store y Google Play.
  </Alert>

  Esta opción está desactivada por defecto.

  Si activas esta opción, asegúrate de eliminar manualmente lo que no quieras enviar usando nuestras funciones para gestionar [*datos sensibles*](../../data-management/sensitive-data/).
</SdkOption>

<SdkOption name="autoSessionTracking">
  Cuando se establece en `true`, el SDK enviará eventos de sesión a Sentry. Esto es compatible con todos los SDK de navegador, emitiendo una sesión por cada carga y navegación de página. En los SDK móviles, cuando la app pasa a segundo plano por más de 30 segundos, las sesiones se finalizan.
</SdkOption>

<SdkOption name="maxValueLength">
  Número máximo de caracteres que puede tener un único valor antes de truncarse (por defecto, `250`).
</SdkOption>

<SdkOption name="normalizeDepth">
  Los SDK de Sentry normalizan cualquier dato contextual hasta una profundidad dada. Cualquier dato más allá de esa profundidad se recortará y se marcará usando su tipo (`[Object]` o `[Array]`), sin recorrer más el árbol. De forma predeterminada, el recorrido se realiza hasta tres niveles de profundidad.
</SdkOption>

<SdkOption name="normalizeMaxBreadth">
  Este es el número máximo de propiedades o entradas que se incluirán en cualquier objeto o matriz al normalizar datos contextuales. Cualquier dato más allá de ese límite se descartará (por defecto, 1000).
</SdkOption>

<SdkOption name="attachScreenshot">
  Toma una captura de pantalla de la aplicación cuando ocurre un error y la incluye como adjunto.
  Obtén más información sobre cómo enriquecer eventos con capturas de pantalla en nuestra <PlatformLink to="/enriching-events/screenshots/">documentación de Capturas de pantalla</PlatformLink>.

  *(Nuevo en la versión 4.11.0)*
</SdkOption>

<SdkOption name="attachViewHierarchy">
  Genera una representación JSON de toda la jerarquía de vistas de la aplicación cuando ocurre un error y la incluye como adjunto.
  Obtén más información sobre cómo enriquecer eventos con la jerarquía de vistas en nuestra <PlatformLink to="/enriching-events/viewhierarchy/">documentación de Jerarquía de vistas</PlatformLink>.
</SdkOption>

<SdkOption name="enabled">
  Especifica si este SDK debe enviar eventos a Sentry. Por defecto es `true`. Establecer `enabled: false` no evita toda la sobrecarga de la instrumentación de Sentry. Para deshabilitar Sentry por completo, según el entorno, llama a `Sentry.init` de forma condicional.
</SdkOption>

<SdkOption name="enableCaptureFailedRequests">
  Al activarla, esta función captura automáticamente errores del cliente HTTP, como códigos de respuesta erróneos, como eventos de error y los reporta a Sentry.

  *(Nuevo en la versión 5.3.0)*
</SdkOption>

<div id="integration-configuration">
  ## Configuración de integraciones
</div>

En muchos SDK de plataforma, las integraciones pueden configurarse junto con el propio SDK. En algunas plataformas esto ocurre como parte de la llamada a `init()`, en otras se aplican diferentes patrones.

<SdkOption name="integrations" />

En algunos SDK, las integraciones se configuran mediante este parámetro al inicializar la biblioteca. Para obtener más información, consulta la documentación de la integración específica.

<PlatformSection notSupported={["java", "dart"]}>
  <SdkOption name="defaultIntegrations" type="bool" defaultValue="true">
    Esto puede usarse para desactivar las integraciones que se añaden de forma predeterminada. Cuando se establece en `false`, no se añade ninguna integración predeterminada.
  </SdkOption>
</PlatformSection>

<PlatformSection supported={["python"]}>
  <SdkOption name="autoEnablingIntegrations" type="bool" defaultValue="true">
    Esto puede usarse para desactivar las integraciones que se habilitan automáticamente si el SDK detecta que el framework o la biblioteca correspondiente está instalado. Cuando se establece en `false`, ninguna de estas integraciones se habilitará por defecto, incluso si se detecta el framework o la biblioteca correspondiente.
  </SdkOption>
</PlatformSection>

<div id="hooks">
  ## Hooks
</div>

Estas opciones se pueden usar para integrar el SDK de varias formas y personalizar el envío de eventos.

<SdkOption name="beforeSend">
  Esta función se invoca con un mensaje específico del SDK o un objeto de evento de error y puede devolver un objeto de evento modificado, o `null` para omitir el envío del evento. Esto puede usarse, por ejemplo, para eliminar manualmente información personal (PII) antes de enviar.

  Cuando se ejecuta <PlatformIdentifier name="beforeSend" />, todos los datos del scope ya se han aplicado al evento. Cualquier modificación posterior del scope no tendrá efecto.
</SdkOption>

<SdkOption name="beforeSendTransaction">
  Esta función se invoca con un objeto de evento de transacción específico del SDK y puede devolver un objeto de evento de transacción modificado, o `null` para omitir el envío del evento. Una forma de uso sería eliminar manualmente información personal (PII) antes de enviar.
</SdkOption>

<SdkOption name="beforeBreadcrumb">
  Esta función se invoca con un objeto de breadcrumb específico del SDK antes de que el breadcrumb se agregue al scope. Si la función no devuelve nada, el breadcrumb se descarta. Para dejar pasar el breadcrumb, devuelve el primer argumento, que contiene el objeto de breadcrumb.
  La devolución de llamada normalmente recibe un segundo argumento (llamado &quot;hint&quot;) que contiene el objeto original a partir del cual se creó el breadcrumb para personalizar mejor cómo debería verse.
</SdkOption>

<div id="transport-options">
  ## Opciones de transporte
</div>

Los transportes se usan para enviar eventos a Sentry. Pueden personalizarse hasta cierto punto para adaptarse mejor a implementaciones muy específicas.

<SdkOption name="transport">
  Cambia el transporte usado para enviar eventos. Cómo funcione depende del SDK. Por ejemplo, puede usarse para capturar eventos en pruebas unitarias o para enviarlos a través de una configuración más compleja que requiera autenticación de proxy.
</SdkOption>

<SdkOption name="transportOptions">
  Opciones para configurar el transporte. Es un objeto con las siguientes claves opcionales:

  * `headers`: Objeto con los encabezados que se enviarán en cada solicitud. Usado por los transportes `fetch` y XHR del SDK.
  * `fetchOptions`: Objeto con las opciones que se pasarán a la llamada `fetch`. Usado por el transporte `fetch` del SDK.
</SdkOption>

<SdkOption name="shutdownTimeout">
  Controla cuántos segundos esperar antes de cerrarse. Los SDK de Sentry envían eventos desde una cola en segundo plano. A esta cola se le concede un tiempo determinado para vaciar los eventos pendientes. El valor predeterminado depende del SDK, pero suele ser de alrededor de dos segundos. Establecer este valor demasiado bajo puede causar problemas al enviar eventos desde aplicaciones de línea de comandos. Establecerlo demasiado alto hará que la aplicación se bloquee durante mucho tiempo para las personas que experimentan problemas de conectividad de red.
</SdkOption>

<div id="tracing-options">
  ## Opciones de trazas
</div>

<SdkOption name="tracesSampleRate">
  Un número entre `0` y `1` que controla el porcentaje de probabilidad de que una transacción determinada se envíe a Sentry. (`0` representa el 0% y `1` representa el 100%.) Se aplica por igual a todas las transacciones creadas en la app. Para habilitar el trazado, debe definirse esta opción o <PlatformIdentifier name="tracesSampler" />.
</SdkOption>

<SdkOption name="tracesSampler">
  Una función responsable de determinar el porcentaje de probabilidad de que una transacción determinada se envíe a Sentry. Recibirá automáticamente información sobre la transacción y el contexto en el que se está creando, y debe devolver un número entre `0` (0% de probabilidad de envío) y `1` (100% de probabilidad de envío). También puede usarse para filtrar transacciones, devolviendo 0 para aquellas no deseadas. Para habilitar el trazado, debe definirse esta opción o <PlatformIdentifier name="tracesSampleRate" />.
</SdkOption>

<SdkOption name="tracePropagationTargets">
  Una propiedad opcional que controla qué servicios downstream reciben datos de trazas, en forma de los encabezados `sentry-trace` y `baggage` adjuntos a cualquier solicitud HTTP saliente.

  La opción puede contener una lista de cadenas o expresiones regulares contra las cuales se comparan las URL de las solicitudes salientes.
  Si una de las entradas de la lista coincide con la URL de una solicitud saliente, se adjuntarán datos de traza a esa solicitud.
  Las entradas de tipo cadena no tienen que ser coincidencias exactas, lo que significa que la URL de una solicitud coincide cuando *contiene* una cadena proporcionada mediante la opción.

  Si no se proporciona <PlatformIdentifier name="tracePropagationTargets" />, se adjuntan datos de traza a todas las solicitudes salientes de la aplicación (móvil, escritorio, TV). Al ejecutarse en la web, los datos de traza solo se adjuntan a las solicitudes salientes que contengan `localhost` en su URL o a solicitudes cuya URL comience con `'/'` (por ejemplo, `GET /api/v1/users`).
</SdkOption>

<SdkOption name="propagateTraceparent" type="bool" defaultValue="false" availableSince="7.4.0">
  Controla si el SDK debe propagar el encabezado HTTP W3C `traceparent` junto con los encabezados `sentry-trace` y `baggage` en las solicitudes HTTP salientes.
</SdkOption>

  ## Funciones experimentales
</div>

<SdkOption name="experimental">
  Propiedad opcional que define qué funciones están en modo experimental. Según el lenguaje, esta propiedad puede ser un `Object Type` con propiedades o un `TypedDict` de pares clave/valor. Las funciones experimentales aún están en desarrollo y pueden tener errores. Sí, vemos la ironía.
</SdkOption>

<div id="hybrid-sdk-options">
  ## Opciones del SDK híbrido
</div>

<SdkOption name="enableNative">
  Establece este booleano en `false` para desactivar el SDK nativo. Esto desactivará todo el manejo nativo de bloqueos y errores y, en su lugar, el SDK solo capturará errores en la capa superior.
</SdkOption>

<SdkOption name="autoInitializeNativeSdk">
  Establece este booleano en `false` para desactivar la inicialización automática del SDK de la capa nativa. Al hacerlo, tendrás que inicializar el SDK nativo manualmente. No uses esto para desactivar la capa nativa.

  Para desactivar la capa nativa, usa [enableNative](#enableNative).

  Debes seguir la [guía de inicialización nativa](/es/platforms/react-native/manual-setup/native-init/) si eliges usar esta opción.
</SdkOption>

<SdkOption name="enableNativeCrashHandling">
  Establece este booleano en `false` para desactivar el manejo de bloqueos graves desde la capa nativa. Al hacerlo, el SDK no capturará eventos por bloqueos graves en Android e iOS si el error fue causado por código nativo.
</SdkOption>

<SdkOption name="enableNativeNagger">
  Establece este booleano en `false` para desactivar la alerta del nagger nativo.
</SdkOption>

<SdkOption name="enableAutoSessionTracking">
  Establece este booleano en `false` para desactivar la función de [salud de la versión](/es/product/releases/health/).
</SdkOption>

<SdkOption name="sessionTrackingIntervalMillis">
  Establécelo para cambiar el intervalo predeterminado para finalizar una sesión (salud de la versión) si la app pasa a segundo plano. El valor predeterminado es 30 000.
</SdkOption>

<SdkOption name="enableNdkScopeSync">
  Establece este booleano en `false` para desactivar la sincronización del scope de Java con NDK en Android.
</SdkOption>

<SdkOption name="attachThreads">
  Establece este booleano en `true` para adjuntar automáticamente todos los hilos a todos los eventos registrados en Android.
</SdkOption>

<SdkOption name="enableAutoPerformanceTracing">
  Establece este booleano en `false` para desactivar el seguimiento automático del [monitoreo de rendimiento](/es/product/insights/).
</SdkOption>

<SdkOption name="enableWatchdogTerminationTracking">
  Establece este booleano en `false` para desactivar el seguimiento de [falta de memoria](/es/platforms/apple/guides/ios/configuration/out-of-memory/) en iOS.
</SdkOption>

<SdkOption name="onReady">
  Establece este callback, que se ejecuta después de que el SDK de Sentry para React Native inicializa sus SDK nativos (Android e iOS).
</SdkOption>
