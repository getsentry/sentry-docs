<div id="core-options">
  ## Opciones principales
</div>

<TableOfContents ignoreIds={["core-options"]} />

Las opciones que se pueden leer desde una variable de entorno (`SENTRY_DSN`, `SENTRY_ENVIRONMENT`, `SENTRY_RELEASE`) se leen automáticamente.

<SdkOption name="dsn" type="string" envVar="SENTRY_DSN">
  El DSN le indica al SDK adónde enviar los eventos. Si no se proporciona este valor, el SDK intentará leerlo de la variable de entorno `SENTRY_DSN`. Si esa variable tampoco existe, el SDK simplemente no enviará ningún evento.

  En entornos de ejecución sin variables de proceso (como el navegador), ese mecanismo de respaldo no aplica.

  Más información sobre el [uso del DSN](/es/product/sentry-basics/dsn-explainer/#dsn-utilization).
</SdkOption>

<SdkOption name="debug" type="bool" defaultValue="false">
  Activa o desactiva el modo de depuración. Si `debug` está habilitado, el SDK intentará imprimir información útil de depuración si algo sale mal al enviar el evento. El valor predeterminado siempre es `false`. En general no se recomienda activarlo en producción, aunque activar el modo `debug` no implica riesgos de seguridad.
</SdkOption>

<SdkOption name="diagnosticLevel" type="enum">
  Habilitar el modo `debug` hace que el SDK genere tantos datos de diagnóstico como sea posible. Sin embargo, si prefieres reducir la verbosidad de los registros de diagnóstico del SDK de Sentry, configura esta opción para establecer el nivel apropiado:

  * `debug`: **predeterminado** El modo más detallado
  * `info`: Mensajes informativos
  * `warning`: Advertencia de que algo podría no estar bien
  * `error`: Solo se imprimen errores internos del SDK
  * `fatal`: Solo se imprimen errores críticos
</SdkOption>

<SdkOption name="releaseName" type="string">
  Configura la versión (release). Algunos SDK intentarán configurar automáticamente una release desde el inicio, pero es mejor configurarla manualmente para garantizar que la release esté sincronizada con tus integraciones de despliegue o cargas de mapas de origen. Los nombres de release son cadenas, pero algunos formatos son detectados por Sentry y podrían mostrarse de manera diferente. Obtén más información sobre cómo enviar datos de release para que Sentry pueda informarte sobre regresiones entre versiones e identificar la fuente potencial en [la documentación de releases](/es/product/releases/) o el <SandboxLink scenario="releases">sandbox</SandboxLink>.
</SdkOption>

<SdkOption name="dist" type="string">
  Configura la distribución de la aplicación. Las distribuciones se usan para desambiguar variantes de compilación o despliegue de la misma release de una aplicación. Por ejemplo, la dist puede ser el número de compilación de una build de Xcode o el código de versión de una build de Android. La dist tiene una longitud máxima de 64 caracteres.
</SdkOption>

<SdkOption name="environment" type="string" envVar="SENTRY_ENVIRONMENT">
  Configura el entorno. Esta cadena es de formato libre y no se establece de forma predeterminada. Una release puede asociarse con más de un entorno para separarlos en la interfaz (piensa en `staging` vs `prod` o similar).

  De forma predeterminada, el SDK intentará leer este valor de la variable de entorno `SENTRY_ENVIRONMENT` (excepto para el SDK del navegador, donde esto no es aplicable).
</SdkOption>

<SdkOption name="sampleRate" type="float" defaultValue="1.0">
  Configura la tasa de muestreo para eventos de error, en el rango de `0.0` a `1.0`. El valor predeterminado es `1.0`, lo que significa que se enviará el 100% de los eventos de error. Si se establece en `0.1`, solo se enviará el 10% de los eventos de error. Los eventos se seleccionan aleatoriamente.
</SdkOption>

<SdkOption name="maxBreadcrumbs" type="int" defaultValue="100">
  Esta variable controla la cantidad total de breadcrumbs que se deben capturar. El valor predeterminado es `100`, pero puedes configurarlo en cualquier número. Sin embargo, debes tener en cuenta que Sentry tiene un [tamaño máximo de carga útil](https://develop.sentry.dev/sdk/data-model/envelopes/#size-limits) y cualquier evento que exceda ese tamaño se descartará.
</SdkOption>

<SdkOption name="maxCacheItems" type="int" defaultValue="30">
  El número máximo de [envelopes](https://develop.sentry.dev/sdk/data-model/envelopes/) que se mantendrán en caché. Los SDK usan envelopes para enviar datos, como eventos, archivos adjuntos, comentarios de usuario y sesiones a sentry.io. Un envelope puede contener varios elementos, como un evento con una sesión y dos archivos adjuntos. Dependiendo del uso del SDK, el tamaño de un envelope puede variar. Si el número de envelopes en la caché local excede `max-cache-items`, el SDK elimina el envelope más antiguo y migra las sesiones al siguiente envelope para mantener la integridad de tus estadísticas de salud de la release. El valor predeterminado es `30`.
</SdkOption>

<SdkOption name="attachStacktrace" type="bool" defaultValue="true">
  Cuando está habilitada, las trazas de pila se adjuntan automáticamente a todos los mensajes registrados. Las trazas de pila siempre se adjuntan a las excepciones; sin embargo, al activar esta opción, también se envían junto con los mensajes. Esto, por ejemplo, hace que las trazas de pila aparezcan junto a todos los mensajes de registro.

  Esta opción viene activada de forma predeterminada.

  El agrupamiento en Sentry difiere entre eventos con y sin trazas de pila. Como resultado, obtendrás grupos nuevos al habilitar o deshabilitar esta opción para ciertos eventos.
</SdkOption>

<SdkOption name="enableSigtermReporting" type="bool" availableSince="8.27.0">
  *(Nuevo en [sentry-cocoa versión 8.27.0](https://github.com/getsentry/sentry-cocoa/blob/main/CHANGELOG.md#8270))*

  Cuando está habilitada, el SDK informa señales SIGTERM a Sentry.

  Es crucial que los desarrolladores entiendan que el SO envía un SIGTERM a su app como preludio de un apagado ordenado, antes de recurrir a un SIGKILL. Este SIGKILL, que tu app no puede capturar ni ignorar, es una orden directa para terminar inmediatamente el proceso de tu app. Los desarrolladores deben tener en cuenta que su app puede recibir un SIGTERM en varios escenarios, como uso excesivo de CPU o disco, terminaciones por watchdog o cuando el SO actualiza su app.
</SdkOption>

<SdkOption name="sendDefaultPii" type="bool" defaultValue="false">
  Si esta opción está habilitada, ciertas informaciones de identificación personal <PlatformLink to="/data-management/data-collected/">(PII)</PlatformLink> son añadidas por integraciones activas. De forma predeterminada, no se envían esos datos.

  <Alert>
    Si usas Sentry en tu app móvil, lee nuestras [preguntas frecuentes sobre privacidad de datos móviles](/es/security-legal-pii/security/mobile-privacy/) para ayudarte con los detalles de privacidad de apps en Apple App Store y Google Play.
  </Alert>

  Esta opción está desactivada de forma predeterminada.

  Si habilitas esta opción, asegúrate de eliminar manualmente lo que no quieras enviar usando nuestras funciones para gestionar [*datos sensibles*](../../data-management/sensitive-data/).
</SdkOption>

<SdkOption name="autoSessionTracking" type="bool">
  Cuando se establece en `true`, el SDK enviará eventos de sesión a Sentry. Esto es compatible en todos los SDK de navegador, enviando una sesión por carga y navegación de página a Sentry. En los SDK móviles, cuando la app pasa a segundo plano por más de 30 segundos, las sesiones finalizan.
</SdkOption>

<SdkOption name="inAppInclude" type="array">
  Una lista de prefijos de cadena de nombres de módulos que pertenecen a la app. Esta opción tiene prioridad sobre `in-app-exclude`.

  Sentry diferencia los frames de pila que están directamente relacionados con tu aplicación (“en la aplicación”) de los que provienen de otros paquetes como la biblioteca estándar, frameworks u otras dependencias. El paquete de la aplicación se marca automáticamente como `inApp`. La diferencia es visible en [sentry.io](https://sentry.io), donde solo los frames “en la aplicación” se muestran de forma predeterminada.
</SdkOption>

<SdkOption name="inAppExclude" type="array">
  Una lista de prefijos de cadena de nombres de módulos que no pertenecen a la app, sino a paquetes de terceros. Los módulos considerados fuera de la app se ocultarán de las trazas de pila de forma predeterminada.

  Esta opción se puede anular usando <PlatformIdentifier name="inAppInclude" />.
</SdkOption>

<SdkOption name="initialScope" type="function" availableSince="8.7.0">
  Un bloque que configura el alcance inicial al iniciar el SDK.

  ```swift
  import Sentry

  SentrySDK.start { options in
      options.dsn = "___PUBLIC_DSN___"
      options.initialScope = { scope in
          scope.setTag(value: "my value", key: "my-tag")
          return scope
      }
  }
  ```

  ```objc {tabTitle:Objective-C}
  @import Sentry;

  [SentrySDK startWithConfigureOptions:^(SentryOptions *options) {
      options.dsn = @"___PUBLIC_DSN___";
      options.initialScope = ^(SentryScope *scope) {
          [scope setTagValue:@"my value" forKey:@"my-tag"];
          return scope;
      };
  }];
  ```

  *(Nuevo en [sentry-cocoa versión 8.7.0](https://github.com/getsentry/sentry-cocoa/blob/main/CHANGELOG.md#870))*
</SdkOption>

<PlatformSection supported={["apple.ios"]}>
  <SdkOption name="attachScreenshot" type="bool">
    Toma una captura de pantalla de la aplicación cuando ocurre un error y la incluye como archivo adjunto.
    Obtén más información sobre cómo enriquecer eventos con capturas de pantalla en nuestra <PlatformLink to="/enriching-events/screenshots/">documentación sobre capturas de pantalla</PlatformLink>.
  </SdkOption>

  <SdkOption name="attachViewHierarchy" type="bool">
    Genera una representación JSON de toda la jerarquía de vistas de la aplicación cuando ocurre un error y la incluye como archivo adjunto.
    Obtén más información sobre cómo enriquecer eventos con la jerarquía de vistas en nuestra <PlatformLink to="/enriching-events/viewhierarchy/">documentación sobre la jerarquía de vistas</PlatformLink>.
  </SdkOption>

  <SdkOption name="idleTimeout" type="int" defaultValue="3000">
    El tiempo de inactividad, medido en ms, que se espera hasta que una <PlatformLink to="/tracing/instrumentation/automatic-instrumentation/#user-interaction-tracing">transacción de interacción del usuario</PlatformLink> se finalice automáticamente. La transacción usará la marca de tiempo del último span finalizado como hora de finalización de la transacción.

    El valor predeterminado es `3000`.
  </SdkOption>
</PlatformSection>

<SdkOption name="sendClientReports" type="bool">
  Establece este booleano en `false` para desactivar el envío de informes del cliente. Los informes del cliente son una característica del protocolo que permite a los clientes enviar informes de estado sobre sí mismos a Sentry. Actualmente se usan principalmente para emitir resultados de eventos que nunca se enviaron.
</SdkOption>

<SdkOption name="enableCaptureFailedRequests" type="bool">
  Una vez habilitada, esta función captura automáticamente errores del cliente HTTP, como códigos de respuesta no satisfactorios, como eventos de error y los informa a Sentry.
</SdkOption>

<div id="integration-configuration">
  ## Configuración de integraciones
</div>

En muchos SDK de la plataforma, las integraciones pueden configurarse junto con ellos. En algunas plataformas eso ocurre como parte de la llamada a `init()`, en otras se aplican distintos patrones.

<SdkOption name="integrations" type="array" />

En algunos SDK, las integraciones se configuran mediante este parámetro durante la inicialización de la librería. Para obtener más información, consulta nuestra documentación de la integración específica.

<PlatformSection notSupported={["java", "dart"]}>
  <SdkOption name="defaultIntegrations" type="bool" />

  Puede usarse para desactivar las integraciones que se agregan de forma predeterminada. Si se establece en `false`, no se agregará ninguna integración predeterminada.
</PlatformSection>

<PlatformSection supported={["python"]}>
  <SdkOption name="autoEnablingIntegrations" type="bool" />

  Puede usarse para desactivar las integraciones que se habilitan por defecto cuando el SDK detecta que el framework o la librería correspondiente está instalado. Si se establece en `false`, ninguna de estas integraciones se habilitará por defecto, incluso si se detecta el framework/librería correspondiente.
</PlatformSection>

<div id="hooks">
  ## Hooks
</div>

Estas opciones se pueden usar para integrar el SDK de varias maneras y personalizar el envío de eventos.

<SdkOption name="beforeSend" type="function">
  Esta función se invoca con un mensaje específico del SDK o un objeto de evento de error, y puede devolver un objeto de evento modificado o `null` para omitir el envío del evento. Esto puede usarse, por ejemplo, para eliminar manualmente datos personales (PII) antes de enviar.

  Para cuando se ejecuta <PlatformIdentifier name="beforeSend" />, todos los datos del alcance (scope) ya se han aplicado al evento. Cualquier modificación posterior del alcance no tendrá efecto.
</SdkOption>

<SdkOption name="beforeBreadcrumb" type="function">
  Esta función se invoca con un objeto de breadcrumb específico del SDK antes de que el breadcrumb se agregue al alcance (scope). Si la función no devuelve nada, el breadcrumb se descarta. Para permitir que el breadcrumb pase, devuelve el primer argumento, que contiene el objeto del breadcrumb.
  La función de retorno de llamada típicamente recibe un segundo argumento (llamado “hint”) que contiene el objeto original a partir del cual se creó el breadcrumb, para personalizar aún más cómo debería verse el breadcrumb.
</SdkOption>

<div id="tracing-options">
  ## Opciones de traza
</div>

<SdkOption name="tracesSampleRate" type="float">
  Un número entre `0` y `1` que controla el porcentaje de probabilidad de que una transacción se envíe a Sentry. (`0` representa el 0% y `1` representa el 100%.) Se aplica por igual a todas las transacciones creadas en la app. Para habilitar la trazabilidad, se debe definir esta opción o <PlatformIdentifier name="tracesSampler" />.
</SdkOption>

<SdkOption name="tracesSampler" type="function">
  Una función responsable de determinar el porcentaje de probabilidad de que una transacción se envíe a Sentry. Recibirá automáticamente información sobre la transacción y el contexto en el que se está creando, y debe devolver un número entre `0` (0% de probabilidad de envío) y `1` (100% de probabilidad de envío). También puede usarse para filtrar transacciones, devolviendo 0 para aquellas no deseadas. Para habilitar la trazabilidad, se debe definir esta opción o <PlatformIdentifier name="tracesSampleRate" />.
</SdkOption>

<SdkOption name="tracePropagationTargets" type="array">
  Una propiedad opcional que controla qué servicios de destino reciben datos de traza, en forma de un encabezado `sentry-trace` y uno `baggage` adjuntos a cualquier solicitud HTTP saliente.

  La opción puede contener una lista de cadenas o expresiones regulares (regex) con las que se comparan las URL de las solicitudes salientes.
  Si una de las entradas de la lista coincide con la URL de una solicitud saliente, los datos de traza se adjuntarán a esa solicitud.
  Las entradas de tipo cadena no tienen que coincidir por completo; es decir, la URL de una solicitud coincide cuando *contiene* una cadena proporcionada mediante la opción.

  Si no se proporciona <PlatformIdentifier name="tracePropagationTargets" />, se adjuntan datos de traza a cada solicitud saliente del cliente instrumentado.
</SdkOption>

<SdkOption name="enablePersistingTracesWhenCrashing" type="bool" defaultValue="false" availableSince="8.41.0">
  <Alert>
    Esta función es experimental y puede tener errores. Está disponible desde la [versión 8.41.0](https://github.com/getsentry/sentry-cocoa/blob/main/CHANGELOG.md#8410).
  </Alert>

  Cuando está habilitada, el SDK finaliza la transacción en curso vinculada al scope y la asocia con el evento de bloqueo cuando tu app se bloquea. El SDK omite agregar perfiles para aumentar la probabilidad de conservar la transacción. Esta opción tiene como valor predeterminado `false`.
</SdkOption>
