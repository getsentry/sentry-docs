Cuando agregas Sentry a tu app, obtienes mucha información valiosa sobre errores y rendimiento. Y tener mucha información es bueno, siempre que sea la adecuada y en un volumen razonable.

Los SDK de Sentry ofrecen varias opciones de configuración para ayudarte a filtrar eventos.

También ofrecemos [filtros de entrada](/es/concepts/data-management/filtering/) para filtrar eventos en sentry.io. Aun así, recomendamos filtrar en el cliente, porque evita la sobrecarga de enviar eventos que en realidad no quieres. Obtén más información sobre los [campos disponibles en un evento](https://develop.sentry.dev/sdk/data-model/event-payloads/).

<div id="filtering-error-events">
  ## Filtrado de eventos de error
</div>

Para evitar que ciertos errores se envíen a Sentry, usa las opciones de configuración <PlatformIdentifier name="before-send" /> o <PlatformIdentifier name="add-exception-filter" />, que te permiten evaluar si enviar o no un error. Como alternativa, también puedes controlar el comportamiento habilitando o deshabilitando integraciones.

### Uso de <PlatformIdentifier name="before-send" />

Todos los SDK de Sentry admiten el método de callback <PlatformIdentifier name="before-send" />. Como se invoca inmediatamente antes de que el evento se envíe al servidor, esta es tu última oportunidad para decidir no enviar datos o editarlos. <PlatformIdentifier name="before-send" /> recibe el objeto de evento como parámetro, que puedes usar para modificar los datos del evento o descartarlo por completo devolviendo `null`, según la lógica personalizada y los datos disponibles en el evento.

<PlatformContent includePath="configuration/before-send/" />

Ten en cuenta también que se pueden filtrar las migas de pan (breadcrumbs), como se explica en [nuestra documentación de Breadcrumbs](/es/product/error-monitoring/breadcrumbs/).

### Uso de <PlatformIdentifier name="add-exception-filter" /> y <PlatformIdentifier name="add-exception-filter-for-type" />

El SDK también te permite definir tus propios filtros de excepciones personalizados. Estos deben heredar de <PlatformIdentifier name="IExceptionFilter" />

```csharp
public class MyExceptionFilter : IExceptionFilter
{
    public bool Filter(Exception ex)
    {
        // TODO: Añade tu lógica de filtrado
    }
}
```

y luego se puede pasar a las opciones durante la inicialización.

```csharp
options.AddExceptionFilter(new MyExceptionFilter());
```

Los tipos de excepciones proporcionados mediante <PlatformIdentifier name="add-exception-filter-for-type" /> se filtran automáticamente y no se envían a Sentry.

```csharp
options.AddExceptionFilterForType<MyCustomException>();
```

<div id="filtering-transaction-events">
  ## Filtrado de eventos de transacción
</div>

Para evitar que se reporten ciertas transacciones a Sentry, usa la opción de configuración <PlatformIdentifier name="traces-sampler" /> o <PlatformIdentifier name="before-send-transaction" />, que te permite proporcionar una función para evaluar la transacción actual y descartarla si no es una que quieras conservar.

### Uso de <PlatformIdentifier name="traces-sampler" />

**Nota:** Las opciones de configuración <PlatformIdentifier name="traces-sampler" /> y <PlatformIdentifier name="traces-sample-rate" /> se excluyen mutuamente. Si defines un <PlatformIdentifier name="traces-sampler" /> para filtrar ciertas transacciones, también debes manejar las transacciones no filtradas devolviendo la tasa a la que quieres muestrearlas.

En su forma más simple, usada solo para filtrar la transacción, se ve así:

<PlatformContent includePath="performance/traces-sampler-as-filter" />

También permite muestrear distintas transacciones a diferentes tasas.

Si la transacción que se está procesando tiene una transacción principal (de un servicio ascendente que llama a este servicio), la decisión de muestreo del principal (ascendente) siempre se incluirá en los datos del contexto de muestreo, para que tu <PlatformIdentifier name="traces-sampler" /> pueda elegir si y cuándo heredar esa decisión. En la mayoría de los casos, heredar es la opción correcta para evitar romper trazas distribuidas. Una traza rota no incluirá todos tus servicios. Consulta <PlatformLink to="/configuration/sampling/#inheritance">Heredar la decisión de muestreo del principal</PlatformLink> para obtener más información.
