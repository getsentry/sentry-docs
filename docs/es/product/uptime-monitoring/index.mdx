El Monitoreo de disponibilidad de Sentry te permite supervisar la disponibilidad y la confiabilidad de tus servicios web sin esfuerzo. Una vez habilitado, realiza un seguimiento continuo de las URL configuradas y envía alertas e información instantáneas para identificar rápidamente interrupciones y resolver problemas.

Al aprovechar el [trazado distribuido](/es/concepts/key-terms/tracing/distributed-tracing/), Sentry te permite identificar cualquier error que ocurra durante una verificación de disponibilidad, lo que simplifica la triage y acelera el análisis de la causa raíz. Esto te ayuda a mejorar tanto la confiabilidad como la disponibilidad de tus servicios web.

<div id="set-up">
  ## Configuración
</div>

Uptime se [configura automáticamente](/es/product/uptime-monitoring/automatic-detection/) como una alerta nueva para el nombre de host que aparece con mayor frecuencia en todas las URL de tus datos de errores, lo que garantiza la supervisión continua de tu nombre de host más crítico desde el primer momento.

También puedes [crear alertas de supervisión de Uptime](/es/product/alerts/create-alerts/) para URL específicas. Son totalmente personalizables con detalles de la solicitud, como el método HTTP, los encabezados y el cuerpo.

<div id="uptime-check-criteria">
  ## Criterios de verificación de disponibilidad
</div>

Nuestro sistema de monitoreo de disponibilidad comprueba la accesibilidad de tus URL realizando solicitudes HTTP a intervalos regulares y preconfigurados. Para que una URL se considere en funcionamiento, la respuesta debe cumplir los siguientes criterios:

1. **Respuesta correcta (códigos de estado 2xx):**
   La URL debe devolver un código de estado HTTP en el rango 200–299, lo que indica que la solicitud fue correcta.
2. **Gestión automática de redirecciones (códigos de estado 3xx):** Sentry seguirá las redirecciones de las URL que devuelvan un código de estado HTTP en el rango 300–399 y verificará que la URL de destino final devuelva una respuesta correcta. Esto garantiza que las redirecciones no activen falsamente alertas de inactividad.
3. **Configuración del tiempo de espera (timeout):** Cada solicitud tiene un umbral de 10 segundos.
   Si el servidor no responde dentro de ese período, la verificación se marcará como fallida,
   lo que indica una posible inactividad o problemas de rendimiento.
4. **Detección de problemas de DNS:** Nuestro monitoreo también incluye la detección de problemas de resolución de DNS.
   Si se detecta un problema de DNS, la verificación se marcará como fallida,
   lo que te permitirá abordar los problemas de conectividad subyacentes.

<div id="uptime-check-failures">
  ### Fallos en las comprobaciones de disponibilidad
</div>

Una alerta de disponibilidad supervisa continuamente la URL configurada según los criterios definidos arriba. Si se produce un fallo,
se crea un nuevo [incidente de disponibilidad](/es/product/issues/issue-details/uptime-issues/) que incluye detalles sobre la comprobación fallida y los errores relacionados.

Para evitar alertas falsas causadas por problemas temporales de red, **solo se genera un incidente después de tres fallos consecutivos** tras la detección inicial de la caída. Además, las comprobaciones de disponibilidad se ejecutan desde varias ubicaciones geográficas en modo round-robin. Esto garantiza que cada comprobación fallida provenga de una región distinta, reduciendo la probabilidad de falsos positivos por fallos de red localizados.

*En casos poco frecuentes en los que Sentry no pueda realizar una comprobación de disponibilidad programada —como durante interrupciones—, el estado de la comprobación se marcará como &quot;Desconocido&quot;.*

<div id="uptime-request-spans">
  ## Spans de solicitud de Uptime
</div>

Las comprobaciones de Uptime incluyen spans llamados *spans de solicitud de Uptime* que Sentry crea automáticamente para la solicitud. El span `uptime.request` actúa como span raíz para las trazas relacionadas con las comprobaciones de Uptime.

![Span de solicitud de Uptime](./img/uptime-request-span.png)

**Beneficios clave:**

* Contexto: Sabes al instante que una traza fue generada por una comprobación de Uptime en lugar de por tráfico de usuarios
* Ciclo de vida de la solicitud: Puedes ver el recorrido completo desde la comprobación de Uptime inicial hasta la respuesta de tu aplicación
* Depuración mejorada: Puedes ver más detalles sobre exactamente dónde y por qué se produjo el fallo para distinguir entre problemas relacionados con Uptime y otros problemas de la aplicación

<Alert type="info">
  Los spans de solicitud de Uptime son gratuitos y no contarán para tu [cuota de spans](/es/pricing/quotas/manage-transaction-quota/). Estos spans siempre se crean, incluso si tienes el muestreo deshabilitado.
</Alert>

<div id="notifications">
  ## Notificaciones
</div>

Para empezar a recibir notificaciones sobre un nuevo incidente de caída, [configura una alerta de incidencia](/es/product/alerts/create-alerts/issue-alert-config/) y elige la categoría de incidencia “outage”. Luego elige cómo quieres que te avisemos (por correo electrónico, Slack, etc.).
