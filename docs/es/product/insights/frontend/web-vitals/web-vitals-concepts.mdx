[Web Vitals](https://web.dev/vitals/) es un conjunto de métricas definidas por Google para medir el tiempo de renderizado, el tiempo de respuesta y el desplazamiento de diseño. Cada dato ofrece información sobre el [rendimiento](/es/product/insights/overview/) general de tu aplicación.

Los SDK de Sentry para navegadores recopilan información de Web Vitals (cuando está disponible) y la agregan a las [transacciones](/es/product/insights/overview/transaction-summary/) del frontend. Luego, estas Web Vitals se resumen en la [página **Performance &gt; Web Vitals**](/es/product/insights/web-vitals/) para darte una vista rápida de cómo rinde cada página para tus usuarios.

![Visualización de Web Vitals](../../img/diagram-transaction-vitals.png)

<div id="core-web-vitals">
  ## Core Web Vitals
</div>

Google considera que las Core Web Vitals son las métricas más importantes para medir la experiencia de usuario en páginas web. Según una [entrada del blog de Google](https://developers.google.com/search/blog/2020/11/timing-for-page-experience) de mayo de 2021, estas métricas también influyen en tu posicionamiento en las búsquedas.

<div id="largest-contentful-paint-lcp">
  ### Largest Contentful Paint (LCP)
</div>

[Largest Contentful Paint (LCP)](https://web.dev/lcp/) mide cuánto tarda en mostrarse el contenido que ocupa el área de píxeles más grande del viewport; en otras palabras, cuánto tiempo pasa antes de que el usuario vea el contenido principal de una página. Este contenido puede ser cualquier elemento del Document Object Model (DOM), como una imagen, un SVG o un bloque de texto.

<div id="interaction-to-next-paint-inp">
  ### Interacción hasta la siguiente pintura (INP)
</div>

<Alert>
  El 12 de marzo de 2024, Interaction to Next Paint (INP) sustituyó a First Input Delay (FID) como Core Web Vital. Antes de esto, INP era una métrica experimental que Sentry no recopilaba. Para comenzar a recopilar mediciones de INP, asegúrate de que tu versión del SDK de JavaScript sea la [7.104.0](https://github.com/getsentry/sentry-javascript/releases/tag/7.104.0) o superior y de que la opción [`enableInp`](/es/platforms/javascript/tracing/instrumentation/automatic-instrumentation/#enableinp) esté activada (a partir de la versión `8.0.0`, `enableInp` está activada de forma predeterminada).
</Alert>

[Interaction to Next Paint (INP)](https://web.dev/articles/inp) mide el tiempo desde que un usuario interactúa con una página (mediante un clic, un toque o una pulsación de teclado) hasta que se produce la siguiente pintura (renderizado de contenido en pantalla). INP evalúa qué tan rápido los usuarios ven una respuesta del sitio web después de realizar una acción, algo crucial para ofrecer una experiencia de usuario fluida y ágil.

<div id="cumulative-layout-shift-cls">
  ### Cumulative Layout Shift (CLS)
</div>

[Cumulative Layout Shift (CLS)](https://web.dev/cls/) es la suma de las puntuaciones individuales de desplazamiento de diseño por cada cambio inesperado de elementos que ocurre durante el proceso de renderizado. Un ejemplo sería intentar hacer clic en un enlace en una página que aún no ha terminado de cargar y que ese enlace se desplace hacia abajo antes de que alcances a hacer clic debido a problemas al renderizar las imágenes. La métrica web vital de CLS no se basa en la duración: representa la magnitud de los desplazamientos que resultan disruptivos y visualmente inestables.

![Ejemplo de Cumulative Layout Shift](../../img/diagram-transaction-cls.png)

Cada puntuación de desplazamiento de diseño se calcula usando una fracción de impacto y una fracción de distancia. La fracción de impacto es el área visible total que el elemento afecta entre los dos fotogramas renderizados. La fracción de distancia mide cuánto se ha movido en relación con la ventana gráfica (viewport).

```
Puntuación de cambio de diseño = Fracción de impacto * Fracción de distancia
```

Echemos un vistazo al ejemplo anterior, que tiene un elemento inestable: el cuerpo de texto. La fracción de impacto es aproximadamente el 50% de la página y desplaza el cuerpo de texto hacia abajo un 20%. La puntuación de cambio de diseño es 0,1, producto de 0,5 × 0,2. Por lo tanto, el CLS es 0,1.

<div id="other-web-vitals">
  ## Otros Web Vitals
</div>

Estos Web Vitals suelen ser menos visibles para el usuario, pero son útiles para diagnosticar problemas con los Core Web Vitals.

<div id="first-paint-fp">
  ### First Paint (FP)
</div>

First Paint (FP) mide el tiempo que tarda en aparecer el primer píxel en el viewport, mostrando cualquier cambio visual respecto a lo que se veía antes. Esto puede provenir de cualquier elemento del Document Object Model (DOM), como el color de fondo, un lienzo (canvas) o una imagen. FP ayuda a los desarrolladores a entender si sucede algo inesperado al renderizar la página.

<div id="first-contentful-paint-fcp">
  ### First Contentful Paint (FCP)
</div>

[First Contentful Paint (FCP)](https://web.dev/fcp/) mide el tiempo que tarda en mostrarse el primer contenido en el área visible (viewport). Este contenido puede provenir de cualquier elemento del modelo de objetos del documento (DOM), como imágenes, SVG o bloques de texto. FCP a menudo se superpone con First Paint (FP). FCP ayuda a los desarrolladores a entender cuánto tarda el usuario en ver cualquier cambio de contenido en la página.

<div id="time-to-first-byte-ttfb">
  ### Time To First Byte (TTFB)
</div>

[Time To First Byte (TTFB)](https://web.dev/articles/ttfb) mide el tiempo que tarda el navegador de una persona en recibir el primer byte del contenido de una página. TTFB ayuda a los desarrolladores a determinar si la lentitud se debe a la respuesta inicial o, por el contrario, a contenido que bloquea el renderizado.

<div id="first-input-delay-fid">
  ### First Input Delay (FID)
</div>

<Alert>
  Desde el 12 de marzo de 2024, First Input Delay (FID) fue reemplazado como Core Web Vital por INP. Aunque Sentry ya no incluye FID en nuestros cálculos de la puntuación de rendimiento ni en la página de rendimiento **Web Vitals**, seguimos recopilando métricas de FID. Puedes consultar las mediciones de FID en [Discover](/es/product/explore/discover-queries/).
</Alert>

[First Input Delay (FID)](https://web.dev/fid/) mide el tiempo de respuesta cuando una persona intenta interactuar con el área visible (viewport) haciendo clic en un botón, en un enlace o en cualquier otro controlador personalizado de JavaScript. Los datos de FID son fundamentales para comprender si las interacciones en una página de la aplicación tienen éxito o no.

<div id="thresholds">
  ## Umbrales
</div>

Google utiliza los umbrales &quot;Good&quot;, &quot;Needs Improvement&quot; y &quot;Poor&quot; para clasificar los puntos de datos en verde, amarillo y rojo para los Web Vitals correspondientes. En Sentry, &quot;Needs Improvement&quot; se denomina &quot;Meh&quot;.

| Web Vital                                                       | Good     | Meh      | Poor    |
| --------------------------------------------------------------- | -------- | -------- | ------- |
| [Largest Contentful Paint](#largest-contentful-paint-lcp) (LCP) | &lt;= 2.5s  | &lt;= 4s    | &gt; 4s    |
| [Interaction to Next Paint](#interaction-to-next-paint-inp) (INP)| &lt;= 200ms | &lt;= 500ms | &gt; 500ms |
| [Cumulative Layout Shift](#cumulative-layout-shift-cls) (CLS)   | &lt;= 0.1   | &lt;= 0.25  | &gt; 0.25  |
| [First Paint](#first-paint-fp) (FP)                             | &lt;= 1s    | &lt;= 3s    | &gt; 3s    |
| [First Contentful Paint](#first-contentful-paint-fcp) (FCP)     | &lt;= 1s    | &lt;= 3s    | &gt; 3s    |
| [Time To First Byte](#time-to-first-byte-ttfb) (TTFB)           | &lt;= 100ms | &lt;= 200ms | &gt; 600ms |
| [First Input Delay](#first-input-delay-fid) (FID)               | &lt;= 100ms | &lt;= 300ms | &gt; 300ms |

<Alert>
  Algunos Web Vitals, como FP, FCP, LCP y TTFB, se miden con respecto al inicio de la transacción. Los valores pueden diferir frente a los generados con otras herramientas como [Lighthouse](https://github.com/GoogleChrome/lighthouse).
</Alert>
