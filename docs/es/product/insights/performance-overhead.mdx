Habilitar Insights y Tracing en los SDK de Sentry añade cierta sobrecarga, pero debería tener un impacto mínimo en el rendimiento de tu aplicación.

Funcionan instrumentando partes importantes de aplicaciones típicas, como:

* Llamadas a bases de datos
* Solicitudes HTTP entrantes y salientes
* Tareas de trabajo en segundo plano
* Carga y navegación de páginas en el navegador
* Ciclo de vida de componentes en frameworks de UI
* Web Vitals

Para realizar esta instrumentación, los SDK de Sentry envuelven o aplican parches (monkeypatch) a funciones específicas dentro de bibliotecas o frameworks populares y miden el tiempo de ejecución de estas operaciones. Luego, estos datos se empaquetan y serializan en una carga útil que contiene una [Transaction](/es/product/insights/overview/transaction-summary/#what-is-a-transaction) (que contiene Spans) que se transmite a los servidores de ingesta de eventos a través de Internet.

Los SDK también se aseguran de propagar ciertos encabezados en las solicitudes salientes para que el [Distributed Tracing](/es/product/sentry-basics/tracing/) funcione de manera efectiva.

<Alert>
  Para la mayoría de las aplicaciones, la sobrecarga de rendimiento de habilitar Insights y Tracing en producción es imperceptible para los usuarios finales.
</Alert>

<div id="will-performance-break-my-application">
  ## ¿Hará que Performance falle mi aplicación?
</div>

Cada SDK tiene límites en las estructuras de datos que se usan para registrar toda la información anterior. Si tu aplicación hace demasiadas cosas, simplemente alcanzaremos esos límites y así evitaremos escenarios en los que el propio SDK consuma demasiado tiempo de CPU o haga que el uso de memoria crezca sin control. También registraremos Outcomes (estadísticas) cuando se den estos escenarios para ser transparentes sobre qué eventos estamos descartando y por qué.

<div id="backpressure-management">
  ### Gestión de backpressure
</div>

Algunos de nuestros SDK de backend gestionan el backpressure internamente reduciendo la tasa de muestreo dinámica cuando el sistema está en un estado no saludable. Esto garantiza una mayor estabilidad del sistema y una menor sobrecarga del SDK bajo cargas altas.

Las últimas versiones de los siguientes SDK son compatibles con la gestión de backpressure:

* Python
* Ruby
* Java

Iremos implementando esta función gradualmente en otros lenguajes de backend en el futuro.

<div id="hosting-environment">
  ### Entorno de hosting
</div>

Algunos entornos de hosting tienen mecanismos que afectan negativamente la sobrecarga del SDK:

* Las Lambdas de AWS congelan por completo cualquier ejecución de código cuando se envía una respuesta. Esto significa que los SDK, por lo general, bloquearán la finalización de la respuesta hasta que hayan enviado a Sentry todos los datos de telemetría registrados. Esto puede ocasionar una sobrecarga no trivial en algunas situaciones. En este caso, podría ser beneficioso ejecutar una instancia de Sentry Relay físicamente cerca de tus lambdas para reducir la latencia.
* Los SDK diseñados para ejecutarse en Vercel Edge Runtime detendrán las respuestas a las solicitudes hasta que el SDK haya enviado a Sentry toda la telemetría de esa solicitud. Esta es una limitación impuesta por Edge Runtime, ya que no permite que las operaciones de E/S se compartan o persistan entre invocaciones del controlador de solicitudes, lo que significa que el SDK no puede enviar sus datos de forma asíncrona.
