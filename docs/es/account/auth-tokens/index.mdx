Los tokens de autenticación (abreviados como *auth tokens*) son una forma de autenticarse en Sentry. Son similares a las contraseñas, pero están diseñados para la interacción programática con Sentry. Algunos ejemplos de uso de los tokens de autenticación incluyen:

* Cargar source maps durante tu build de CI
* Usar [Sentry CLI](/es/cli/) para interactuar con Sentry
* Usar la [API de Sentry](/es/api/auth/)

Cada token de autenticación se crea con un conjunto específico de permisos y ámbitos que se corresponden con los [Permissions &amp; Scopes](/es/api/permissions/) de la API de Sentry. Algunos tipos de tokens de autenticación tienen permisos fijos que no se pueden editar, mientras que otros se pueden personalizar al crear el token y editar posteriormente.

Recomendamos usar un token de autenticación distinto para cada caso de uso. Por ejemplo, usarías un token diferente para cargar source maps que el que usas con Sentry CLI. Esto tiene la ventaja de que, si se compromete un token de autenticación, puedes revocarlo sin afectar el resto de tu flujo de trabajo.

<div id="types-of-auth-tokens">
  ## Tipos de tokens de autenticación
</div>

Hay tres tipos principales de tokens de autenticación en Sentry:

* [Organization Tokens](#organization-tokens):
  Estos tokens están vinculados a una organización y tienen acceso a todos los proyectos dentro de esa organización. Tienen un conjunto limitado de permisos y están diseñados para usarse en entornos de CI y con Sentry CLI.

* [Internal Integrations](#internal-integrations):
  Estos tokens están vinculados a una organización y tienen acceso a todos los proyectos dentro de esa organización. Se pueden crear con un conjunto personalizado de permisos y están diseñados para usarse cuando los organization tokens no tienen permisos suficientes.

* [Personal Tokens](#personal-tokens):
  Estos tokens están vinculados a un usuario y tienen acceso a todas las organizaciones y proyectos a los que ese usuario tenga acceso.

<div id="when-should-i-use-which">
  ### ¿Cuándo debería usar cuál?
</div>

Para la mayoría de los casos, recomendamos usar [Organization Tokens](#organization-tokens). Están diseñados para usarse en entornos de CI y tienen un conjunto limitado de permisos. Esto significa que, si se compromete el lugar donde almacenaste el token de autenticación, el atacante solo podrá causar un daño limitado.

Los permisos de los organization tokens no son personalizables. Están configurados para permitir la mayoría de las tareas relacionadas con CI, sin permisos innecesarios.

Las [Internal Integrations](#internal-integrations) deben usarse cuando necesitas acceso completo a la API (que los organization tokens no pueden otorgar) y quieres interactuar con la API de Sentry en nombre de una organización. Por ejemplo, para crear un proyecto nuevo de forma programática, usarías una internal integration.

Los permisos de los tokens de autenticación creados como parte de una internal integration son personalizables y editables.

Los [Personal Tokens](#personal-tokens) deben usarse para interactuar con la API de Sentry en nombre de un usuario. Por ejemplo, para obtener todos los issues de un usuario, usarías un personal token. No recomendamos usar personal tokens para tareas de CI porque, si se elimina al usuario que creó el token de la Organization, el token dejará de funcionar.

Los permisos de los personal tokens son personalizables, pero no pueden editarse después.

<div id="organization-tokens">
  ### Tokens de la organización
</div>

[Los tokens de la organización](https://sentry.io/orgredirect/organizations/:orgslug/settings/auth-tokens/) se pueden crear en [sentry.io](https://sentry.io) en la página **Auth Tokens**, dentro de **Settings &gt; Developer Settings &gt; Organization Tokens**.

![](./img/org-auth-tokens-overview.png)

<Alert>
  También se pueden generar en ciertas páginas de la documentación de Sentry si has iniciado sesión, y usando Sentry Wizard para configurar la subida de source maps.
</Alert>

Los nombres de los tokens de la organización se generan automáticamente, a menos que crees el token a través de la interfaz de Sentry. Este nombre solo se utiliza con fines de visualización: ayuda a identificar un token de autenticación por si quieres revocarlo más tarde. Puedes cambiar el nombre de un token de la organización en [sentry.io](https://sentry.io), en la página **Edit Auth Token** dentro de **Settings &gt; Developer Settings &gt; Organization Tokens**.

Por motivos de seguridad, los tokens de la organización solo son visibles una vez, justo después de crearlos. Si pierdes el token de autenticación, tendrás que crear uno nuevo. Esto significa que no puedes ver el token completo en la página de resumen ni en la página de detalles; solo puedes ver los últimos caracteres del token para ayudarte a identificarlo.

Cualquier usuario puede crear tokens de la organización para cualquiera de sus organizaciones. Esto permite que cualquier usuario (no solo los propietarios de la organización) configure un SDK de Sentry y ponga en marcha procesos de CI creando y usando tokens de la organización. Dado que los tokens de la organización tienen acceso limitado, el potencial de abuso también es limitado.

![](./img/org-auth-token-create.png)

Todos los propietarios de la organización recibirán un correo de seguridad cuando se cree un nuevo token de la organización y podrán revocar estos tokens en cualquier momento. *Solo* los propietarios y administradores de la organización pueden revocar tokens de la organización.

<div id="internal-integrations">
  ### Integraciones internas
</div>

Recomendamos usar una integración interna solo cuando quieras interactuar con la API de Sentry en nombre de una organización. Consulta la documentación de [autenticación](/es/api/auth/) de la API de Sentry para obtener más información sobre cómo hacerlo.

Para empezar, consulta nuestra documentación sobre [integraciones internas](/es/organization/integrations/integration-platform/).

<div id="personal-tokens">
  ### Tokens personales
</div>

[Tokens personales](https://sentry.io/settings/account/api/auth-tokens/) se pueden crear en [sentry.io](https://sentry.io) en la página **Tokens personales**, dentro del menú Cuenta en la esquina superior izquierda.

![](./img/user-auth-tokens-menu.png)

Cualquier usuario puede crear tokens personales, y estos quedan vinculados a dicho usuario. Los tokens pueden recibir permisos para todas las organizaciones y proyectos a los que ese usuario tenga acceso. Esto significa que el alcance *máximo* de un token personal abarca todos los permisos a los que el usuario tenga acceso dentro de una organización determinada. Un token personal no puede exceder los permisos del usuario que lo creó. Consulta [Administración de organizaciones y usuarios](/es/organization/membership/) para más detalles sobre cómo funcionan los permisos en Sentry.

Cuando creas un nuevo token personal, puedes seleccionar en la interfaz qué [alcances](/es/api/permissions/) debe tener el token:

![](./img/user-auth-token-create.png)

Actualmente, puedes ver los tokens personales en la interfaz después de crearlos. Este es un comportamiento heredado que podría cambiar en el futuro. Recomendamos usar cada token personal solo una vez y crear un token nuevo para cada caso de uso. Cualquier usuario puede revocar en cualquier momento cualquiera de sus tokens personales.

![](./img/user-auth-tokens-overview.png)

Recomendamos usar un token personal únicamente para interactuar con la API de Sentry en nombre de un usuario. Consulta la documentación de [Autenticación](/es/api/auth/) de la API de Sentry para más información sobre cómo hacerlo.
