title: Dynamic Sampling by Example
description: >-
  A concrete walkthrough of how Dynamic Sampling rules combine to produce final
  sample rates for a single event.

---

Dynamic Sampling's rule based system can be hard to understand at first. This example will help you understand how the rules are combined to produce the final sample rate for each event. Rules have two components: a "sampling strategy" and a "rule type". The sampling strategy determines what the rule does, and the rule type determines what the rule is applied to. So, the rule type is the "when to match", and the sampling strategy is the "what to do with it".

The available sampling strategies are:
1. SampleRate: sets a specific sample rate (e.g. 50%)
2. Factor: multiplies the sample rate by a factor (e.g. 2.0 or 0.4)
3. Reservoir: samples a fixed number of events (e.g. 100 events)
4. MinimumSampleRate: sets a minimum sample rate (e.g. 0.1) to ensure that the sample rate does not drop below a certain threshold

The available rule types are:
1. Trace: a trace rule matches against the Dynamic Sampling Context (DSC) of the trace, and applies to all spans in a trace
2. Transaction: a transaction rule matches against the transaction of the event, independent of the trace it belongs to
3. Project: a project rule matches against the project of the event, and applies to all events in the project

These rules are stacked on top of each other to calculate the final sample rate for each event. To make this a bit more concrete, let's look at an example.

Let's say we have an organization with two projects of different volumes, and the larger project has transactions with very different volumes. Let's take an example of a webshop:

- Project A: 1,000,000 transactions per day
    - Transaction `/product/list`: 900,000 events per day
    - Transaction `/user/login`: 99,900 events per day
    - Transaction `/checkout/complete`: 100 events per day
- Project B: 1,000 transactions per day
    - Transaction `/support/contact-form/submit`: 1,000 events per day

Obviously, we would like to see all of the data from Project B, but we can quite heavily sample Project A. However, we also want to make sure that the low-volume but important `/checkout/complete` transactions are not drowned out by the high-volume endpoints in the same project. For simplicity, let's say our organization has a target sample rate of 10%.

Dynamic Sampling would generate the following rules:
- Project A:
    - Project: 9.99% sample rate
    - Trace rule: `trace.transaction:/product/list` -> 0.999
    - Trace rule: `trace.transaction:/user/login` -> 0.999
    - Trace rule: `trace.transaction:/checkout/complete` -> 10
- Project B:
    - Project: 100% sample rate

We will see later how these rules are generated, but for now let's focus on the final sample rate for each event:
- Event from `/product/list`: 9.99% * 0.999 ≈ 9.99%
- Event from `/user/login`: 9.99% * 0.999 ≈ 9.99%
- Event from `/checkout/complete`: 10% * 10% = 1%
- Event from `/support/contact-form/submit`: 100%

So, we would see 9.99% of the data from Project A, and 100% of the data from Project B. This is not a perfect solution, but it is a good compromise between seeing all the data from Project B and not drowning out the rare, business-critical events in Project A.

This simple example shows how Dynamic Sampling rules are combined to produce the final sample rate for each event. In reality, Dynamic Sampling is a lot more complex, and there are a lot of different rules that are applied to each event, but this example should give you a good idea of how the rules are stacked on top of each other to produce the final sample rate for each event in Dynamic Sampling.
