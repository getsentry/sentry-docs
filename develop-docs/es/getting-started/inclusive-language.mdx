---
title: Lenguaje inclusivo
description: Sentry es un producto usado y desarrollado por muchas personas de distintos orígenes culturales, y procuramos evitar el lenguaje identificado como dañino o insensible.
sidebar_order: 40
---

Esto se aplica no solo a la documentación, sino también al producto en sí y a las API internas.

Qué constituye lenguaje inclusivo es un tema en constante evolución y,
como Sentry es un producto con historia, es probable que siempre quede
algo en la base de código de lo que sea difícil deshacerse.

Por lo tanto, este documento define en gran medida qué debe hacerse en el código nuevo, pero
no necesariamente en el código existente.

<div id="phased-out-terms">
  ## Términos en desuso
</div>

Hemos identificado algunos términos que ya no queremos usar en adelante, pero que aún podrían existir en partes del código.

<div id="masterslave">
  ### master/slave
</div>

Por lo general, en el contexto de bases de datos deben usarse los términos "primary"/"secondary" o "primary"/"replica".

<div id="whitelistblacklist">
  ### whitelist/blacklist
</div>

En general, no deberían usarse. Procuramos evitar la palabra "lista"
  y ser más descriptivos en la interfaz. Por ejemplo, el uso de
  "ip blacklist" se cambió a "ip filters" como parte del producto de filtros.
  Si necesitas un término genérico, "allowlist" y "denylist" o "blocklist" son
  buenas alternativas.

<div id="whitelistedblacklisted">
  ### whitelisted/blacklisted
</div>

Procura evitar verbos genéricos aquí y describe con mayor precisión lo que
  intentas hacer. Por ejemplo, cambiamos "whitelisted environment
  variables" por "safe environment variables".

<div id="grandfathered">
  ### grandfathered
</div>

Intenta evitarlo. Si puede reemplazarse por "legacy", usa ese término; de lo contrario, si quieres enfatizar que la funcionalidad solo está disponible porque la utilizabas en el pasado, sé explícito. Cambia "available on grandfathered plans" por "available on old plans no longer available for sign-up".

<div id="acceptable-terms">
  ## Términos aceptables
</div>

<div id="master">
  ### master
</div>

Master de forma aislada es aceptable si el origen del término permite excluir el
  significado asociado de "slave". Por ejemplo, este es el caso cuando se
  hace referencia a una "master copy" o a un "master's degree", etc. Para el nombre de la rama de Git, consulta más abajo.

<div id="git-branch-naming">
  ## Nomenclatura de ramas de Git
</div>

Nos parece bien mantener el término «master» para los repositorios existentes, pero por lo general se recomienda cambiar el nombre de la rama predeterminada a «main» en adelante en los repositorios nuevos. Aunque el término «master» de forma aislada suele ser aceptable, el origen del nombre de la rama en Git proviene de BitKeeper, donde esta terminología se refería a repositorios maestro y esclavo. En la actualidad, cambiar la rama predeterminada en repositorios existentes tiene consecuencias negativas para los usuarios que ya han clonado los repositorios, por lo que no deberían cambiarse.

A partir de Git 2.28, puedes reconfigurar Git para usar un nombre de rama predeterminado diferente para `git init`:

```bash
git config --global init.defaultBranch main
```

A partir de ahora, los futuros repositorios usarán el nombre de rama predeterminado &quot;main&quot;.


<div id="changing-code">
  ## Cambiar el código
</div>

Si identificas código o comentarios que usan lenguaje problemático, puedes
cambiarlos. Sin embargo, ten en cuenta que algunos usos de ese lenguaje
no se han eliminado porque implican cambios incompatibles con versiones anteriores y
requieren mucho más trabajo. En concreto, por ejemplo, el código de limitación de
tasa en Sentry usa el término "whitelist" para datos persistentes y, por coherencia, el
código que trata con esta estructura de datos, por ahora, mantiene esos usos.
