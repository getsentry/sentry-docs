<div id="introduction">
  ## Introducción
</div>

Esta guía explica el funcionamiento de la nueva estructura y ofrece instrucciones paso a paso para añadir y actualizar los documentos de Introducción.

<div id="structure">
  ## Estructura
</div>

La documentación está basada en React y sigue el siguiente formato:

1. **Instalar**: Instrucciones para instalar el lenguaje o framework. Suele incluir un fragmento de código simple con el comando de instalación.
2. **Configurar**: Explica cómo configurar el lenguaje o framework después de instalarlo. Esta sección incluye un ejemplo de código que ya incorpora el DSN del proyecto creado.
3. **Verificar**: Ofrece instrucciones y un fragmento de código que muestran cómo simular un error en la aplicación para comprobar que la instalación y la configuración se realizaron correctamente.
4. **Próximos pasos:** Otras configuraciones, generalmente relacionadas con funcionalidades, que podrían interesar a los usuarios.

<div id="adding-documentation-for-a-platform">
  ## Agregar documentación para una plataforma
</div>

<div id="language">
  ### Idioma
</div>

1. Crea una nueva carpeta con el nombre del idioma dentro de [gettingStartedDocs](https://github.com/getsentry/sentry/tree/master/static/app/gettingStartedDocs).
2. Dentro de la carpeta, crea un archivo `index.tsx`.
3. Copia el contenido de otro archivo, como [react.tsx](https://github.com/getsentry/sentry/blob/master/static/app/gettingStartedDocs/javascript/react.tsx), y adáptalo o actualízalo para que cumpla los requisitos de la plataforma actual. Asegúrate de darle a la función del archivo nuevo un nombre apropiado.
4. Actualiza la lista [MIGRATED_GETTING_STARTD_DOCS](https://github.com/getsentry/sentry/blob/c2d4d527dcf640b616c73921a1294834720c76bd/src/sentry/models/project.py#L54) añadiendo el slug de la plataforma que se está incorporando. Esto evitará que el backend arroje el error “Platform invalid”. Podemos replantear esta validación más adelante.
5. Asegúrate de que el repositorio [platformicons](https://github.com/getsentry/platformicons) contenga el ícono del idioma que estás agregando.
6. Actualiza la lista [platformIntegrations](https://github.com/getsentry/sentry/blob/0f1dab84775d0d4c7eebbd167dce5835e7a4ee8c/static/app/data/platforms.tsx#L64) con una nueva entrada para tu idioma. Esta entrada se encarga de mostrar el ícono del idioma en la lista “Select the platform you want to monitor/Choose your platform”.
7. Para garantizar el funcionamiento correcto, crea un archivo `index.spec.tsx` con pruebas unitarias para la nueva plataforma. Puedes tomar como referencia una de las pruebas existentes, por ejemplo, [react.spec.tsx](https://github.com/getsentry/sentry/blob/master/static/app/gettingStartedDocs/javascript/react.spec.tsx), copiar su contenido y modificarlo según corresponda para adaptarlo a la plataforma recién migrada.
8. Por último, antes de concluir, ejecuta Sentry de manera local (tiene que ser local debido al cambio en el backend descrito en el paso 4) y realiza una prueba manual para asegurarte de que la documentación de inicio se renderiza correctamente tanto en el proceso de incorporación de una nueva organización como en el flujo de creación de un proyecto. Este paso es esencial para verificar el correcto funcionamiento y la apariencia visual de la documentación.

<div id="framework">
  ### Framework
</div>

1. Ubica la carpeta de idioma del framework dentro de [gettingStartedDocs](https://github.com/getsentry/sentry/tree/master/static/app/gettingStartedDocs).
2. Dentro de esa carpeta, crea un archivo cuyo nombre sea el slug del framework y usa la extensión `.tsx`.
3. Copia el contenido de otro archivo, como [react.tsx](https://github.com/getsentry/sentry/blob/master/static/app/gettingStartedDocs/javascript/react.tsx), y adáptalo o actualízalo para que cumpla con los requisitos de la plataforma actual. Asegúrate de darle a la función del nuevo archivo un nombre adecuado.
4. Actualiza la lista [MIGRATED_GETTING_STARTD_DOCS](https://github.com/getsentry/sentry/blob/c2d4d527dcf640b616c73921a1294834720c76bd/src/sentry/models/project.py#L54) agregando el slug de la plataforma que estás incorporando. Esto evitará que el backend arroje el error “Platform invalid”. Podemos replantear esta validación más adelante.
5. Asegúrate de que el repositorio [platformicons](https://github.com/getsentry/platformicons) contenga el ícono del framework que estás agregando.
6. Actualiza la lista [platformIntegrations](https://github.com/getsentry/sentry/blob/0f1dab84775d0d4c7eebbd167dce5835e7a4ee8c/static/app/data/platforms.tsx#L64) con una nueva entrada para tu framework. Esta entrada se encarga de mostrar el ícono del framework en la lista “Select your platform”.
7. Para garantizar el funcionamiento correcto, crea un archivo de prueba cuyo nombre sea el slug del framework y usa la extensión `.spec.tsx`. Este archivo debe contener pruebas unitarias para la documentación recién creada. Puedes consultar una de las pruebas existentes, por ejemplo, [react.spec.tsx](https://github.com/getsentry/sentry/blob/master/static/app/gettingStartedDocs/javascript/react.spec.tsx), copiar su contenido y modificarlo según corresponda para ajustarlo a la plataforma recién migrada.
8. Por último, antes de concluir, ejecuta Sentry localmente (debe ser local debido al cambio de backend descrito en el paso 4) y realiza una prueba manual para asegurarte de que la documentación de inicio se renderiza correctamente tanto en el proceso de incorporación de una nueva organización como en el flujo de creación de proyectos. Este paso es esencial para verificar el correcto funcionamiento y la apariencia visual de la documentación.

<div id="editing-documentation-of-a-platform">
  ## Edición de la documentación de una plataforma
</div>

<div id="language">
  ### Idioma
</div>

1. Ubica la carpeta del idioma dentro de [gettingStartedDocs](https://github.com/getsentry/sentry/tree/master/static/app/gettingStartedDocs).
2. Actualiza el archivo `index.tsx` con los cambios necesarios.
3. Es posible que debas actualizar las pruebas ubicadas en el archivo `index.spec.tsx`.
4. Por último, antes de concluir, ejecuta Sentry localmente o usa el comando `pnpm dev-ui` con el [Frontend Development Server](https://develop.sentry.dev/frontend/development-server/) y realiza una prueba manual para asegurarte de que la documentación de inicio se renderice correctamente tanto en el proceso de incorporación (onboarding) de una organización nueva como en el flujo de creación de proyectos. Este paso es esencial para verificar el correcto funcionamiento y la apariencia visual de la documentación.

<div id="framework">
  ### Framework
</div>

1. Ubica la carpeta de idioma del framework dentro de [gettingStartedDocs](https://github.com/getsentry/sentry/tree/master/static/app/gettingStartedDocs).
2. Dentro de esa carpeta, busca el archivo cuyo nombre coincide con el slug del framework y actualízalo con los cambios necesarios.
3. Es posible que debas actualizar las pruebas del archivo cuyo nombre coincide con el slug del framework y que tiene la extensión `.spec.tsx`.
4. Por último, antes de finalizar, ejecuta Sentry localmente o usa el comando `pnpm dev-ui` y realiza una prueba manual para verificar que la documentación de primeros pasos se renderiza correctamente tanto en el proceso de incorporación de una nueva organización como en el flujo de creación de proyectos. Este paso es esencial para confirmar el correcto funcionamiento y la apariencia visual de la documentación.

<div id="notes">
  ## Notas
</div>

- Al documentar un nuevo lenguaje o framework, usa la estructura «Install», «Configure» y «Verify». Incluye información adicional solo si es esencial para que las personas nuevas puedan empezar. La documentación de incorporación debe ser directa, mientras que las configuraciones avanzadas pertenecen a la documentación completa en el [sitio de documentación](https://docs.sentry.io/). Mantén la sencillez para una experiencia fluida.
- Puede que encuentres documentación que no se ajuste a la estructura descrita arriba, pero ten la seguridad de que estamos trabajando activamente en consolidarla.
- Para acomodar documentación diversa durante la migración, hemos implementado una estructura de pasos dinámica. A veces, podrías preguntarte si usar «additionalInformation» o «description» para explicar ciertos aspectos. Si la información adicional cubre todo el bloque y «description» ya está en uso, usa «additionalInformation»; de lo contrario, usa «description». Estamos trabajando activamente para simplificar esta estructura y eliminar la confusión en un futuro cercano.
- Actualmente, no toda la documentación incluye «Next Steps», y no pasa nada.
- Los documentos de «Getting Started» se muestran durante la incorporación de una organización nueva y el flujo de creación de proyectos
