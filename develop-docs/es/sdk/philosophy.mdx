<div id="trust-is-1">
  ## La confianza es lo primero
</div>

La misión de Sentry es “permitir que los desarrolladores entreguen con confianza”, y su experiencia siempre comienza con nuestros SDK. Por encima de todo, debemos garantizar que nuestros clientes puedan confiar en la estabilidad de nuestros SDK, porque dependen de ellos. Estamos comprometidos a tener los SDK más estables del mercado y asumiremos nuestros errores con transparencia y claridad.

<div id="dependencies-cost">
  ## Coste de las dependencias
</div>

Las dependencias tienen un coste, y es alto. Cada dependencia que usamos amplía la superficie del SDK y añade más aspectos a considerar en cuanto a licencias, mantenimiento y seguridad. Entendemos que las dependencias son necesarias para habilitar integraciones, pero nunca deben ser obligatorias para la funcionalidad básica de un SDK. Obviamente, toda regla tiene excepciones y en algunas plataformas no podremos funcionar sin un nivel mínimo de dependencias. Un buen ejemplo es Python, donde necesitamos una biblioteca externa para realizar solicitudes HTTP de forma segura.

Consulta también [Micropackages and Open Source Trust Scaling](http://lucumr.pocoo.org/2016/3/24/open-source-trust-scaling/)

<div id="degrade-gracefully">
  ## Degradación elegante
</div>

Nunca queremos que nuestros SDK sean la causa de que algo se rompa en el cliente. Si nuestro SDK hace que el cliente falle, hemos fallado, incluso si el cliente lo ejecuta en una plataforma obsoleta. En particular, si nuestro SDK se ejecuta en un navegador desactualizado, debemos ser lo suficientemente resilientes como para degradar elegantemente y, en última instancia, no hacer nada de forma silenciosa.

Si eso no es posible, debemos asegurarnos de informar al usuario con antelación, alertando sobre estas situaciones durante la instalación. Por ejemplo, si un SDK requiere una versión mínima incluso para compilar, debemos garantizar que el usuario esté informado y evitar errores confusos más adelante.

<div id="compatibility-is-king">
  ## La compatibilidad es fundamental
</div>

Aunque una plataforma esté perdiendo relevancia, queremos seguir dándole soporte. No es nuestra responsabilidad enseñar a los clientes que usan plataformas obsoletas; probablemente ya lo sepan. No debemos fijar el mínimo para las plataformas de los clientes. Si un cliente necesita soportar un navegador antiguo, como mínimo no solo debemos evitar romper ese navegador, también deberíamos plantearnos soportarlo. Las plataformas antiguas son especialmente relevantes para clientes empresariales, que con más frecuencia mantienen pilas tecnológicas antiguas que necesitan soporte.

<div id="customer-value-matters">
  ## El valor para el cliente importa
</div>

Cada cambio tiene un coste y hay que evitar cambiar por cambiar. Si el cambio beneficia al cliente, adelante; si solo se justifica por sí mismo, no hace más que aumentar el riesgo de romper código existente o frustrar a los usuarios. No hay nada peor que un cliente note que sus informes han empeorado por un refactor sin sentido.

<div id="configuration-comes-at-a-cost">
  ## La configuración tiene un costo
</div>

Optimizamos para ofrecer una excelente experiencia desde el primer momento. La personalización debe ser justamente eso: personalización. Los valores predeterminados importan y deben ser sensatos. Las integraciones deberían activarse automáticamente cuando sea posible, ya que cada integración que no se habilita empeora la experiencia del cliente.

<div id="prioritize-customer-convenience-over-correctness">
  ## Prioriza la comodidad del cliente por encima de la corrección
</div>

Está bien hacer algo ligeramente incorrecto si eso da como resultado una mejor experiencia para el cliente. Está bien que nuestro SDK incluya código desactualizado si con ello aumenta la compatibilidad con más plataformas. Tiene sentido que nuestros SDK hagan “cosas que no deberían hacerse” si eso mejora la experiencia del cliente. Por ejemplo, preferimos el monkey patching a la configuración manual, incluso si esos parches pueden ser frágiles en su implementación.

<div id="assume-novice-developers">
  ## Supón desarrolladores principiantes
</div>

La documentación, las guías y el material de marketing deben partir de que el público es un desarrollador principiante no familiarizado con el lenguaje. No uses características del lenguaje potencialmente desconocidas solo para conseguir ejemplos más cortos.

<div id="write-down-the-rules">
  ## Deja las reglas por escrito
</div>

Si se toman decisiones que contravienen o aclaran este documento, deben registrarse por escrito. ¿Un SDK está marcando el límite de qué plataformas están admitidas? Debe existir un documento que indique qué se admite. Cuando se abandone la compatibilidad con una plataforma antigua, hay que destacarlo.

En ningún caso un cliente debería actualizar la biblioteca y descubrir que su stack ya no es compatible enterándose solo al escribir al soporte.

<div id="enable-customers">
  ## Habilitar a los clientes
</div>

Si bien en general deberíamos intentar mantener razonablemente pequeñas las superficies de API de los SDK, también debemos asegurarnos de permitir que los clientes alcancen sus objetivos. Piensa en casos que el SDK quizá no pueda resolver de forma predeterminada. Si contamos con suficientes API para que los clientes usen nuestros SDK de maneras más creativas, por lo general lo consideramos una ventaja adicional.

<div id="handle-pii-and-sensitive-data-with-care">
  ## Manipula la PII y los datos sensibles con cuidado
</div>

Algunos tipos de errores no pueden resolverse sin los datos proporcionados al programa.
Pero asegúrate de que las instrumentaciones automáticas no adjunten PII sin el consentimiento explícito del usuario.
El servidor debe identificar las partes del protocolo que incluyen PII para depurarlas de forma predeterminada.
Consulta <Link to="/sdk/expected-features/data-handling">Data Handling</Link> para más detalles.

<div id="dont-forget-the-big-picture">
  ## No pierdas de vista el panorama general
</div>

Tener razón es importante, pero más importante aún es hacer lo correcto. Sí, estamos creando bibliotecas, y con eso viene mucha responsabilidad. Pero, al final, estamos construyendo un producto que debe resolver un problema que los usuarios tienen en su aplicación. Este principio se relaciona con muchas otras cosas mencionadas en esta página, pero, sobre todo, debería recordarnos que solo tendremos éxito si logramos resolver un problema de principio a fin.

<div id="write-less-code">
  ## Escribe menos código
</div>

La función de un SDK es instrumentar la aplicación del usuario, recopilar el contexto de los tipos de eventos que queremos capturar y enviar a Sentry los datos de eventos con contexto útil. Debemos evitar incluir lógica de negocio extensa en el SDK que transforme estos datos, ya que puede volverse compleja, difícil de mantener y realizar cambios permanentes en el formato de transmisión de los eventos. Debemos procurar que los datos recopilados por el SDK sean lo más “en bruto” posible. En el servidor contamos con más flexibilidad y control para manipular los datos de manera más efectiva.
