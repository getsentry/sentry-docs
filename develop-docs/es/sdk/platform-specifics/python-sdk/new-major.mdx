---
title: Nueva versión mayor
description: Qué tener en cuenta al preparar una nueva versión mayor.
sidebar_order: 50
---

Incluso si decidimos lanzar una nueva versión mayor, es importante priorizar la facilidad de uso, la facilidad de migración y la compatibilidad retroactiva.

Actualizar requiere esfuerzo. Cada versión mayor aumenta las probabilidades de que la gente se quede para siempre en una versión anterior.

No rompas las APIs existentes por un beneficio de mantenimiento menor de tu lado.

<div id="making-migrating-easier">
  ## Facilitar la migración
</div>

Crea una [guía de migración](https://docs.sentry.io/platforms/python/migration/). Incluye ejemplos fáciles de entender y de copiar y pegar siempre que sea posible.

Muchas personas podrían intentar actualizar usando IA. Pruébalo tú mismo en un proyecto de ejemplo. Considera agregar reglas específicas para la migración.

<div id="dropping-version-support">
  ## Fin de compatibilidad con versiones
</div>

Al preparar una nueva versión principal, considera si debemos dejar de admitir una versión antigua de Python. Hay datos en Looker que puedes usar para tomar una decisión informada.

Investiga si debemos dejar de admitir versiones antiguas de frameworks y bibliotecas. No es viable hacerlo manualmente porque son demasiadas. Para ello, hay un script en `scripts/update_integration_support.py` que detecta automáticamente la versión mínima efectivamente admitida según nuestras pruebas y los metadatos del paquete. Puedes ejecutar el script después de retirar la compatibilidad con una versión antigua de Python, y debería detectar automáticamente qué versiones mínimas dejan de estar admitidas.

Para especificar por completo las nuevas versiones mínimas:

- Actualiza `_MIN_VERSIONS` en `sentry_sdk/integrations/__init__.py`
- Actualiza los límites inferiores en `extras_require` en `setup.py`, si están definidos
- Actualiza las páginas de documentación de las integraciones afectadas (por lo general, la versión mínima admitida se menciona al final)
