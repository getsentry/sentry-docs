---
title: Lista de verificación para APIs públicas
sidebar_order: 5
---

A continuación se presenta una lista de verificación que las personas desarrolladoras deben revisar antes de hacer públicas las APIs.

1. Las APIs deben devolver JSON. Las excepciones a esta regla son APIs en las que la persona usuaria solicita específicamente datos que no sean JSON. Por ejemplo, solicitar un archivo CSV o un archivo de depuración.
1. La API debe estar disponible si se solicita con un bearer token desde la plataforma de integración.
1. Existen permisos (scopes) que se pueden seleccionar en la plataforma de integración y que habilitan o deshabilitan el acceso al recurso expuesto por la API.
1. Si la llamada a la API que se realiza consume muchos recursos en la infraestructura de Sentry, debe existir un límite de tasa que restrinja las llamadas excesivas.
1. Las APIs que devuelvan un número variable de elementos como lista deben estar paginadas. Deben paginarse utilizando el estándar del encabezado Link.
1. Las solicitudes de usuarios finales no deberían poder provocar fácilmente errores 5xx en nuestra aplicación manipulando los datos de la solicitud o enviando datos no válidos o malformados.
1. Debe haber pruebas asociadas a la API que verifiquen la corrección. También debe haber pruebas que revisen el formato de la respuesta para asegurar que la API no se desvíe de su contrato.
1. La API debe devolver el mismo formato para un código de estado, independientemente de la entrada proporcionada. El contenido puede ser diferente, pero el formato de la respuesta debe ser el mismo.
1. Cuando una API solo esté disponible si ciertas feature flags están habilitadas (quizá mediante un plan de precios), la API debe comprobarlo. Si se usa un bearer token de una organización que no tiene acceso a la función, la API debe devolver un 403 Forbidden y un error que explique qué feature flag se requiere.
1. Las APIs deben usar camelCase para los parámetros de consulta y los cuerpos de solicitud/respuesta.
