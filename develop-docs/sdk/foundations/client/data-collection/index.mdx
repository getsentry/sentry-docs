## Overview

This spec defines how SDKs control **what data is collected automatically** from the runtime (device, requests, responses, user context). It replaces the single `sendDefaultPii` (or platform-equivalent) flag with a structured `dataCollection` configuration so users can enable or restrict collection by category and by field.

Previously, `sendDefaultPii` acted as a broad binary toggle that controlled a wide range of data types without offering granular control.
The new `dataCollection` configuration allows SDKs to include rich context for debugging by default, while keeping PII gated behind explicit options to protect user privacy.

SDKs provide **preset factories** (e.g., `DataCollection.default()`, `DataCollection.withUserData()`) that return a fully resolved configuration object. Users can use a preset as-is or tweak individual fields before passing it to `init`.

Related specs:

- [Data Handling](/sdk/expected-features/data-handling/) — structuring data for scrubbing (spans, breadcrumbs), variable size limits
- [Client](/sdk/foundations/client/) — client lifecycle and event pipeline
- [Configuration](/sdk/foundations/client/configuration/) — top-level init options including `send_default_pii` (deprecated in favor of this spec)

## Concepts

<SpecSection id="data-sensitivity-levels" status="draft" since="0.1.0">

### Data Sensitivity Levels

Collected data is grouped into three sensitivity levels. SDKs **MUST** treat these levels consistently when applying defaults and user configuration.

#### 1. Technical Context Data

Non-identifying context used for debugging and performance. Examples include, but are not limited to:

- Device and environment context (OS, runtime, non-PII identifiers)
- Performance and error context (stack frames, breadcrumbs, span metadata)
- Framework/routing context where it does not contain PII or secrets
- AI agent messages (input, output, metadata)

For the full list of supported context types (device, OS, runtime, app, browser, GPU, culture, cloud resource, memory info, and more), see the [Contexts Interface](/sdk/foundations/transport/event-payloads/contexts/).

This sensitivity level is **not** gated by the `dataCollection` configuration. SDKs **SHOULD** collect it by default.

#### 2. PII Data

Personally Identifiable Information (PII) or user-linked data. Examples include, but are not limited to:

- Identity (name, email, user ID, username)
- Contact (email, phone number, address)
- IP address
- Cookies and headers that identify the user or session
- HTTP request data (TBD)

For context types that may carry PII fields (e.g. `device_unique_identifier` in [Device Context](/sdk/foundations/transport/event-payloads/contexts/#device-context), or fields in the [User Interface](/sdk/foundations/transport/event-payloads/user/)), see the linked specs.

This sensitivity level **MUST** be disabled by default unless the user opts in via `userInfo: true` or explicit allowlists. See [`userInfo`](#user-info-behavior), [Option Types](#option-types), and [Default Denylist](#default-denylist).

#### 3. Sensitive Data

Credentials and secrets that **MUST NOT** be sent by default. Examples include, but are not limited to:

- Passwords, tokens, API keys, bearer tokens
- Header or cookie values matching known sensitive names (auth, token, secret, password, key, jwt, etc.)

For the canonical list of terms used to detect sensitive values at collection time, see the [Default Denylist](#default-denylist) in this spec.

SDKs **MUST** never send sensitive **values** through automatic instrumentation. The SDK replaces sensitive values with `"[Filtered]"` and retains key names (see [Default Denylist](#default-denylist)). Users can use `beforeSend` (or equivalent) to remove or redact keys if needed.

</SpecSection>

## Behavior

<SpecSection id="configuration-surface" status="draft" since="0.1.0">

### Configuration Requirements

All data-collection options reside under a single top-level key: `dataCollection`. Each option within the object has a fixed default that does not depend on any other option.

SDKs **MUST** support at least the `userInfo` option and the `CollectBehavior` / Boolean options listed in the [Public API](#public-api). SDKs **MAY** omit options that do not apply to their platform (e.g., `outgoingResponseBody` on a client-only SDK).

SDKs **MUST** also provide preset factory methods (or language-idiomatic equivalents such as static builders, profiles, or named constructors) that return a fully resolved `DataCollection` object. See [Preset Factories](#preset-factories).

</SpecSection>

<SpecSection id="user-info" status="draft" since="0.1.0">

### `userInfo` Behavior

The `userInfo` option controls whether the SDK automatically attaches user identity fields to events (e.g., `user.id`, `user.email`, `user.username`, `user.ip_address`).

| Value | Behavior |
|-------|----------|
| `true` | Attach user identity fields captured by automatic instrumentation from any source. Equivalent to the legacy `sendDefaultPii` flag scoped to user data. |
| `false` | Do not attach user identity fields from automatic instrumentation. |

This distinction matters because collecting a header value and setting a user identity field are two separate concerns. A developer may explicitly allow `x-forwarded-for` in `httpHeaders`. The SDK **MUST** capture and attach that raw header value regardless of `userInfo`. The `userInfo` option only governs whether the SDK automatically infers and sets `user.*` fields from instrumentation sources (e.g., a web framework passing the client IP, or the SDK reading request context and calling the equivalent of `Sentry.setUser({ ip_address: "..." })`).

For example: `httpHeaders: { allow: ["x-forwarded-for"] }` exposes the real value of that header in events (instead of `"[Filtered]"`), while `userInfo: false` prevents the SDK from ever automatically populating `user.ip_address`. Conversely, with `userInfo: true`, the SDK **SHOULD** automatically populate `user.ip_address` from any instrumentation source where an IP address can be reliably determined.

When user data is set **explicitly** on the scope via `Sentry.setUser()` (or equivalent), it is **always** attached regardless of this setting. See [User-Set Data and Scrubbing](#user-set-data-and-scrubbing).

</SpecSection>

<SpecSection id="option-types" status="draft" since="0.1.0">

### Option Types

Each option in `dataCollection` maps to a single category of automatically collected data. The option type depends on whether the data has a key-value structure.

**Boolean Options**: Used where data cannot be meaningfully filtered at the key level. The SDK either collects the entire category or skips it.

| Value | Behavior |
|-------|----------|
| `true` | Collect and attach this data category. |
| `false` | Do not collect this data category. |

**`CollectBehavior` options** — used for key-value data (cookies, headers, query params). Key names are **always included** in the event. The allow/deny configuration controls only which **values** are sent in plaintext vs. replaced with `"[Filtered]"`.

| Value | Behavior |
|-------|----------|
| `true` | Collect this category. All key names are included. The SDK replaces values for keys matching the denylist with `"[Filtered]"`; all other values are sent as-is (see [Default Denylist](#default-denylist)). |
| `false` | Do not collect this category at all — no keys or values are attached to the event. |
| `{ deny: string[] }` | Collect values for keys matching the base denylist **plus** the specified extra terms; the SDK replaces values for those keys with `"[Filtered]"`. All other values are sent as-is. |
| `{ allow: string[] }` | **Only** keys in this list send their real value; the SDK replaces values for all other keys with `"[Filtered]"`. Base denylist scrubbing still applies — a value matching a sensitive pattern is always scrubbed even if its key is in the allow list. |

> **Note:** `allow` and `deny` are mutually exclusive forms and cannot be combined.

**Example:** `httpHeaders: { allow: ["x-request-id", "content-type"] }` → all header key names are present in the event, but only `x-request-id` and `content-type` show their real values; every other header value appears as `"[Filtered]"`.

</SpecSection>

<SpecSection id="preset-factories" status="draft" since="0.1.0">

### Preset Factories

Because the `dataCollection` object is flat, the runtime configuration is fully explicit — no option implicitly changes the behavior of another option. To make common configurations easy to reach, SDKs **MUST** provide **preset factory methods** (or language-idiomatic equivalents) that return a complete, ready-to-use `DataCollection` object.

#### Requirements

- SDKs **MUST** provide at least `DataCollection.default()` and `DataCollection.withUserData()`.
- Factory methods **MUST** return a mutable object so users can override individual fields before passing it to `init`.
- SDKs **MAY** offer additional presets (e.g., `DataCollection.minimal()`) as long as they return the same flat structure.
- SDKs **SHOULD** use idiomatic naming for factories (e.g., `DataCollection.default()` in JavaScript/TypeScript, `DataCollection.with_user_data()` in Python, `DataCollectionConfig.withUserData()` in Java).

The two required presets differ in two dimensions: whether user identity fields are attached automatically, and which deny terms are applied to `CollectBehavior` options (cookies, headers, query params).

| | `DataCollection.default()` | `DataCollection.withUserData()` |
|---|---|---|
| `userInfo` | `false` — user identity fields are not collected automatically | `true` — user identity fields are collected automatically |
| `CollectBehavior` deny terms | Base denylist **+** extended PII terms (see [Extended Deny Terms](#extended-deny-terms)) | Base denylist only |
| Request/response bodies | Disabled | Enabled |

#### `DataCollection.default()`

Returns the safe-by-default configuration. User identity is not collected automatically and `CollectBehavior` options apply the extended PII deny terms on top of the base denylist. See [Extended Deny Terms](#extended-deny-terms).

```pseudocode
DataCollection.default() → {
  userInfo: false,
  cookies: { deny: ["forwarded", "-ip", "remote-", "via", "-user"] },
  httpHeaders: { deny: ["forwarded", "-ip", "remote-", "via", "-user"] },
  queryParams: { deny: ["forwarded", "-ip", "remote-", "via", "-user" ] },
  aiAgentMessages: true,
  stackFrameVariables: true,
  incomingRequestBody: false,
  incomingResponseBody: false,
  outgoingRequestBody: false,
  outgoingResponseBody: false,
  frameContextLines: 5,
}
```


#### `DataCollection.withUserData()`

Returns a configuration with user data collection enabled. User identity fields are attached automatically, request/response bodies are enabled, and `CollectBehavior` options apply only the base denylist (no extended PII terms).

```pseudocode
DataCollection.withUserData() → {
  userInfo: true,
  cookies: true,
  httpHeaders: true,
  queryParams: true,
  aiAgentMessages: true,
  stackFrameVariables: true,
  incomingRequestBody: true,
  incomingResponseBody: true,
  outgoingRequestBody: true,
  outgoingResponseBody: true,
  frameContextLines: 5,
}
```

</SpecSection>

<SpecSection id="default-denylist" status="draft" since="0.1.0">

### Default Denylist

For key-value data (HTTP headers, cookies, URL query params), key names are **always included** in the event. SDKs **MUST** apply a **default denylist** to decide which values are sent in plaintext and which the SDK replaces with `"[Filtered]"`. The denylist matches on key name. The SDK never removes or redacts keys themselves.


#### Matching Rule

SDKs **MUST** perform a **partial, case-insensitive match** when comparing key names against the denylist. A key is treated as sensitive if any denylist term appears as a substring in the key name (e.g., the term `auth` matches `Authorization` and `X-Auth-Token`).

#### Base Denylist (Sensitive Data)

The following terms **MUST** be included in the default denylist and applied to headers, cookie names, and query param keys. For cookies and query params that arrive as an unparsed string, see [Cookies and Query Params](#cookies-and-query-params).

`["auth", "token", "secret", "password", "passwd", "pwd", "key", "jwt", "bearer", "sso", "saml", "csrf", "xsrf", "credentials", "session", "sid", "identity"]`

Values for keys that match **MUST** be replaced with `"[Filtered]"`.

#### Extended Deny Terms (`DataCollection.default()`)

The `DataCollection.default()` preset adds the following terms to the deny list for `CollectBehavior` options (cookies, headers, query params). The SDK replaces values of keys matching these terms with `"[Filtered]"` because they commonly carry user-identifying information. Key names are still included in the event.

- Any key containing data about email, user ID, IP address, username, or machine name (if applicable).
- Any key containing **`forwarded`** (e.g., `forwarded`, `x-forwarded-for`) — often carries client IP or host information.
- Any key containing **`-ip`** (e.g., `x-real-ip`, `cf-connecting-ip`) — non-standard headers carrying IP addresses.
- Any key containing **`remote-`** (e.g., `remote-addr`, `remote-host`) — commonly maps to the originating client or proxy.
- Any key equal to or containing **`via`** (e.g., `via`) — often carries proxy host information.
- Any key containing **`-user`** (e.g., `x-user-id`, `remote-user`) — often carries user identifiers.

Effective deny terms for `DataCollection.default()` `CollectBehavior` options: Base list + `["forwarded", "-ip", "remote-", "via", "-user"]` (partial match, case-insensitive).

These terms are **not** added when using `DataCollection.withUserData()`, which relies on the base denylist alone.

#### Cookies and Query Params

Cookies and query params may arrive as a single unparsed string (e.g., `Cookie: user_session=abc; theme=dark-mode`) rather than pre-split key-value pairs. SDKs **MAY** handle both cases:

**When the string can be parsed into individual key-value pairs**, apply the denylist per-key — the same matching rule and terms used for headers apply to cookie names and query param keys. The SDK replaces values for sensitive keys with `"[Filtered]"` while keeping non-sensitive values as-is. This selective filtering retains harmless contextual information for debugging while protecting sensitive fields.

For example, a `Cookie` header parsed into individual cookies:

```
http.request.header.cookie.user_session: "[Filtered]"   // matches "session" in base denylist
http.request.header.cookie.theme: "dark-mode"           // not sensitive — value sent as-is
http.request.header.set_cookie.theme: "light-mode"      // not sensitive — value sent as-is
```

**When individual key-value pairs cannot be extracted** (e.g., malformed or opaque cookie string), the entire `Cookie` or `Set-Cookie` header value **MUST** be replaced with `"[Filtered]"`. This value is used as a fallback:

```
http.request.header.cookie: "[Filtered]"   // fallback: cookie header could not be parsed
```

Unfiltered, raw cookie header values **MUST NOT** be sent. When in doubt, treat the entire cookie header as sensitive and use the fallback.

#### Request and Response Bodies

When request or response bodies are collected (`incomingRequestBody` / `incomingResponseBody` / `outgoingRequestBody` / `outgoingResponseBody`):

- **Parseable as JSON or form data:** SDKs **MAY** extract key-value pairs and apply the same denylist rules to keys. Values for matching keys **MUST** be replaced with `"[Filtered]"`. This allows selective scrubbing while retaining non-sensitive fields.
- **Not parseable (raw bodies):** The body **MUST NOT** be attached to the event. When the SDK cannot parse the body into a key-value structure, the entire body **MUST** be replaced with `"[Filtered]"`.

No built-in option scrubs **keys**; users who need to hide header or cookie names **MUST** use `beforeSend` (or equivalent).

</SpecSection>

<SpecSection id="user-set-data-scrubbing" status="draft" since="0.1.0">

### User-Set Data and Scrubbing

Data **explicitly** set by the user on the scope (user, request, response, tags, contexts, etc.) or on a span, log, or other telemetry is **not** gated by `dataCollection`. It **MUST** always be attached to outgoing telemetry. This also applies to data provided via `beforeSend` or event processors.

SDKs **SHOULD** only replace sensitive values with `"[Filtered]"` when the data is gathered **automatically** through instrumentation. If the user explicitly provides data (e.g., by setting a request object on the scope), the SDK **MUST NOT** modify it; the user is responsible for the data they attach.

Users can register callbacks (e.g., `beforeSend`, event processors) to remove or redact any data — including keys — before events are sent. This spec does not replace those hooks; they remain the primary mechanism for custom filtering and key removal.

</SpecSection>

## Public API

Pass the `dataCollection` option to the SDK's `init` function. Users can construct the object manually, use a preset factory, or tweak a factory's output. All fields are optional when constructing manually; omitting a field uses the same default as `DataCollection.default()`.

```pseudocode
init({
  dataCollection: {
    userInfo: boolean,                   // default: false
    cookies: CollectBehavior,            // default: true
    httpHeaders: CollectBehavior,        // default: true
    queryParams: CollectBehavior,        // default: true
    aiAgentMessages: boolean,            // default: true
    stackFrameVariables: boolean,        // default: true
    outgoingRequestBody: boolean,      // default: TBD
    incomingResponseBody: boolean,     // default: TBD
    incomingRequestBody?: boolean,     // default: TBD
    outgoingResponseBody?: boolean,    // default: TBD
    frameContextLines: number,           // default: 5 (boolean fallback: true)
  },
})
```

### `dataCollection` Options

| Key | Option Type | Default | Since | Description |
|-----|-------------|---------|-------|-------------|
| `userInfo` | Boolean | `false` | 0.1.0 | Attach user identity fields (`user.id`, `user.email`, `user.username`, `user.ip_address`) from automatic instrumentation. |
| `cookies` | CollectBehavior | `true` | 0.1.0 | Collect cookies. All key names are always included; the SDK scrubs values based on the denylist or allow/deny configuration. |
| `httpHeaders` | CollectBehavior | `true` | 0.1.0 | Collect HTTP headers. All key names are always included; the SDK scrubs values based on the denylist or allow/deny configuration. |
| `queryParams` | CollectBehavior | `true` | 0.1.0 | Collect URL query parameters. All key names are always included; the SDK scrubs values based on the denylist or allow/deny configuration. |
| `aiAgentMessages` | Boolean | `true` | 0.1.0 | Include AI agent input and output messages. |
| `stackFrameVariables` | Boolean | `true` | 0.1.0 | Include local variable values captured within stack frames. |
| `outgoingRequestBody` | Boolean | `false` | 0.1.0 | Include full body of outgoing HTTP requests. |
| `incomingResponseBody` | Boolean | `false` | 0.1.0 | Include full body of the incoming HTTP response. |
| `incomingRequestBody` | Boolean? | `false` | 0.1.0 | Include full body of the incoming HTTP request (applicable in server-environments). |
| `outgoingResponseBody` | Boolean? | `false` | 0.1.0 | Include full body of outgoing HTTP responses (applicable in server-environments). |
| `frameContextLines` | Number (Boolean fallback) | `5` (`true`) | 0.1.0 | Number of lines of context to include around stack frames. |

<Expandable title="Why are some options boolean-only?">
  Unlike cookies or headers, some data (e.g. request bodies) has no predictable key structure for the SDK to filter. Data can still be redacted in `beforeSend` or event processors if needed.
</Expandable>

### Preset Factories

SDKs **MUST** expose the following factory methods (or language-idiomatic equivalents):

| Factory | Description |
|---------|-------------|
| `DataCollection.default()` | Returns the safe-by-default configuration: `userInfo: false`, bodies disabled, and known PII fields explicitly denied. |
| `DataCollection.withUserData()` | Returns a full-collection configuration: `userInfo: true`, bodies enabled, and standard collection defaults without explicit PII denials. |

SDKs **MAY** provide additional factories. All factories **MUST** return a mutable `DataCollection` object that users can modify before passing to `init`.

## Examples

The following examples show common configurations using presets and manual construction.

### Using the Default Preset

The minimal configuration uses the factory directly:

```typescript
init({
  dsn: "...",
  dataCollection: DataCollection.default(),
});
```

This is equivalent to omitting `dataCollection` entirely — the SDK applies the same defaults.

### Tweaking a Preset

Start from a preset and override individual fields before passing to `init`:

```typescript
const config = DataCollection.default();
config.incomingRequestBody = true;
config.httpHeaders = { allow: ["x-request-id"] };

init({
  dsn: "...",
  dataCollection: config,
});
```

**Result:** Default safe configuration, but incoming request bodies are collected. All header key names are present in the event, but only `x-request-id` sends its real value. Every other header value appears as `"[Filtered]"`.

### Enabling Full PII Collection

Use `DataCollection.withUserData()` to mirror the legacy `sendDefaultPii: true` posture with bodies enabled:

```typescript
init({
  dsn: "...",
  dataCollection: DataCollection.withUserData(),
});
```

**Result:** User identity fields, request/response bodies, headers, cookies, query params, and AI agent messages are all collected. The base denylist still scrubs sensitive values.

### Granular Debugging (Manual Construction)

Users can also construct the object manually without a factory:

```typescript
init({
  dsn: "...",
  dataCollection: {
    userInfo: true,
    httpHeaders: { allow: ["x-request-id", "x-trace-id", "x-correlation-id"] },
    queryParams: false,
  },
});
```

Omitted fields use the same defaults as `DataCollection.default()`.

### Migration from `sendDefaultPii`

- **`sendDefaultPii: true`** (legacy) → `dataCollection: DataCollection.withUserData()` (or manually: `{ userInfo: true, incomingRequestBody: true, ... }`)
- **`sendDefaultPii: false`** (legacy) → `dataCollection: DataCollection.default()` (or omit entirely — same as default)

SDKs **SHOULD** document this mapping and **MAY** implement `send_default_pii` as a compatibility shim that maps to the appropriate preset.
