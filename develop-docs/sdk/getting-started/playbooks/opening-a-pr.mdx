## Steps

#### 1. Branch from the default branch

Use the naming convention `<type>/<short-description>` ([Branch naming](../standards/code-submission#branch-naming)). Examples: `feat/add-user-auth`, `fix/rate-limit-parsing`.

#### 2. Confirm a linked issue exists

If there isn't one, create it first — even for internal work. Non-trivial changes require an issue before a PR (Principle 4).

#### 3. Write commits

Use [conventional format](../standards/code-submission#commit-message-format). If AI-assisted, add `Co-Authored-By` in the commit footer ([AI attribution](../standards/code-submission#ai-attribution)). **Skill: `sentry-skills:commit`**

#### 4. Verify locally

Tests pass, linter passes, the SDK builds cleanly.

#### 5. Open the PR as a draft

([PR draft mode](../standards/code-submission#pr-draft-mode)). **Skill: `sentry-skills:create-pr`**

#### 6. Write the description

What changed, why, linked issue, context for reviewers. No boilerplate, no test plan sections ([PR description quality](../standards/code-submission#pr-description-quality)).

#### 7. If the change is user-facing

Add a changelog entry ([Changelog entry](../standards/code-submission#changelog-entry)) and open or link a docs PR ([Documentation-with-code](../standards/repository-docs#documentation-with-code)).

#### 8. Wait for CI to pass

When green, mark the PR as ready for review.

#### 9. Assign 1–2 reviewers

If the PR touches public API, dependencies, or security-sensitive areas, assign a senior reviewer ([Required reviewers](../standards/review-ci#required-reviewers)).
