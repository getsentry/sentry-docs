These standards define how work gets planned, scoped, and tracked before anyone writes code. Good planning avoids mid-implementation surprises and keeps work visible to the team.

Work is organized in three levels: **issues**, **projects**, and **initiatives**. All issues in SDK GitHub repositories are automatically and bi-directionally synced with Linear. Triaging and planning happens in Linear.

<SpecSection id="issues" status="stable" since="1.0.0">

## Issues

An issue is the unit of work — one assignee, one thing to do. Issues map directly to GitHub issues and are synced bi-directionally with Linear.

Every issue **MUST** have these attributes set in Linear, before it moves to **In Progress**:

- **Assignee**: a single owner (not a team)
- **Description**: verbose enough for anyone on the team to understand scope and acceptance criteria without asking the author
- **Labels**: both an **SDK label** (the SDK(s) the issue affects) and a **Type label** (one of `Bug`, `Docs`, `Feature`, `Improvement`, `Task`, or `Tests`). These are required for workspace-level reporting in Linear.
- **Estimate**: story-point or t-shirt size agreed by the team's estimation convention
- **Priority**: set for all issues in the backlog or later states

Issues estimated larger than **M** (or the team's medium equivalent) **MUST NOT** move to **Todo** until they are either split into smaller issues (each ≤ M) or converted to a project.

### When is an issue done?

- All acceptance criteria in the description are met
- Code is reviewed and merged
- Tests cover the changed behavior
- Any user-facing documentation impact is either shipped or tracked in a follow-up issue

</SpecSection>

## Projects

A project is a collection of related issues that together deliver a cohesive outcome for a single SDK. Projects live in Linear.

Use a project when work spans multiple issues that need to be tracked together — for example, adding a new feature that requires API changes, tests, and docs. If the work fits in a single issue, don't create a project for it.

### Design-first gate

A design document is required before implementation begins when:

- The project estimate is greater than **M**, or
- The change affects more than one SDK

Acceptable document types (one is sufficient): **Develop Docs** (implementation-heavy, scoped to one or a few SDKs), **RFC** (needs broad stakeholder input), **PRFAQ** (product-facing or user-visible), or **DACI** (needs explicit role clarity).

The document **MUST** be complete enough that a reviewer can approve the approach — a draft or stub doesn't count. It must also identify who needs to review and approve the approach and who needs to be kept informed. A short list is sufficient; the goal is to surface dependencies early.

Link the design document in the Linear project's **Documentation** field. The first implementation PR should not be approved until a reviewer verifies the document is present.

The design document **MUST** include a risk section identifying technical risks (implementation unknowns, performance concerns, platform-specific challenges), integration risks (upstream API dependencies, cross-SDK coordination), and anything else the team anticipates. High-rated risks **MUST** be addressed before the first implementation issue moves to **In Progress** — either by running a spike or by documenting a concrete mitigation plan. Don't start on the fun parts while high risks remain unresolved.

### Naming

**Standalone projects** should use a clear, action-oriented title that describes what will be delivered, not what will be done.

**Cross-SDK rollout projects** — when the same effort is delivered independently by multiple SDK teams — **MUST** use the format `[Effort name] [SDK name]`. For example: `OTLP Integration [Python]`, `Implement Global Attributes API [Android]`. This makes it unambiguous which team owns the project and easy to filter.

### When is a project done?

- All issues are **Done** or **Cancelled** (with a note explaining why)
- User-facing documentation is shipped, or explicitly deferred with a tracked follow-up
- No known regressions have been introduced without a tracked resolution plan
- A final project update has been posted summarizing what shipped

"Mostly done" is not done. Defer remaining work to a follow-up issue or project instead.

</SpecSection>

## Initiatives

An initiative is a cross-SDK effort that spawns one or more projects. Initiatives can contain sub-initiatives (forming a tree), with projects sitting on the leaves.

Use an initiative when the same effort needs to be delivered independently by multiple SDK teams. Each SDK gets its own project under the initiative, following the `[Effort name] [SDK name]` naming convention.

</SpecSection>

## AI-assisted planning

AI tools can help generate issue descriptions and decompose work into sub-issues. When they do:

- **AI-generated issue descriptions**: the assigned human **MUST** review and edit the description before the issue moves to **Todo**. Unreviewed AI output is not authoritative scope.
- **AI-decomposed sub-issues**: the project lead **MUST** review and approve all sub-issues before any of them move to **Todo**. Don't auto-accept in bulk.

</SpecSection>
